# Contributing

These rules are what make small UI changes safe to hand off — to a human
in a hurry or to an agent. The full version, with the history behind each
decision, is `crates/ironclaw_webui/DESIGN_SYSTEM.md`.

## The rules

1. **Colors come from tokens.** No raw hex, no `rgb()`, no legacy alias
   utilities (`text-white`, `text-iron-*`, `bg-signal`, …). CI runs a
   token ratchet (`scripts/check-design-tokens.mjs`) that fails when a
   file gains a raw color; the baseline only goes down. `color-mix(...)`
   over `--v2-*` vars is fine. One trap: the compat shim remaps
   `text-white` to the theme ink — on an accent fill you want
   `text-[var(--v2-on-accent)]`.
2. **Control heights come from the density scale.** Interactive controls
   share `--v2-control-h-sm/md/lg` (28/32/36px) and the matching paddings
   so mixed rows align. Don't invent a height.
3. **Radii, spacing, type sizes, z-index: pick a step.** Every scale is
   enumerated in Tokens/Reference. If the step you want is missing, that's
   a design conversation, not a one-off value.
4. **Status colors follow the canon.** `STATUS_CANON` in `tokens.ts` is
   the single mapping from status words to tokens. Text, dot, and progress
   fill draw from the same pair — don't mix hues.
5. **Motion is token-driven and restrained.** Pick a `--v2-duration-*` +
   `--v2-ease-*` pair (or `MOTION_DURATION` for motion/react). Entrances
   ease out, exits are quicker than entrances, and nothing bounces on
   pointer or keyboard input. The only ambient loops are typing, spin,
   breathe, and skeleton shimmer — and everything respects
   `prefers-reduced-motion`.
6. **Focus is instant.** Focus rings are never transitioned. Hover is a
   fast color/border shift with no movement.
7. **Third-party brand colors** (Gmail red, Slack aubergine, …) stay out
   of the semantic system. Use them inside the brand mark itself, never
   for UI chrome.
8. **New components sit on Radix when a primitive exists.** Accessibility
   comes from the primitive; the component adds tokenized styling and an
   API. Follow the CVA + `cn()` pattern already in the package.

## Adding a component

1. Build it in `src/` on the relevant Radix primitive, styled with
   Tailwind + `--v2-*` tokens.
2. Export it from `src/index.ts` — the barrel is the public API.
3. Add a story file in `stories/components/` covering every variant and
   state, with `argTypes` for the props.
4. New token? Add the value to `tokens.css` (both themes), catalog it in
   `tokens.ts`, and pin it in `tokens.test.ts`.
5. Run the checks: `pnpm typecheck && pnpm test` here, plus the app build
   and the token ratchet at the frontend root.

## Testing in product

The WebUI `/playground` route renders the full component set inside the
real app shell. Use it to check a component against the actual canvas,
chrome, and both themes before you ship.
