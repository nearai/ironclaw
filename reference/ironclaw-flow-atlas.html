The Flow Fingerprint

For any prompt, walk these questions in order. This is the fastest way to avoid confusing product UX, loop behavior, kernel authority, and substrates.

Question Expert answer shape Primary source
1. Where did it enter? Product surface or trigger submits into shared runtime/conversation services. runtime-workflows.md
2. What durable work exists? Accepted message, submitted turn, queued run, active-thread lock. Architecture.md
3. Who runs it? Runner claims the queued run, heartbeats the lease, then invokes one loop driver. turn-runner.md
4. Is it a reply or an effect? Loop emits Reply or explicit CapabilityCalls. agent-loop-protocol.md
5. If effectful, what authority is needed? CapabilityHost checks grants, leases, approval/auth gates, trust, resources, mounts, secrets, and network policy. kernel-boundary.md
6. What executes it? RuntimeDispatcher routes already-authorized work to WASM, script/process, MCP, or first-party lanes. host-runtime.md
7. What is visible to users? Transcript stores final replies; event/projection streams show redacted progress; run state tracks current lifecycle. run-state.md
8. What if it blocks or crashes? Approval/auth blocks keep the active lock; expired leases go to recovery, not blind retry. approvals.md

Scenario Atlas

User prompt Flow skeleton Diagnostic question
"Explain this Rust module." Product -> thread -> turn/run -> runner -> loop -> model -> final reply. Did the loop need side effects, or could it answer from visible context?
"Read src/app.rs and summarize it." Loop asks file-read capability -> CapabilityHost checks exact scoped read -> first-party/file runtime returns safe result -> loop replies. Was the read authorized for this run, path, scope, and mount view?
"Edit the file and run tests." Write/shell capability calls -> authorization/approval/obligations -> runtime lane -> refs/results -> loop decides next step. Which side effects need approval, and what result refs prove completion?
"Post this summary to GitHub." External write capability -> approval may block -> approved lease resumes exact invocation -> staged credentials/network policy -> runtime HTTP. Is the approval exact-invocation, and are credentials one-shot staged?
"Connect my Google account, then list docs." Capability reports auth required -> run becomes BlockedAuth -> product presents auth -> scoped secret lease -> resume original action. Is this auth-blocked, not approval-blocked, and is retry replay-safe?
"Run this every weekday at 9." trigger_create writes schedule -> poller claims due slot -> trusted synthetic inbound turn -> normal turn pipeline. Is this a trigger feeding normal turns, or did someone create a parallel routine loop?
"Spawn a subagent to investigate." Spawn capability -> child thread/run -> runner claims child -> child loop executes through same host ports -> parent consumes result refs. Does the child use the normal runner/driver/executor path?
"Cancel that." Product asks coordinator to cancel -> host ports expose cancellation -> loop exits cancelled -> runner validates evidence -> active lock releases. Did durable cancellation state exist before terminal cancellation?
"The worker died mid-run." Heartbeat expires -> reconciler marks RecoveryRequired -> active lock remains -> no blind retry of uncertain effects. Could retry duplicate side effects? If yes, recovery must stay explicit.
"I installed a new extension." Registry/catalog refresh -> visible capability surface may change -> loop sees new metadata -> exact calls still require authorization. Are we mixing visibility with authority? They are different gates.

Permission Ladder

Do not collapse these rungs. Most security bugs come from treating one rung as if it implied the next.

Rung Means Does not mean
Registered capability The system knows a capability descriptor exists. The model can see it or run it.
Visible capability The model may see a scoped tool description. The exact invocation is authorized.
Grant or active lease A principal has constrained authority under scope/effects. The runtime may skip obligations.
Approval lease A human/policy approved one fingerprinted invocation. Future different inputs are approved.
Prepared obligations Resources, mounts, network policy, secrets, audit, or redaction are staged. The runtime gets ambient host authority.
Runtime dispatch Already-authorized work enters a concrete lane. The dispatcher decided permission.

Kernel Smell Tests

  • If a loop can touch filesystem, network, secrets, process, or memory without a host port, suspect a kernel-boundary violation.
  • If a product adapter decides authorization or dispatches tools directly, suspect a product/kernel boundary violation.
  • If a trigger fire does not become a normal turn, suspect a parallel routine runtime.
  • If a worker crash auto-retries side-effecting work, suspect unsafe recovery semantics.
  • If public events include raw prompts, tool JSON, secrets, host paths, or backend diagnostics, suspect a redaction failure.
