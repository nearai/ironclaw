The Four-Layer Question

This is an ownership map, not a timeline. There is one kernel boundary. In a turn timeline, the same boundary is crossed once for run control and again for side-effect control.

Product CLI, WebUI, product adapters. Owns UX, transport normalization, approval/auth presentation, and user-visible event surfaces.
Kernel Boundary Turn coordination, active locks, runner leases, approvals, authorization, capability host, process/resource authority, network/secrets policy.
Userland Loop Prompt/model/tool strategy. Decides whether to reply, call capabilities, checkpoint, retry, ask follow-up, or stop.
Substrates Durable stores and reusable primitives: threads, events, memory, filesystem, approvals, leases, runtime lanes, resources.

The practical question is: who owns the decision? Products own presentation, loops own behavior, the kernel boundary owns authority and recovery, and substrates own durable primitives.

Common Flow Skeletons

Plain answer

  1. Product accepts text into thread storage.
  2. TurnCoordinator queues a run and locks the canonical thread.
  3. Runner claims the run and starts a loop driver.
  4. Loop assembles prompt and calls model.
  5. Model returns Reply.
  6. Host stores final assistant message; runner validates refs and completes.

Tool or capability call

  1. Loop sees model CapabilityCalls.
  2. Loop calls host capability port.
  3. CapabilityHost checks descriptor, trust, grants, approvals, obligations.
  4. RuntimeDispatcher routes already-authorized work to a runtime lane.
  5. Safe summaries and refs return to the loop.

Approval needed

  1. Authorization returns RequireApproval.
  2. Host stores pending approval and blocks the run with a gate/checkpoint ref.
  3. Product renders approval UI.
  4. Approval resolver issues exact fingerprinted lease.
  5. Coordinator resumes the same run; capability dispatch claims and consumes the lease.

Scheduled routine

  1. Trigger poller evaluates due trigger records.
  2. Worker claims one fire slot and mints trusted trigger scope.
  3. Conversation layer converts the fire into a synthetic inbound turn.
  4. Normal turn pipeline runs; no separate routine agent loop appears.
  5. Trigger active-fire state clears after terminal turn outcome.

Boundary Rules To Memorize

  • A registered or visible capability is not authority.
  • The dispatcher routes already-authorized requests; it does not decide permissions.
  • The loop is behavior, not the security perimeter.
  • LoopExit is a claim; runner validation turns it into durable state.
  • Blocked approval/auth states keep the active thread lock until resume, cancel, fail, or complete.
  • Triggers create normal inbound turns; they must not own a parallel loop.
  • Raw prompts, tool inputs, secrets, host paths, and backend diagnostics do not belong in turn state or public events.
