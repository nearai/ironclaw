Lesson 0002

Prompt Flow Atlas

Learn the expert move: take any user prompt, turn it into a system movie, then name the subsystem that owns each step.

Reference: Ironclaw Flow Atlas. Primary source: crates/Architecture.md.

Learning Target

After this lesson, you should be able to classify common prompts as plain replies, capability calls, approval/auth blocks, trigger-created turns, subagent child runs, cancellation, or recovery.

The Expert Shortcut

Do not start by asking "which crate?" Start by asking whether the prompt needs a side effect.

No side effect

The loop can gather allowed context, call the model, and store a final assistant reply. The kernel still owns run coordination, but there may be no capability execution.

Example: "Explain this code."

Side effect

The loop must emit explicit capability calls. Reading files, writing files, HTTP, shell, GitHub, triggers, and subagents all cross the kernel-mediated capability path.

Example: "Edit this file and run tests."

Source: the parent loop envelope is Reply | CapabilityCalls in agent-loop-protocol.md.

Kernel In One Sentence

The kernel is not "the brain." It is the security and recovery boundary that asks: can this exact scoped run do this exact thing now, and can we record the outcome safely?

In practice, that boundary includes turn coordination, runner leases, loop-exit validation, CapabilityHost, authorization, approvals, obligations, scoped filesystems, network/secrets policy, resources, events, and recovery. There does not need to be an ironclaw_kernel crate for a kernel boundary to exist.

Source: kernel-boundary.md.

Six Scenario Movies

1. "Explain this code."

  1. Product writes the user message into thread storage.
  2. TurnCoordinator queues a run and holds the active-thread lock.
  3. Runner claims the run and starts the loop driver.
  4. Loop builds prompt context and calls the model.
  5. Model returns Reply.
  6. Final assistant reply is stored; runner completes the run.

2. "Read this file."

  1. Same turn/run setup as a plain reply.
  2. Loop asks for a file-read capability.
  3. CapabilityHost checks the exact scope, capability, effects, and mounts.
  4. Authorized runtime reads only the scoped path.
  5. Safe result returns to the loop; the loop replies.

3. "Post this to GitHub."

  1. Model asks for an external-write capability.
  2. Authorization may return RequireApproval.
  3. Run becomes BlockedApproval with gate/checkpoint refs.
  4. Product renders approval UI; resolver issues a fingerprinted lease.
  5. Resume replays that exact invocation, stages credentials/network policy, then dispatches.

4. "Connect my account."

  1. Capability or runtime reports auth required.
  2. Run becomes BlockedAuth, not BlockedApproval.
  3. Product presents the OAuth/token path.
  4. Secret lease is recorded under scope.
  5. Resume retries the original action without leaking raw secrets to transcript.

5. "Run every weekday at 9."

  1. trigger_create validates schedule and timezone before persistence.
  2. Trigger record captures creator/user/agent/project scope.
  3. Poller claims one due fire slot for the trigger.
  4. Trusted trigger submitter creates a synthetic inbound turn.
  5. Normal turn, runner, loop, permission, and recovery pipeline takes over.

6. "Spawn a subagent."

  1. Parent loop calls a spawn capability, not a private executor.
  2. Child thread/run is created from parent scope and delegation.
  3. Runner claims the child run like any other run.
  4. Child loop uses the same host ports and capability gates.
  5. Parent consumes child result refs when the child reaches terminal state.

Permission Ladder

A model-visible tool is not permission. The ladder looks like this:

Catalog Capability exists.
Visibility Model may see it.
Authorization Exact invocation is allowed, denied, or requires approval.
Obligations Audit, redaction, mounts, resources, network, secrets are prepared.
Dispatch Already-authorized work enters a runtime lane.

Source: capability-access.md, capabilities.md, and host-runtime.md.

Routines Are Triggers

Product language may say "routine" or "automation." In Reborn's current architecture, scheduled routine-like work should become a trigger fire, then a normal inbound turn. That gives scheduled work the same active-lock, runner lease, approval, auth, event, and recovery machinery as an ordinary user message.

The smell: if scheduled work owns a separate agent loop, separate tool executor, or hidden outbound delivery path, it is probably bypassing the architecture.

Source: triggers.md.
