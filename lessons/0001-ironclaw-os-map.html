Lesson 0001

One Message Through Ironclaw

Before the architecture map, build the movie: you type a message, Ironclaw accepts it, a worker runs it, a loop thinks, permissions guard side effects, and the reply lands back in the thread.

Reference: Ironclaw System Map. Next lesson: Prompt Flow Atlas. Primary source: crates/Architecture.md.

Learning Target

After this lesson, you should be able to explain one plain chat request without naming every crate. The crate names come later. First, you need the shape of the system in motion.

What Ironclaw Is

Ironclaw Reborn is a secure agent engine. In normal words: it receives requests from places like the CLI or WebUI, turns each request into durable work, lets an agent loop decide what to do, and forces every side effect through permission checks.

Not one giant chatbot

The UI, the agent loop, the permission system, the tool runner, the transcript store, and the event stream are separate parts. They cooperate, but they do not own each other's jobs.

More like a small OS

A request becomes a scheduled unit of work. A worker claims it. The "brain" asks for effects. The "kernel boundary" decides whether those effects are allowed and how they run.

The One-Message Story

Imagine you type: "Read README.md and summarize it." Here is the movie.

The front door accepts the message

The product surface, such as CLI or WebUI, receives your text. It should not decide tool permissions or agent strategy. It just normalizes the request into Reborn's shared shape.

Product surface: CLI, WebUI, product adapter.

The message is written into a thread

Ironclaw stores the accepted inbound message in conversation/thread storage. This creates a durable reference to the thing you asked.

Thread storage: the transcript source of truth.

A turn run is queued

The turn coordinator creates work for the message. It also makes sure there is only one active run for the same conversation, so two workers do not race to answer the same thread.

TurnCoordinator: admission, active lock, run record.

A runner claims the work

A worker claims the queued run and keeps renewing a temporary "I am working on this" claim while it is active. If the worker disappears, the run goes to recovery instead of blindly retrying unknown side effects.

TurnRunnerWorker: claim, heartbeat, recovery.

The loop thinks

The agent loop gathers the information it is allowed to see, calls the model, and decides whether the next move is a user-visible reply or a request to do something.

Userland loop: prompt, model, plan, stop/retry/checkpoint.

Reading the file must pass through permissions

If the loop asks to read README.md, it does not directly touch the filesystem. It asks the host through a controlled doorway. The capability host checks whether this exact run is allowed to read that exact file before anything touches disk.

CapabilityHost: exact request permission gate.

The result comes back to the loop

The approved file read happens. The loop receives the result, asks the model to summarize it, and writes a final assistant reply. The runner only marks the run complete after the reply has actually been stored.

Complete run: execute, summarize, store reply, mark done.

Jargon Alarm

If you see a phrase like scoped loop host port, translate it like this:

A host port is a controlled doorway. The loop cannot directly grab files, secrets, tools, or transcript state. It can only ask the host through specific doorways. Scoped means the doorway only works inside the current tenant/user/thread/run permissions.

For lesson 1, you do not need to memorize that phrase. The useful idea is simpler: the loop asks; the host decides and performs.

Now The Map

Now the four big buckets should have somewhere to attach. Products and loops are replaceable userland. The kernel boundary owns authority and recovery. Substrates own durable primitives. There is one kernel boundary; a normal turn crosses different kernel-owned services at different moments.

Product Accepts a user or external event and translates it into Reborn's shared request shape.
CLIWebUIAdapters
Kernel Boundary Coordinates runs and mediates effects: one active run per thread, temporary worker claims, approvals, permissions, and recovery.
TurnCoordinatorCapabilityHost
Userland Loop Decides what the agent does: prompt, model, tool request, checkpoint, reply, retry, stop.
Planned loopSubagent loop
Substrate Stores and services: threads, memory, filesystem, events, resources, and tool executors.
EventsFilesystemMCP

Why you may see the kernel twice in flow diagrams: that is not two kernels. It means the same boundary is crossed in two roles: first for turn control (TurnCoordinator, one-active-run rules, worker claims), then later for effect control (CapabilityHost, permissions, approvals, execution).

The loop is not trusted because it is clever. It is constrained because it can only ask the host to do things through controlled doorways. Those requests eventually route privileged work through CapabilityHost, permission checks, approvals, and tool execution.

The Two Golden Splits

Answer vs request to act

The loop has two basic choices: answer the user, or ask the host to do something. If work has side effects, prose is not enough; it must become an explicit request to act.

Source: agent-loop-protocol.md.

Visibility vs Authority

The model can be told a tool exists and still not be allowed to use it. "This tool exists" is different from "this exact request is allowed." CapabilityHost decides the second one.

Source: capability-access.md.

Common Prompt Flows

"Explain this code."

  1. Product accepts the user message and writes it to the thread.
  2. TurnCoordinator queues a run and reserves the conversation.
  3. Runner claims the run and gives the loop controlled doorways for this run.
  4. Loop gathers allowed context and calls the model.
  5. Model returns an answer.
  6. Final assistant message is stored, then the run is marked complete.

"Read this file and summarize it."

  1. The loop may choose to ask for a file read.
  2. The host checks whether this exact file read is allowed.
  3. The host only gives the file reader access to approved paths.
  4. The file read executes after authorization.
  5. The result returns to the loop, then the model replies.

"Post this to GitHub."

  1. Model asks for an action, not just intent text.
  2. The permission check may require existing permission or human approval.
  3. If approval is needed, the run pauses and remembers where to resume.
  4. Approval creates a one-time permission for that exact request.
  5. Resume uses that one-time permission, then runs the request.

"Run every weekday at 9."

  1. A trigger/routine capability creates a durable trigger record.
  2. The trigger poller evaluates schedules and claims due fire slots.
  3. The fire becomes a trusted synthetic inbound turn.
  4. The same turn, runner, loop, capability, and recovery pipeline handles execution.
  5. No hidden routine-only agent loop should exist.

Practice

Which statement is the expert answer?

A model sees builtin.http in its visible capability list and asks to call an external API. What must still happen before outbound HTTP?

Classify the bug

A loop says "I am done," but the assistant reply was never actually stored. Which layer should reject completion?
