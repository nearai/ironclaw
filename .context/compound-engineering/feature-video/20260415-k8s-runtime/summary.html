IronClaw / K8s Runtime

Allowlist gating now matches the real runtime contract

This change narrows Kubernetes messaging to the rule the code already enforces: allowlist networking depends on native network controls, while projected runtime config stays a separate delivery capability.

Rule 1

Allowlist uses one gate

Kubernetes allowlist-only networking is now described as depending on IRONCLAW_K8S_NATIVE_NETWORK_CONTROLS alone.

The older wording that bundled it into a broader “Stage 3 prerequisites” message has been removed.

Rule 2

Projected config stands alone

IRONCLAW_K8S_PROJECTED_RUNTIME_CONFIG is now described only as runtime config delivery.

It no longer reads like an extra prerequisite for Kubernetes allowlist support.

Rule 3

Write-back wording is corrected

User-facing text no longer says workspace-write one-shot commands must fall back to Docker.

The summary now reflects the shipped behavior: pending workspace changes come back for explicit review and apply.

Verified with targeted Rust tests for readiness helpers, capability gating, and doctor output strings.
