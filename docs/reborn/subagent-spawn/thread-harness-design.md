under PR #5176 ‚Äî no code from either survives beyond what ¬ß3 below marks
SURVIVES. This design was hardened through multiple rounds of adversarial
technical review against the live crates; every ruling cites the concrete
call sites, line numbers, and existing production patterns it verified
against.

## 1. Inherited core

- Child = normal Turn thread + lineage header (`parent_run_id`, `tree_root_run_id`, `depth`). Inspect/human-attach/re-activate are ordinary thread operations.
- One await-edge file per parent-awaits-child; CAS `open ‚Üí settled ‚Üí drained`, `open ‚Üí abandoned`.
- Terminal is per-run; threads park, never die. `activate(thread, input, provenance)` is the single re-activation primitive; `ThreadBusy` if a run is live.
- Consent-to-wake: an agent wakes only its own direct live child; orphaned descendants are human-wake-only.
- Parent resume via `TurnStatus::BlockedDependentRun` + `gate:subagent-await-<child_run_id>` + `resume_turn` with `ResumeTurnPrecondition::BlockedDependentRunGate` (verified: `crates/ironclaw_turns/src/request.rs:33,48` maps this to `required_status() == Some(BlockedDependentRun)`).
- Child run record is the result source of truth (no `CapabilityResultStore`).
- Safety: child‚Üíparent framed-as-untrusted + byte-capped via `wrap_untrusted_subagent_text`; child‚Üíhuman raw; gates bubble up lineage (generalized in ¬ß9).
- **No flag.** `disabled_capability_ids` gates the capability.

## 2. Closed edges are deleted

CLAUDE.md: *"LLM data is never deleted... 'Cleanup' means evicting from in-memory caches, never deleting database rows."* Scoped to **model I/O** ‚Äî transcripts, run records, events. The await-edge is control-plane delivery bookkeeping, not a record of what any LLM said or did. The child's run record, its lifecycle events, and the framed content the parent consumed (written at drain) are the durable facts ‚Äî the edge is scaffolding for one-time delivery; once delivered it has no further job.

**Ruling: on CAS to `drained`, and on `abandoned` finalization, the edge file is deleted ‚Äî via the new CAS-guarded `delete_if_version` (¬ß4.0), never a blind delete.** `RootFilesystem::delete(path)` has **no CAS precondition** today (`crates/ironclaw_filesystem/src/root.rs:158` ‚Äî no `CasExpectation` param, unlike `put`); ¬ß4.0 adds the missing primitive, closing the silent blind-delete gap this design corrects.

**Crash-ordering and self-healing.** Edge close is durable across **three** steps, every time ‚Äî not a special recovery-only path: (1) CAS the edge's own state to its terminal value, (2) drive `reservation_release` (¬ß5.5) `Unclaimed ‚Üí Claimed ‚Üí Released`, (3) `delete_if_version(edge_path, released_version)`. **Order is fixed ‚Äî state transition, then release, then delete** (deleting before the state CAS would make a live-looking edge vanish with no record of why; deleting before the release reaches `Released` would strand `SpawnTreeReservation` capacity with no edge left on disk to recover it from). A crash leaves either a **terminal-but-unreleased edge** (crashed between (1) and (2)) or a **terminal-and-released-but-undeleted edge** (crashed between (2) and (3)) on disk ‚Äî **recovery (boot pass ¬ß4.3/¬ß5.3, and the lazy backstop) must enumerate `drained`/`abandoned` edges the same way it enumerates open ones**, complete whichever of steps (2)/(3) is outstanding, in order, and only then move on; `NotFound`/`VersionMismatch` on the retried delete is itself benign (someone else finished it, or it's mid-handling) and reported `already_closed`. **This makes the orphan window self-healing by construction, not merely invisible** ‚Äî recovery cost stays bounded, now also covering both post-terminal windows, not just true open/settled state.

**Round-4 fix ‚Äî the delete token is always the version from the *last* successful CAS write, never the state-transition CAS from step (1).** Every CAS on the edge file ‚Äî the terminal-state transition in step (1), and each of the `Unclaimed ‚Üí Claimed` / `Claimed ‚Üí Released` transitions in step (2) ‚Äî bumps the same file's `RecordVersion`, because ¬ß5.5's tri-state lives on that same edge file, not a separate one. A naive reading of step (3) as "delete using the version step (1)'s CAS returned" is therefore always wrong by the time step (3) runs: step (2)'s `Released` CAS has already superseded it, so `delete_if_version(edge_path, <step-1's version>)` would deterministically fail `VersionMismatch` on every close, piling up terminal-but-undeleted edges instead of ever deleting them. **The correct rule, stated once here and consistent everywhere this sequence is named (¬ß4.0, ¬ß5.5): `released_version` is the `RecordVersion` returned by the caller's own `Claimed ‚Üí Released` CAS in step (2) ‚Äî the freshest write the caller just made ‚Äî never a version captured earlier in the sequence.** Recovery follows the identical rule when completing an outstanding step: whichever of (2)/(3) it drives to completion, it uses the `RecordVersion` its own most recent successful write returned as the token for the next write, re-reading current state rather than reusing a version captured before it started its recovery pass.

**Consequences:** the per-parent live-index approach from earlier drafts is **deleted outright** (deletion gets there for free). No closed-root/archive language anywhere. Settled-but-undrained edges are the pending-delivery queue and are never deleted ‚Äî only terminal transitions delete. `NotFound` on any edge read is benign ‚Äî treat it like a completed settle-then-drain.

**Abandon is mode-scoped, not a blanket "parent terminal ‚Üí abandoned" rule.** ¬ß1's inherited open‚Üíabandoned transition applies only to:
- **(a) Blocking edges whose resume became impossible** ‚Äî the parent run went `Cancelled`/`Failed` while parked on the gate/dependent-run wait, and was never resumed. This is the failure case `abandoned` exists to capture.
- **(b) Explicit tree teardown** ‚Äî an operator/human tears the tree down (`subagent_cancel`, ¬ß7 PR6) while the edge is still open.

**For background edges, parent-run completion with the edge open is the *normal* delivery case, not an abandonment trigger** ‚Äî see ¬ß8. Threads park, never die (¬ß1); a live parent thread finishing its run is not "parent gone," it's exactly the state ¬ß8's `System`-provenance activate exists to wake. The edge stays `open` (then `settled`) and drains the ordinary way, whether the parent run that opened it is still live or has already gone terminal.

## 4.0 CAS-guarded delete primitive

PR1 merge blocker: **P1.0b**.

**Gap:** `RootFilesystem::delete` (`root.rs:158`) takes no `CasExpectation`, unlike `put`. Every backend's delete is unconditional: `in_memory.rs:142` (`state.entries.remove(path)`, no version compare), `libsql.rs:875` (`DELETE ... WHERE path = ?1 OR path LIKE ?2`, no version predicate), `postgres.rs:664`/`postgres_delete_with_client` (`postgres.rs:1584`, same shape). ¬ß2's edge-close and ¬ß4.5's roster-prune both need TOCTOU-safe delete; neither exists today.

**Ruling: add `delete_if_version(&self, path: &VirtualPath, expected_version: RecordVersion) -> Result<(), FilesystemError>` as a new, additive `RootFilesystem` method ‚Äî not a signature change to `delete`.** No `CasExpectation` parameter: `Any` would be behaviorally identical to the existing `delete(path)` (offering it would duplicate a sibling method), and `Absent` has no meaning for a delete (nothing to compare against) ‚Äî both are unrepresentable in the parameter type rather than runtime-rejected branches. Every call site passes a concrete `RecordVersion` it already holds ‚Äî for the edge, that is always the version returned by the caller's own last successful CAS write (the `Claimed ‚Üí Released` reservation-release transition, ¬ß5.5/¬ß2's round-4 fix ‚Äî never an earlier transition's version, since every CAS on the same file supersedes the one before it); for the roster, the marker's just-read version. `RootFilesystem::delete(path)` has ~20+ production call sites across unrelated subsystems (verified: `ironclaw_secrets`, `ironclaw_skills`, `ironclaw_outbound/src/filesystem_store.rs`, `ironclaw_threads/src/filesystem_service.rs`, `ironclaw_product_workflow/src/filesystem_ledger.rs`, `ironclaw_reborn_composition/{slack_setup,extension_lifecycle,bundled_skills,llm_key_store,product_auth_durable/*}.rs`, `ironclaw_resources/src/cas_snapshot.rs`, `ironclaw_reborn/src/subagent/goal_store.rs`, `ironclaw_approvals/src/capability_permission.rs`) ‚Äî none need CAS, none should have their blind-delete semantics touched. Forcing a `CasExpectation` onto all of them is exactly the blast-radius churn this design avoids (`P1.0b` is scoped to await-edges/roster only).

**Error taxonomy is NOT inherited from `put` ‚Äî it's a new, small diagnosis branch.** `put`'s `Version` arm never emits `NotFound` for an absent row: `postgres.rs`'s `diagnose_put_failure` (`postgres.rs:1461-1477`) and `libsql.rs`'s `Version(expected)` arm (`libsql.rs:1280-1287` ‚Äî not the `Absent`-arm conflict branch at :1243-1250) both collapse "row absent" and "row present at wrong version" into the same `VersionMismatch{found: None}`; the in-memory `check_cas` helper (`in_memory.rs:514-534`) does the same ‚Äî its `(CasExpectation::Version(expected), found)` catch-all arm returns `VersionMismatch{expected: Some(expected), found}` regardless of whether `found` is `None` (absent) or `Some` (wrong version). That collapse is correct for `put` (an absent row on write just means "create it"), but wrong for `delete_if_version`, where the caller needs to tell "already gone" (`NotFound`, benign per ¬ß2) apart from "gone stale, don't touch it" (`VersionMismatch`). **`delete_if_version` therefore implements its own two-branch diagnosis on the 0-rows/absent case, per backend: read current state first (or follow up a failed conditional delete with a read) ‚Äî row absent ‚Üí `NotFound`; row present at a different version ‚Üí `VersionMismatch`.** This is new logic sized to the one new method, not a reuse of `put`'s diagnosis. **Narrowed parameter simplifies this further** ‚Äî since `delete_if_version` never takes an `Any`, its diagnosis is always exactly these two branches, no third "unconditional" case to special-case: row absent ‚Üí `NotFound`; row present at a version other than the caller's `expected_version` ‚Üí `VersionMismatch{expected: Some(expected_version), found: Some(found_version)}`.

**Shape, per backend:**
- Default trait impl: `Unsupported`, same pattern as `put`'s default.
- `in_memory.rs` (new fn beside :134, the `delete` impl): look up the current version before removing ‚Äî `None` ‚Üí `NotFound`; `Some(v)` where `v != expected_version` ‚Üí `VersionMismatch{expected: Some(expected_version), found: Some(v)}`; matching version ‚Üí remove, no mutation on either error branch. Cannot reuse `check_cas` as-is (its catch-all arm doesn't distinguish absent from wrong-version, per above) ‚Äî the two-branch check is written directly in the new fn.
- `libsql.rs` (new fn beside :871, the `delete` impl): `DELETE ... WHERE path = ?1 AND version = ?2` bound to the caller's `expected_version`; 0 rows ‚Üí a follow-up `SELECT version` distinguishes `NotFound` (no row) from `VersionMismatch` (row exists at a different version) ‚Äî a new two-step diagnosis for this method, not the reused `put` shape. No unconditional-delete arm ‚Äî that's `delete(path)`, a separate existing method.
- `postgres.rs` (new fn beside :664): same `AND version = $2` addition bound to `expected_version`, with its own follow-up `SELECT version` on 0-rows to distinguish `NotFound` from `VersionMismatch` ‚Äî `version` column already exists, read by `get`/`query` (`postgres.rs:442,482-489,1009-1030`), but the diagnosis branch itself is new.
- `ScopedFilesystem` (new wrapper beside `delete` at `scoped.rs:526`): pure passthrough to `self.root.delete_if_version(path, expected_version)`.
- `CompositeRootFilesystem` (new fn beside `delete` at `catalog.rs:360`): `self.matching_mount(path)?.backend.delete_if_version(path, expected_version)` ‚Äî pure dispatch.
- `HsmBackend` (new fn beside `delete` at `hsm.rs:111`): pure delegation to the inner backend (`hsm.rs:115-122`) ‚Äî **in scope, not a placeholder skip.** `HsmBackend::declared_capabilities()` already advertises `Delete` + CAS; falling through to the trait default `Unsupported` would pass mount-time capability validation but fail at call time, so it must delegate like every other unified-entry-plane method on this wrapper.
- `MountScopedRootFilesystem` (`crates/ironclaw_host_runtime/src/invocation_services.rs:435`, beside its `delete` override at :430) and `SkillManagementRootFilesystem` (`crates/ironclaw_skills/src/management.rs:148`, beside its `delete` override at :144) ‚Äî both are decorators that forward `capabilities()` to their inner backend, so a CAS-capable mount under either one needs the forward for `delete_if_version` too, or it silently falls through to `Unsupported` despite the backend underneath supporting it. Pure passthrough, same pattern as `ScopedFilesystem`/`CompositeRootFilesystem` above.
- **Out of scope:** `LocalFilesystem::delete` (`local.rs:354`) ‚Äî `LocalFilesystem::put` already rejects `CasExpectation::Version` as `Unsupported` (`local.rs:201-206`, no per-path versioning), so it was never viable for CAS'd edges. `StorageTxn::delete` (`backend.rs:44`) is the separate multi-key-transaction mechanism; edges are single-key CAS-only ("Stores must always work with CAS... as the floor," `root.rs:33`) and never open a `StorageTxn` ‚Äî unaffected.

**Consequences:** (a) **Edge close** = CAS state-to-terminal, then drive `reservation_release` to `Released` (¬ß5.5), then `delete_if_version(edge_path, released_version)` ‚Äî `released_version` is the version returned by that `Released` CAS itself, per ¬ß2's round-4 fix; ordering in ¬ß2/¬ß5.5. (b) **Roster prune** (¬ß4.5) = boot reads the roster marker's current version at the instant it observes the scope's open-edge dir empty, then `delete_if_version(roster_path, that_version)`. A concurrent spawn writing a new edge for that scope bumps the marker's version via its own idempotent-but-version-bumping upsert, so the prune's CAS fails with `VersionMismatch` and aborts ‚Äî no TOCTOU loss of a roster entry newly needed.

**Residual race, closed by two symmetric compensating checks.** ¬ß4.5's roster-before-edge order is never violated by this race ‚Äî the roster marker for a scope is always written before that scope's first edge. The residual race is purely about **boot's read-then-delete window landing inside the gap between a scope's roster upsert and its first edge write**, for a scope currently in the middle of its very first spawn: (1) spawn upserts the roster marker (version bump) ‚Äî roster-before-edge holds; (2) boot's recovery pass lists that scope's open-edge dir and finds it still empty (the edge hasn't been written yet); (3) boot reads the roster marker's *current*, already-bumped version at that same moment; (4) spawn now writes its first edge for the scope; (5) boot's `delete_if_version(roster_path, expected_version = the version read in step 3)` **succeeds** ‚Äî nothing bumped the version again between steps 3 and 5, so the version boot read is still current. The result: a real edge now exists for the scope, but its roster marker is gone. Two compensating checks close this, symmetric in time: **(i) spawn self-heal** ‚Äî after writing the first edge for a scope, spawn re-reads the roster marker; `NotFound` triggers an idempotent re-put (`CasExpectation::Absent`; a `VersionMismatch` on that re-put means someone else already restored it, which is fine ‚Äî this re-put is a `put`, so it still uses `CasExpectation`, unaffected by `delete_if_version`'s narrower parameter). **(ii) boot prune self-check** ‚Äî immediately after a successful `delete_if_version` of a roster marker, boot re-lists that scope's open-edge dir; if it is now non-empty, boot restores the marker via the same idempotent re-put. The two checks are complementary in time ‚Äî whichever of them runs later than the actual edge write is the one that observes and heals the stranded state, so every interleaving converges to a live roster entry; a subsequent boot pass sees a non-empty dir and never attempts the delete at all.

**Generalizes the ABA caller invariant (`docs/reborn/contracts/filesystem.md` ¬ß14.1).** Version tokens are not generation-stable ‚Äî a path's version restarts at 1 on a fresh `put` after a prior delete, so a token captured before a delete+recreate cycle can match a different incarnation of the same path (ABA). **Edge files are structurally immune**: a `(parent_run_id, child_run_id)` pair never recurs, so an edge path is never recreated and no token can go stale-in-a-way-that-matches. **Roster markers are recreated** (the same scope's marker is deleted and later re-put across its lifetime) ‚Äî exactly the case that invariant warns about. But check (ii) above is not a race-window bound that more concurrent activity could defeat; it is a **postcondition recheck**: the re-`list_dir` fires after *any* successful `delete_if_version` on the roster marker, unconditionally, regardless of how stale or fresh the token that authorized the delete was. That is what makes it converge across arbitrarily many intervening delete-recreate cycles for the same scope ‚Äî the check never trusts the delete's success as final; it always re-observes real state immediately after, covering ABA without needing a generation-tracking token.

**Leaf-key note.** Edge and roster files are leaf keys with no children, so `delete`'s cascade semantics (the `... OR path LIKE ...` subtree sweep visible in `libsql.rs:875` and mirrored in the in-memory/postgres backends) are immaterial here ‚Äî `delete_if_version` is pinned to single-key semantics only; it never needs subtree-delete behavior.

**Required tests (P1.0b) ‚Äî crate-tier** (`RootFilesystem`/backend primitive, no production caller to drive through yet): dual-backend parity: (1) correct-version delete succeeds; (2) stale version ‚Üí `VersionMismatch`, no mutation; (3) missing path ‚Üí `NotFound`; (4) roster-prune-vs-concurrent-spawn race, split into (4a) and (4b) per the residual-race ruling above ‚Äî (4a) version-mismatch abort: a concurrent version bump lands before prune's delete call ‚Üí prune's CAS fails, roster entry survives untouched; (4b) CAS-delete-succeeds-but-edge-lands-anyway: for a scope's very first spawn, the roster upsert lands first (roster-before-edge preserved), then boot's `list_dir` observes the still-empty open-edge dir and reads the already-bumped marker version, then the spawn writes its first edge ‚Äî prune's `delete_if_version(roster_path, expected_version = the version read)` succeeds because nothing bumped the version again in between, stranding a roster marker a now-real edge needs; the test asserts convergence to a restored roster entry via either compensating path (integration-tier, since (4b) exercises both spawn and boot in the same race window). This test **is** the design's ABA regression ‚Äî the roster marker is the one recreated path type in this design, and (4b)'s convergence assertion is exactly the postcondition-recheck property the ABA-invariant generalization above depends on. **P1.0b also inherits `docs/reborn/contracts/filesystem.md` ¬ß15's `delete_if_version` cases** (default-trait-impl backend ‚Üí `Unsupported`; denied when the mount lacks `delete` permission) ‚Äî not restated here to avoid drift between the two docs, but part of the same task's acceptance. **Until P1.0b lands, nothing else in PR1 merges** (sibling gate alongside P1.0, ¬ß4.4).

The optional-axis segments use the `{some/<v>|none}` **nested-directory** encoding, the same shape as the existing `local_trigger_access` precedent (`crates/ironclaw_reborn/src/local_trigger_access/filesystem.rs:403-419`, `optional_axis_path`); the goal store (`crates/ironclaw_reborn/src/subagent/goal_store.rs:213-228`) independently establishes the precedent of inserting `agents/<agent_id>`/`projects/<project_id>` segments in-path for these same two optional `TurnScope` axes (its variant omits the segment entirely when absent rather than encoding `none` ‚Äî the edge path here uses the `{some|none}` form instead, so every path has a fixed, unambiguous segment count a **scope-prefixed** `list_dir` can enumerate without branching on presence). **The scope roster (¬ß4.5) encodes the same two optional axes but deliberately does NOT use this nested form** ‚Äî ¬ß4.5's flattened, single-segment filename encoding diverges here on purpose, because the roster (unlike the edge tree) must be enumerable by one **global** `list_dir`, not a scope-prefixed one; see ¬ß4.5's "Asymmetry with edge paths" note for why. Deleted (via `delete_if_version`, ¬ß4.0) on drain/abandon-final (¬ß2).

**4.3 Boot-recovery driver: roster- and store-driven.** Enumerate the scope roster (¬ß4.5), then per scope `list_parents_with_unclosed_edges(scope)` ‚Äî a plain bounded, **scope-isolated** `list_dir` under that scope's **full-axis prefix** (¬ß4.2's `agents/{some/<agent>|none}/projects/{some/<project>|none}` segments, not just the tenant/user-mounted `/turns/subagent-await-edges` root) (¬ß2, ¬ß4.5a ‚Äî this only ever lists that one scope's mounted-**and**-axis-qualified subtree), **plus** the terminal-but-undeleted sweep from ¬ß2's crash-ordering ruling. No `TurnStateStore` active-run query.

**Recovery concurrency is bounded, not one-scope-at-a-time nor unbounded fan-out ‚Äî and the bound is shared across both recovery paths (round-4 fix).** A single global limiter, `BOOT_RECOVERY_MAX_CONCURRENT_SCOPES = 4`, gates scope-recovery work regardless of which path enqueued it: the boot pass walking the roster, and the lazy per-scope backstop (¬ß5.3) starting a recovery task for an unbooted scope on first touch. At most 4 scope-resolves run concurrently across both origins combined ‚Äî boot does not get its own separate pool, and a burst of lazy first-touches does not get an unbounded one. This bounds backend/connection load during a cold-boot recovery sweep *and* during a lazy-recovery burst, regardless of how many scopes are pending from either source.

PR1's merge gate is P1.0 (and P1.0b, ¬ß4.0) done** ‚Äî not "upstream fixes it eventually."

**4.5 Scope roster + scope-key encoding.** Boot/lazy recovery needs which `(tenant, user, agent, project)` scopes have unclosed edges without a global walk (why a global walk isn't available at all: ¬ß4.5a). At first edge write for a scope, spawn idempotently writes a roster marker under `ResourceScope::system()` (`crates/ironclaw_host_api/src/resource.rs:28,112`; used this way by `ironclaw_conversations/src/filesystem_store.rs:203,237`, `ironclaw_reborn_composition/src/llm_key_store.rs:116`).

**Encoding ‚Äî flattened, single path segment (round-3 fix).** Round-2's `{some/<v>|none}` nested-directory encoding ‚Äî reusing `crates/ironclaw_reborn/src/local_trigger_access/filesystem.rs:403-419`'s `(agent_id, project_id, user_id)` key shape, one directory level per axis ‚Äî turns out to be the wrong shape for the roster specifically: the roster is the one artifact in this design that boot must enumerate with a **single global `list_dir`** (¬ß4.3), and nested markers put different scopes' entries behind different intermediate directories that one `list_dir` call cannot see past its immediate children. A single `list_dir(/turns/subagent-await-scopes/)` under the nested form would only discover scopes whose markers happen to be direct children of the roster root ‚Äî every scope nested one or more levels deeper (i.e. almost every real scope) is silently missed by boot recovery.

**Ruling: the roster flattens to one file per scope directly under `/turns/subagent-await-scopes/`, filename = the full encoded scope key in one path segment.** The optional-axis encoding is preserved conceptually (`{some/<v>|none}` per axis) but adapted from nested directories into `__`-delimited filename segments: `/turns/subagent-await-scopes/<tenant_id>__<user_id>__<agent_segment>__<project_segment>.json`, where `<agent_segment>` is `agent-<agent_id>` when the scope carries an agent or the literal `agent-none` otherwise, and `<project_segment>` is `project-<project_id>` when the scope carries a project or the literal `project-none` otherwise. `tenant_id` stays in-path for the same reason as before ‚Äî the roster lives under the system sentinel (constant `__system__`/`__system__` target, verified `ironclaw_reborn_composition/src/lib.rs:718-760`), so one predictable global tree needs tenant encoded in the key itself. **The project axis matches ¬ß4.2's edge path** ‚Äî both the roster key and the edge path partition on the same four axes (`tenant`, `user`, `agent`, `project`); a roster entry that only encoded `agent` while the edge path also varied by `project` would let two different-project scopes for the same agent collide onto one roster marker.

**`__` is a delimiter, not a safe one, without escaping (round-4 fix).** Round-3's claim that "ids cannot themselves contain `__` by construction" is false: `validate_scope_id` (`crates/ironclaw_host_api/src/ids.rs:18-51`, the sole validator behind `TenantId`/`UserId`/`AgentId`/`ProjectId`) only rejects empty values, length > 256 bytes, the exact segments `.`/`..`, any `/` or `\`, and control characters ‚Äî it has no underscore restriction at all, so `_` and `__` are legal anywhere in an id (confirmed by the regression test at `crates/ironclaw_host_api/tests/host_api_contract.rs:200-223`, which never tests `__`). Naive `__`-joining is therefore genuinely ambiguous: `(tenant_id="a__b", user_id="c")` and `(tenant_id="a", user_id="b__c")` are two distinct scope tuples whose `tenant_id`/`user_id` pair both concatenate to the identical `a__b__c` prefix before the agent/project segments are even appended ‚Äî a real filename collision, not a hypothetical one.

**Ruling: each of the four filename components is percent-encoded independently before the `__` join.** Encoding: for each component (`tenant_id`, `user_id`, the agent segment, the project segment), every byte outside `[A-Za-z0-9.-]` ‚Äî which includes the underscore itself ‚Äî is replaced by `%` followed by its two-digit uppercase hex value (so a literal `_` in any id becomes `%5F`); the four escaped components are then joined with the literal, unescaped `__` exactly as before, so the on-disk shape (`<tenant_id>__<user_id>__<agent_segment>__<project_segment>.json`) is unchanged, only the byte content of each segment. Decoding reverses this: split the filename on `__` into exactly four parts ‚Äî safe now because `__` can never occur *inside* an escaped component, only *between* them ‚Äî then percent-decode each part back to its original id. This is the simplest reversible scheme available (no length-prefix framing, no new dependency) and it generalizes past `__` collisions to any non-path-safe byte `validate_scope_id` still permits (spaces, non-ASCII, `%` itself, etc.).

**Required test (fold into P1.6a/P1.6b), unit-tier**, `roster_key_encoding_disambiguates_naive_join_collision`: encode the two colliding tuples above ‚Äî `(tenant_id="a__b", user_id="c", agent=None, project=None)` and `(tenant_id="a", user_id="b__c", agent=None, project=None)` ‚Äî and assert the two produced filenames are distinct (`a%5F%5Fb__c__agent-none__project-none.json` vs `a__b%5F%5Fc__agent-none__project-none.json`), and that decoding each filename recovers its own original tuple exactly, not the other's.

**Payload vs. filename.** The marker's **payload** still carries the full structured `(tenant_id, user_id, agent_id, project_id)` scope ‚Äî that payload, not the filename, is the source of truth boot uses to construct the `ResourceScope`/`TurnScope` it mounts and walks. The filename exists purely as the enumeration key so a flat `list_dir` can discover every marker without parsing structure out of the name; parsing the filename back into components is a convenience for tooling/debugging only, never load-bearing for correctness.

**Asymmetry with edge paths, and why.** Await-edge paths (¬ß4.2) stay **nested** under their per-scope, per-axis directories ‚Äî unchanged, and correct ‚Äî because edges are always listed **per-scope** (`list_parents_with_unclosed_edges(scope)`, ¬ß4.3), never globally; only the roster is listed globally, which is exactly why it ‚Äî and only it ‚Äî must flatten.

**Required test (fold into P1.6a/P1.6b), integration-tier:** write roster markers for at least three scopes differing across every axis (different tenant; different user, same tenant; same user, different agent; same agent, different project) ‚Äî assert a single `list_dir(/turns/subagent-await-scopes/)` call enumerates all of them (regression test for the round-2 nested-marker miss described above).

**Write ordering:** the roster marker is written **before** the first edge file for that scope, never after. A crash between the two leaves a roster **superset** ‚Äî harmless, since boot finds an empty open-edge dir and the CAS'd prune (¬ß4.0(b)) removes the stale entry next pass. The unsafe ordering (edge first, roster after) would let a crash hide a real open edge from boot recovery entirely ‚Äî never chosen.

Boot enumerates the roster (`list_dir` on `/turns/subagent-await-scopes/`); lazy per-scope backstop (¬ß5.3) stays as roster-miss safety net. **Dual-backend:** `list_dir` verified on all three backends ‚Äî `postgres.rs:639`, `libsql.rs:830`, `in_memory.rs:197`.

**Roster cardinality.** One marker per `(tenant, user, agent, project)` scope with at least one unclosed edge ‚Äî cardinality tracks **active-subagent scopes**, not edge count (a scope with 16 open edges is still one roster entry). Roster `list_dir` enumeration may paginate per the backend's ordinary `list_dir` semantics ‚Äî no full-listing-in-one-call assumption is baked into this design. Tenant sharding (¬ß12 tripwire 2) remains the escape hatch if a single tenant's roster subtree itself grows pathological; the ¬ß4.3 concurrency bound above is committed now, independently of that tripwire.

**4.5a Scope-aware construction ‚Äî no fixed-view.** The await-edge store and the scope roster are both built on the **same single shared `ScopedFilesystem` handle**, constructed *once* via `crate::wrap_scoped(root)` ‚Üí `ScopedFilesystem::new(root, invocation_mount_view)` (`crates/ironclaw_reborn_composition/src/lib.rs:848-852`; resolver `invocation_mount_view` at `:753-763`, alias table `PER_USER_ALIASES` ‚Äî including `/turns` ‚Äî at `:718-733`) ‚Äî **never** via `ScopedFilesystem::with_fixed_view`. Every op (`put`/`get`/`list_dir`/`delete_if_version`) takes the caller's live `ResourceScope` as an explicit argument (`scoped.rs:216-233,487,526`); the resolver recomputes the `MountView` for *that* scope on *that* call. This is the same pattern already load-bearing in production for `ironclaw_conversations::filesystem_store` (`.get(&scope, &path)` / `.put`, re-evaluating a fresh scope per call at `filesystem_store.rs:205,238`) and `ironclaw_reborn_composition::llm_key_store` (`store.put(scope(), ...)`, `scope()` re-evaluated per call at `llm_key_store.rs:39,115-116`) ‚Äî neither constructs a filesystem baked to one scope.

**Anti-pattern, named explicitly:** `owner_turn_state_filesystem` (`crates/ironclaw_reborn_composition/src/factory.rs:1927-1938`, used in production at `:2079,:4314`) builds a **`ScopedFilesystem::with_fixed_view`** baked to a single `owner_scope` at composition time ‚Äî a single-boot-owner view. That is the exact bug class behind #5720/#5721 (per-user `/turns` state collapses onto one owner on multi-user boxes). **New await-edge/roster code must not copy this constructor** ‚Äî it must be built the `wrap_scoped`/`invocation_mount_view` way: one process-wide handle, scope threaded per call, never per-scope-baked at construction. (The only other `with_fixed_view` hit under `subagent/` ‚Äî `factory.rs:5011`'s `subagent_goal_filesystem` ‚Äî is test-only fixture code inside `mod tests` opening at line 4647; production's goal store at `factory.rs:2232` is correctly built via `wrap_scoped`.)

**Consequence ‚Äî ¬ß4.2's canonical path is scope-relative, not scope-agnostic ‚Äî and now axis-relative too.** `ResourceScope` (`crates/ironclaw_host_api/src/resource.rs:31`) carries only `tenant_id`/`user_id` ‚Äî it has no `agent_id`/`project_id` field at all ‚Äî and `invocation_mount_view` (`ironclaw_reborn_composition/src/lib.rs:718-763`) rewrites `PER_USER_ALIASES` (including `/turns`) using only those two fields. **The mount alone cannot isolate two agents (or two projects) of the same user** ‚Äî that isolation has to come from the path itself, which is exactly what ¬ß4.2's `agents/{some/<agent>|none}/projects/{some/<project>|none}` in-path segments provide. `/turns/subagent-await-edges/agents/{some/<agent>|none}/projects/{some/<project>|none}/<parent_run_id>/<child_run_id>.json` is a `ScopedPath` alias; the physical `VirtualPath` it resolves to is `/tenants/<tenant>/users/<user>/turns/subagent-await-edges/agents/{some/<agent>|none}/projects/{some/<project>|none}/<parent_run_id>/<child_run_id>.json` (`/turns` is tenant/user-rewritten ‚Äî it is one of the `PER_USER_ALIASES`; the agent/project segments are ordinary path segments beneath the alias, untouched by that rewrite). Two different `TurnScope`s' edges for the *same* `parent_run_id`/`child_run_id` pair therefore land in physically distinct trees on the backend whenever **any** of tenant, user, agent, or project differs ‚Äî not just tenant/user. That is what makes `list_parents_with_unclosed_edges(scope)` (¬ß4.3) a plain, bounded, **scope-isolated** `list_dir` under `/turns/subagent-await-edges/agents/{...}/projects/{...}` for that scope's own agent/project axis values: it only ever lists that one scope's mounted-**and**-axis-qualified subtree, never a global one and never a sibling agent's or project's edges under the same user. The scope roster (¬ß4.5) exists precisely because that isolation means boot cannot discover *which* scopes have edges without a scope-agnostic index ‚Äî the roster marker (itself written under the system sentinel scope, so its own listing genuinely *is* global, and now carrying the same agent/project axes per ¬ß4.5) is what tells boot which real `(tenant, user, agent, project)` scopes to mount-and-walk one at a time.

**Acceptance tests (fold into P1.6a/P1.6c) ‚Äî integration-tier** (drive the real composed `ScopedFilesystem`/`invocation_mount_view` stack): (a) **two-users-distinct-paths** ‚Äî write an await-edge for two different `ResourceScope`s (different `user_id`; same or different `tenant_id`) using the same `parent_run_id`/`child_run_id`, then assert ‚Äî read back at the filesystem layer, not just via two `ScopedFilesystem` calls that could coincidentally agree ‚Äî that the two writes landed at two distinct physical paths (`/tenants/<A>/users/<A>/turns/...` vs `/tenants/<B>/users/<B>/turns/...`); (b) **two-agents-same-user-distinct-paths** ‚Äî same tenant, same user, two different `agent_id` values (or one `Some` and one `None`) on the `TurnScope`, same `parent_run_id`/`child_run_id` ‚Äî assert the two writes land at two distinct physical paths differing only in the `agents/{...}` segment, and that `list_parents_with_unclosed_edges(scope)` for one agent's scope never enumerates the other agent's edge.

## 5. Core delivery mechanics

The following are carried forward unchanged from the prior durability-design lineage (restated here so this doc is self-contained; ¬ß5.5 and ¬ß5.6 are new).

**5.1 Capacity: reservation is the cap, listing advisory.** Unchanged. `SpawnTreeReservation` (depth ‚â§ 1, ‚â§ 4 spawns/turn, ‚â§ 16 descendants/tree) is the sole admission mechanism.

**Made explicit: the 16-descendant cap counts non-terminal children ‚Äî deliberately, including gate-parked ones.** A child sitting on `BlockedApproval`/`BlockedAuth` still holds its reservation slot. The cap bounds **outstanding side-effect obligations** ‚Äî 16 approval-parked children is 16 pending sensitive actions paging one human ‚Äî not compute. Compute concurrency is excluded on a separate, already-existing axis: the scheduler deschedules a blocked run (no worker slot held), so a tree full of gate-parked children costs zero live workers, only zero remaining reservation headroom. These are deliberately different budgets; this design does not collapse them.

**Never release-on-park; release only at terminal (¬ß5.5); never re-claim on ordinary resume.** A child resuming past its own gate (approval granted, auth completed) is not a re-admission event ‚Äî it already holds its slot from spawn, uninterrupted. **Ruling: resumption must never require re-admission.** An approved child that can't resume because the tree filled up while it waited would be a deadlock-by-design (the human who just unblocked it discovers the unblock didn't work) ‚Äî never chosen.

**Relief valves, when a tree does fill:** `subagent_inspect` (¬ß7 PR3) shows which children are blocked and why; `subagent_cancel` (¬ß7 PR6) reclaims slots by driving a child to terminal early (terminal ‚Üí release, ¬ß5.5); a human resolving the gate queue directly (¬ß9.2) drains blocked children the ordinary way, which also releases their slots. `ironclaw-reborn subagent edges` (¬ß5.4) reports per-scope open-reservation counts alongside edge counts, so an operator sees a tree approaching the cap before it deadlocks.

**`subagent_extend` is the one re-claim path ‚Äî by design, not by exception.** Unlike ordinary resume, `subagent_extend` (¬ß6, ¬ß7 PR4) re-activates an *already-terminal* child (¬ß0 requirement 3, "no cold start"). A terminal child already released its slot (¬ß5.5: `Claimed ‚Üí Released` on terminal). Re-activating it is therefore a **fresh outstanding obligation**, not a continuation ‚Äî `subagent_extend`'s `activate()` call re-claims a descendant-reservation slot as a normal admission check, and fails with the existing capacity error (the same one spawn uses) if the tree is already full. This is the only place in the design where activating a child re-runs admission.

Required tests (**P4.4**): extend on a full tree ‚Üí capacity rejection (integration-tier ‚Äî drives extend through the real `activate()`/reservation admission path); extend releases its slot at the new run's terminal (crate-tier is sufficient for the reservation-accounting half).

**5.2 Two-layer exactly-once, `InvalidTransition` discriminator.** Layer 1: CAS single-winner per transition. Layer 2: `resume_turn`'s durable idempotency replay (`crates/ironclaw_turns/src/memory/mod.rs:187,1031-1046,2248-2269`, CAS'd in the same snapshot as every run-state transition). No `resume_dispatched_at` field needed.

**Boot/drain contract, matched on `from` ‚Äî no wildcard.** `resume_turn_once` (`memory/mod.rs:2363-2384`) sets `from: record.status.get()` when `BlockedDependentRunGate`'s required status (`BlockedDependentRun`, `request.rs:48`) mismatches, raising `InvalidTransition{from, to: Queued}`.

- **Benign `already_closed`:** `from ‚àà {Queued, Running, Completed}`.
- **`ResolveReport.failed`, surfaced:** everything else. No wildcard match.

Required tests (**P1.3**), crate-tier (`InvalidTransition` discriminator + CAS layer are pure state-machine logic, no external seam to drive through): (a) double-settle race ‚Üí one resume; (b) crash-after-settle-before-resume ‚Üí one resume via durable idempotency key; (c) crash-after-resume-before-drain ‚Üí `InvalidTransition{from: Queued|Running}`-as-`already_closed` lets drain complete without a second resume.

**5.3 Boot enumeration: roster-driven + lazy backstop.** Eager: background boot task walks roster (¬ß4.5), resolves unclosed edges per scope, never blocks foreground. Lazy: a spawn/activate against an unbooted scope does **not** run scoped recovery inline on the caller's path ‚Äî it starts-or-joins the single in-flight scoped-recovery task for that scope (`in_progress` guard dedupes concurrent triggers onto one task) and immediately returns a retryable rejection, `ScopeRecoveryInProgress { retry_after_hint }` ‚Äî named and treated by callers exactly like `ThreadBusy` (retry/backoff, no special-casing at call sites). This preserves the foreground latency guarantee: no caller ever blocks on a scoped recovery walk. Once that scope's recovery task completes, the next spawn/activate against it is admitted normally.

**Distinct from `ThreadBusy`/`ThreadReserved` (¬ß6a) in granularity, not just name.** `ScopeRecoveryInProgress` is a **scope-wide** rejection ‚Äî it can affect every thread/caller under that scope at once, tied to the background recovery task's lifetime, not to any single thread's run state ‚Äî whereas `ThreadBusy` and `ThreadReserved` are **per-thread**. Callers still retry/back off identically for all three (no special-casing at call sites), so this doesn't change handling; it only means a human reading logs should not conflate "this one thread is busy" with "this whole scope is still recovering." **Lazy per-scope recovery tasks triggered by first-touch admission share the same bounded limiter as the boot pass (round-4 fix)** ‚Äî `BOOT_RECOVERY_MAX_CONCURRENT_SCOPES = 4` (¬ß4.3) governs the background scope-recovery work regardless of which path enqueued it, one global limiter for both. A burst of first-touch calls across many distinct unbooted scopes still dedupes to one recovery task per scope via the `in_progress` guard, and each such task competes for the same 4-wide bound boot uses ‚Äî it never gets a separate or unbounded pool. **The bound applies only to the background recovery work, never to the foreground admission call**: `activate()`/spawn admission against a scope with an in-flight or now-queued recovery task still returns `ScopeRecoveryInProgress` immediately (the opening ruling above) ‚Äî the caller is never made to wait on the limiter itself; only the recovery task it triggered queues behind whatever else is already occupying the 4 slots.

**Required test (P1.9 extension), integration-tier** (drives spawn/activate admission through the real lazy-backstop path against an unbooted scope): admission attempted against a scope with an in-flight scoped recovery ‚Üí `ScopeRecoveryInProgress` retryable rejection; retried after the recovery task completes ‚Üí admitted normally. **Extend the same test (round-4 fix) for the shared limiter:** with `BOOT_RECOVERY_MAX_CONCURRENT_SCOPES` slots already saturated by a mix of boot-triggered and lazy-triggered recovery tasks, a new lazy first-touch against a distinct unbooted scope still returns `ScopeRecoveryInProgress` immediately (never blocks on the limiter) while its recovery task queues behind the others; once a slot frees, the queued task runs and that scope becomes admissible in turn.

**5.4 Observability.** `ResolveReport { resumed, drained, abandoned, already_closed, failed }` per scope, `debug!` + counters. Under ¬ß2, `drained`/`abandoned` now mean "resumed/finalized **and the edge file deleted**." `failed > 0` is actionable. `ironclaw-reborn subagent edges [--scope ‚Ä¶]` lists unclosed edges off the roster. **Terminal byte-length accounting:** today's `AwaitedChildState.terminal_byte_len` (`gate_resolution.rs:27`) is set by `record_terminal_byte_len` (`:186-198`) as a best-effort **second** write *after* `mark_terminal_result_written` in `write_terminal_result` (`completion_observer.rs:497-518`) ‚Äî **moves onto the edge as an additive `terminal_byte_len: Option<u64>` field, set in the same CAS write that transitions the edge to terminal.** Strictly better than today (folds two mutations that could drift into one atomic write). Still feeds `ByteCapStrategy` observability only, never durability ‚Äî a missing value is logged at `debug!` and never blocks delivery.

## 5.5 Descendant-reservation release tri-state on the edge

**Gap:** the single-winner release guard for `SpawnTreeReservation` capacity lives entirely on the dying store ‚Äî `descendant_reservation_released` / `descendant_reservation_release_claimed` fields plus `claim_descendant_reservation_release`/`mark_descendant_reservation_released`/`release_descendant_reservation_claim` (`gate_resolution.rs:334-379`, exact range). ¬ß3's replacement inventory lists this as "surviving" logic without saying where its state lives once the store dies ‚Äî this section closes that gap.

**Ruling: the tri-state moves onto the `AwaitEdge`'s own state payload, CAS'd like every other transition ‚Äî no separate store, no new file.** Add `reservation_release: ReservationReleaseState` (`Unclaimed | Claimed | Released`):
- `Unclaimed ‚Üí Claimed`: CAS'd (`CasExpectation::Version(edge_version)`) exactly like today's in-memory check-then-set; only the CAS winner releases `SpawnTreeReservation` capacity.
- `Claimed ‚Üí Released`: CAS'd after the capacity-release call succeeds, mirroring `mark_descendant_reservation_released`.
- A failed release attempt CAS's back `Claimed ‚Üí Unclaimed`, mirroring `release_descendant_reservation_claim`'s retry-unlock, so a transient failure doesn't permanently strand the reservation.

Lives on the **same edge file** as settle/drain state ‚Äî one more field, one more CAS'd path, not a second file ‚Äî so it's deleted with the edge on drain/abandon (¬ß2).

**Boot recovery also owns the claim-but-not-yet-released window.** An edge that crashed between the `Unclaimed ‚Üí Claimed` CAS and the capacity-release call ‚Äî i.e. scanned during recovery and found sitting in `Claimed` ‚Äî is retried by the same boot/lazy recovery pass (¬ß4.3/¬ß5.3) that resolves every other unclosed edge; the `Claimed ‚Üí Released` (or its failure-path retry-unlock back to `Unclaimed`) is just another CAS attempt the recovery walk makes, no special-cased recovery path needed.

**Ordering with edge deletion.** An edge may be deleted (¬ß2, ¬ß4.0) **only after** its `reservation_release` field has reached `Released` ‚Äî never while `Unclaimed` or `Claimed`. The full terminal sequence is: settle ‚Üí resume/drain ‚Üí drive `reservation_release` to `Released` ‚Üí `delete_if_version`. Recovery's terminal-but-undeleted sweep (¬ß2's crash-ordering ruling) must honor this same sequencing: on finding a terminal-but-undeleted edge, it first completes the release (retry `Claimed ‚Üí Released`, or the failure-path retry-unlock to `Unclaimed` and re-attempt) and only *then* calls `delete_if_version` ‚Äî a terminal edge whose `reservation_release` isn't yet `Released` is never deleted, so capacity accounting is never lost with no edge left to recover it from.

**The version token for that `delete_if_version` call is the `Released` CAS's own return value (round-4 fix).** Because `reservation_release` lives on the same edge file as every other field, the `Claimed ‚Üí Released` CAS in this sequence bumps the file's `RecordVersion` past whatever version the earlier terminal-state CAS (¬ß2 step (1)) returned ‚Äî that earlier version is stale by the time deletion runs and would fail `VersionMismatch` every time if used. The caller must carry forward the version *this* CAS returns, not any version read or returned earlier in the sequence; recovery, which may itself be the one performing the `Claimed ‚Üí Released` transition, uses the version its own write just returned for the same reason ‚Äî it never reuses a version it captured before starting its recovery pass.

**Required test (P1.9 extension), crate-tier:** (a) double-release race ‚Äî two concurrent resolver instances (crash-retry racing a fresh recovery pass) both attempt to release the same child's reservation; assert capacity is released **exactly once** (the loser's `Unclaimed ‚Üí Claimed` CAS fails, observes `Claimed`/`Released` already, no-ops); (b) boot-recovery-retries-`Claimed` ‚Äî seed an edge already at `Claimed` (simulating the crash window above), run a recovery pass, and assert **both**: it drives the edge to `Released` (or retry-unlocks to `Unclaimed` on a simulated release failure) rather than leaving it stuck, **and** ‚Äî extending the same seeded case rather than a separate test ‚Äî that the edge is deleted only *after* reaching `Released`, never before (the release-before-delete ordering).

## 6. Run-budget floor (derived, not stored)

Cap: **8 consecutive `ParentAgent`-provenance activations per thread.** `Human` resets to 0, never capped. 9th consecutive `ParentAgent` ‚Üí `subagent_extend_budget_exhausted`, sanitized, no internal identifiers cross the model-visible boundary.

**Counter ‚Äî derived at `activate()`, not stored.** `SessionThreadRecord.metadata_json` (`crates/ironclaw_threads/src/contract.rs:184`) has **no mutation path**. Instead: `TurnRunRecord` gets one additive field, `subagent_activation_provenance: Option<ActivationProvenance>` (`ActivationProvenance { Human, ParentAgent, System }` ‚Äî `System` is ¬ß8's background-wake provenance) set once at run creation, immutable thereafter.

`activate()` derives the cap via a **bounded** query, not an open-ended walk: fetch at most the newest **8** non-`System`-provenance run records for the thread (`K = cap = 8`, one indexed/ordered query, `LIMIT K`) ‚Äî enough to decide the cap outright for the **pending** activation being admitted, without ever materializing a 9th `ParentAgent` record: if a `Human` run appears anywhere in that 8-record window, or history has fewer than 8 non-`System` records before its start, the pending activation is admitted (the streak resets or hasn't reached the cap); if all `K` fetched records are `ParentAgent` with no `Human` among them, the pending activation would be the 9th consecutive `ParentAgent` and is refused. `System` runs are excluded from the fetch itself (not filtered post-fetch), so they never count against the `K`-record budget. The constant is tied to the cap by construction (`K = cap`, not `cap + 1` ‚Äî the cap is enforced at admission time on the *next* activation, so history itself never holds more than `cap` consecutive `ParentAgent` records to begin with) ‚Äî raising the 8-activation cap means raising `K` in lockstep, never decoupling them (distinct from ¬ß12 tripwire 4, which is about the separate depth cap).

**Required tests (P4.2), crate-tier** (bounded `LIMIT`-8 query is pure state-machine logic over run records, no external seam to drive through yet): one test row, four assertions ‚Äî (a) under-8 consecutive `ParentAgent` activations are admitted; (b) exactly 8 consecutive `ParentAgent` activations admitted, the 9th refused with `subagent_extend_budget_exhausted`; (c) a `Human` activation anywhere in the window resets the streak to 0; (d) `System` runs are excluded from the fetch and never count toward, nor are ever blocked by, the cap.

Unchanged: per-child SUBAGENT budget (`iteration_limit` 16, made per-flavor-overridable in ¬ß10); `SpawnTreeReservation` cap (16), depth cap (1, depth-agnostic per ¬ß9/¬ß10).

## 6a. Human input priority ‚Äî no content queue

Two mechanisms give a human priority over `ParentAgent`/`System` activation traffic on a thread. **The no-queue non-goal stands**: neither mechanism stores any payload, so none of a content queue's ordering, injection, or budget questions apply ‚Äî a reservation is a marker, not a mailbox.

**(i) Next-slot reservation.** A human whose `activate()` hits `ThreadBusy` may set a `human_waiting` reservation marker on the thread. While set, `activate()` admission refuses `ParentAgent` and `System` provenance with a new `ThreadReserved` outcome ‚Äî callers treat it exactly like `ThreadBusy` (retry/backoff, no special-casing needed at call sites). The marker clears when the human activates (their `activate()` succeeds) or when they explicitly abandon the reservation.

**The reservation is a lease, not an indefinite hold.** A disconnected client that never activates and never abandons would otherwise block `ParentAgent` **and** `System` (¬ß8's settle-wake delivery) forever ‚Äî a starvation bug, not a feature. **Ruling:** the marker carries `owner` + `expires_at` (default **15 minutes**, `HUMAN_RESERVATION_LEASE_TTL`); admission treats an expired lease as absent via a lazy, read-time check (`now > expires_at` at the point `activate()` reads the marker) ‚Äî no timer task, no dedicated background reaper. No sweep over this per-thread marker is needed for correctness: a live `activate()` call re-checking an expired marker is sufficient on its own, so a stale marker left after a client disconnects is inert ‚Äî it simply never blocks anything again once its `expires_at` has passed, whether or not anything ever cleans up the file itself. Human re-activation or explicit abandon still clears the marker immediately, ahead of expiry; renewing the reservation is a re-`put` with a fresh `expires_at`.

**This does not contradict ¬ß12's "no gate expiry timers" non-goal.** That non-goal is about **security-decision gates** (approval/auth) ‚Äî they guard a sensitive action and must park forever until a human decides, never auto-resolve on a clock. The `human_waiting` reservation carries no security content at all ‚Äî it is a UX courtesy hold on scheduling priority, and letting it lapse only returns the thread to ordinary `ParentAgent`/`System` admission, exactly the state it would be in had the human never reserved it. Lapsing a courtesy hold is safe by construction; auto-resolving a gate would not be.

Deferred `System` settle-wakes are not lost while a reservation holds `ParentAgent`/`System` off the thread ‚Äî the existing run-start sweep and boot pass (¬ß8.2, triggers 2 and 3) already heal any settle-wake that loses to `ThreadBusy`/`ThreadReserved`; the settled edge just waits its turn, same as any other `ThreadBusy` deferral.

**Marker storage:** a small marker file, scoped and CAS'd the same way as every other piece of state in this design (¬ß4.0/¬ß4.5's pattern) ‚Äî not a mutation onto `SessionThreadRecord.metadata_json`, which ¬ß6 already establishes has **no mutation path**. Reusing the filesystem-marker pattern this design already relies on for edges and the scope roster is simpler than opening a first mutation path onto `metadata_json` for a feature that doesn't need anything else `metadata_json` carries. Payload: `{ owner: UserId, expires_at: Timestamp }` ‚Äî `owner` identifies the reserving human for audit/UX, `expires_at` backs the lease check above.

Ships with the extend PR (**P4.3**) ‚Äî `activate()` admission is where `ThreadBusy` already lives, so `ThreadReserved` is a sibling admission outcome in the same place. **Required tests ‚Äî integration-tier**: (a) reservation set ‚Üí parent `subagent_extend` refused with `ThreadReserved` ‚Üí human `activate()` succeeds and clears the marker ‚Üí a subsequent parent extend is admitted normally; (b) reservation set with an expired `expires_at` ‚Üí a `ParentAgent` `activate()` is admitted (expired lease treated as absent), with no timer task or reaper involved.

**(ii) Interrupt & take over.** Console-only compose of two existing primitives ‚Äî the underlying run-cancel mechanism (the same one `subagent_cancel`, ¬ß7 PR6, later wraps as a model-facing tool) and `activate(Human)` (¬ß1) ‚Äî no new state, no new admission outcome, and no dependency on PR6's tool-level capability landing first. A human on a child (or parent) console cancels the live run and activates in its place. Ships with the console UI PR (**P5.4**).

## 8. Background delivery = activate

`PostCapabilityStage::drain_settled` (`crates/ironclaw_agent_loop/src/executor/post_capability.rs:36-37`) is **a permanent no-op stub today, not existing behavior to preserve** ‚Äî it unconditionally `Vec::new()`s. Its doc comment names `LoopBackgroundChildPort` as the replacement; that type **does not exist anywhere in `crates/`** ‚Äî verified, it is prose only in `docs/reborn/2026-06-04-subagent-compaction-design.md` (lines 66,83-84,180,275-287,322) plus a doc-comment pointer (#4474). Also verified: `PostCapabilityStage::process` runs on **every** `TurnCompletedStep::Continue` (`post_capability.rs:60-99`) ‚Äî every non-exiting loop iteration, including a freshly-activated run's first (AssistantReply turns "reach here with an empty map" per the existing comment) ‚Äî a fact ¬ß8.2 relies on.

**Ruling: `drain_settled` is NEWLY IMPLEMENTED by an owned PR2 task (P2.4) wiring it to `AwaitEdgeResolver` ‚Äî list settled edges for the live parent, drain each. This supersedes the #4474 stub contract; no `LoopBackgroundChildPort` is ever built.**
- **Live parent:** `drain_settled` drains each iteration (P2.4).
- **Parked/completed parent:** on child settle, resolver calls `activate(parent_thread, input, provenance=System)`. `ThreadBusy` is a benign no-op ‚Äî edge stays `settled`; ¬ß8.2's retry set picks it back up. `System` activations are exempt from ¬ß6's cap, bounded to **one** attempt per settled child ‚Äî the edge's `settled` state is the dedupe.

**Batched drain, not per-edge rescan.** When `drain_settled` (or a recovery/boot pass) finds **multiple** settled edges for the same parent, it must batch all their `result_ref`/`safe_summary` transcript updates into **one** thread-snapshot read + **one** CAS write pass ‚Äî never one rescan-and-CAS-retry cycle per edge. `update_parent_result_reference` ‚Üí `update_tool_result_reference` (¬ß8.1) is the right primitive for a *single* settled edge's CAS-retry closure; the batch caller accumulates the full set of settled `(result_ref, safe_summary)` pairs for the parent first, then invokes the transcript-update helper **once** over that whole set ‚Äî one snapshot read, every matched message rewritten in place, one CAS write ‚Äî rather than calling it `E` times against a thread with `M` messages. This keeps a multi-edge drain O(E+M) instead of O(E√óM); the O(E√óM) shape is exactly what a naive per-edge loop over `update_parent_result_reference` would produce on a run-start sweep or boot pass that finds several children settled at once.

**8.1 Drain-append idempotency.** The framed-content write into the parent ‚Äî `update_parent_result_reference` ‚Üí `update_tool_result_reference` (`crates/ironclaw_threads/src/filesystem_service.rs:1965-2013`) ‚Äî is a **CAS-guarded in-place field update on an already-existing message**, not an append: it rescans for the tool-result-reference message (`matches_tool_result_reference`) then rewrites its `content` via `apply_message_update`'s CAS-retry closure. Calling it twice with the same `result_ref`/`safe_summary` reproduces the same `content` ‚Äî **already idempotent by construction**, verified not asserted. The payload write it follows (`update_capability_result`, from `write_terminal_result`, `completion_observer.rs:494-518`) is guarded today by the in-memory `terminal_result_written` flag; under the edge design that guard **becomes the edge's own CAS state** (write payload once per `settled`‚Üí`drained` transition ‚Äî a crash before that CAS just retries the whole flow, safe precisely because the transcript write is an overwrite, not an append). **Required test ‚Äî integration-tier** (`tests/integration/`; drives the drain path through the real `AwaitEdgeResolver` + thread service, asserting at the transcript-message seam per `.claude/rules/testing.md`, part of P2.4's acceptance): crash after the transcript field-write but before the edge's CAS to `drained` ‚Üí recovery replays the equivalent write ‚Üí assert exactly one tool-result-reference message for that `result_ref`, content unchanged by the replay.

**Required test (P2.4 extension) ‚Äî integration-tier:** drain of N (‚â•2) settled edges for the same parent in one pass performs exactly one thread-snapshot read and one CAS write ‚Äî not N of each ‚Äî asserted via a write-count seam (e.g. a counting wrapper around the thread service's snapshot-read/CAS-write calls, or an equivalent call-count assertion).

**8.2 System-activate retry set.** A settled edge for a parked/live parent is drained by exactly three independent triggers:
1. **Settle-time** `activate(..., System)` attempt (above) ‚Äî may lose to `ThreadBusy` mid-run.
2. **Every run-start sweep** ‚Äî `PostCapabilityStage::process` runs on every `Continue`, including a fresh run's first iteration, so the *next* time this thread runs for **any** reason, `drain_settled` (P2.4) picks up every still-settled edge as a side effect of that run happening at all.
3. **Boot pass** (¬ß4.3/¬ß5.3) ‚Äî roster-driven, independent of any thread activating.

**Invariant:** a `ThreadBusy` at settle-time (trigger 1) is always healed by trigger 2 or 3 ‚Äî a settled edge can never go permanently undrained while its parent thread ever runs again or a boot pass ever occurs. **Required test ‚Äî integration-tier** (`tests/integration/`, same P2.4 acceptance as ¬ß8.1, driven through the live parent thread + boot-pass harness, not the resolver in isolation), named `settled_edge_threadbusy_is_healed_by_run_start_and_boot_pass`: **both** scenarios below are asserted in the same test, not either/or ‚Äî (a) settle a child mid-parent-run (forcing `ThreadBusy` on trigger 1), assert no further `System` attempt (dedup by `settled` state), then let the live run's next iteration drain it (trigger 2) and assert the edge closes; and, as a separate scenario, (b) settle a child mid-parent-run the same way, then instead run a boot pass and assert the roster sweep drains it (trigger 3). **Make the parent-completed precondition explicit** (¬ß2's mode-scoping): the parent run reaching its own terminal state (`Completed`) while the background child is still running and its edge is `open` is not a special case of the above ‚Äî the test asserts the edge stays `open` (never `abandoned`, per ¬ß2) across the parent's own terminal transition, and that the child's later settle still wakes the parked parent thread via trigger 1 (or trigger 3, if the thread never runs again first) and drains normally.

## 9. Gate propagation ‚Äî approval and auth, always to the tree root

Any gate from any descendant, any depth, bubbles to the **tree root's** originating human surface. Covers **auth/credential gates** (`BlockedAuth`) too, not just approvals. **Surfacing is always the tree root's originating surface** ‚Äî one deterministic paging point, unchanged. **Resolution is not the same thing as surfacing**: it is accepted from any surface authenticated as the tree owner, including a child thread's console a human has open directly ‚Äî ¬ß9.2 lifts the old blanket restriction for owner-authenticated humans specifically; it still holds, without exception, for any LLM in the chain.

**Both kinds, one shape ‚Äî no new gate representation.** `TurnBlockedGateMetadata` (`crates/ironclaw_turns/src/events.rs:59-66`) already carries `gate_kind: TurnBlockedGateKind` (`Approval`, `Auth`, `Resource`, `AwaitDependentRun`, `ExternalTool`) plus `credential_requirements`. **Required tests (P2.5/P2.6) ‚Äî integration-tier** (`tests/integration/`; the walk, root-routing, and the Slack poller extension above all cross the descendant‚Üíroot‚Üídelivery-surface seam, not a single-function unit): the depth-agnostic walk resolving a descendant gate to the tree root's delivery surface for each `RunNotificationOrigin` branch (¬ß9.1), and the extended triggered-run delivery poller (`deliver_triggered_run`, above) picking up the walk's resolved gating-run id/kind/gate_ref instead of only the root run's own status. **Add (¬ß9.2):** resolution submitted from a child-console surface resolves the gating run exactly once ‚Äî the second resolution attempt observes the already-resolved CAS state and no-ops. **Add (¬ß9.1, round-3 `LiveSourceRoute` kind-dispatch):** `LiveSourceRoute` + `ApprovalPrompt`/`AuthPrompt` for a tree rooted in a shared-channel conversation ‚Üí resolves to the owner's preference target, not the shared channel.

**Mechanics ‚Äî depth-agnostic:**
1. The gating run parks. No ancestor run touched.
2. Resolver walks `parent_run_id` upward: skip terminal runs, stop at the run whose `parent_run_id` is `None` ‚Äî the root.
3. ¬ß9.1's resolution engine picks the delivery target from the root's scope + origin.
4. Resolution flows back down: the surface's decision resolves the **gating run itself** ‚Äî not the root run.
5. An `abandoned` (deleted, ¬ß2) edge along the chain doesn't break the walk ‚Äî lineage lives on the immutable run record.

**9.2 Resolution accepted from any owner-authenticated surface.** Surfacing and resolution are decoupled. Surfacing (¬ß9's opening ruling) stays exactly as designed ‚Äî the tree root's originating surface is the sole paging point, unconditionally. What changes: the gate **resolution** itself ‚Äî the actual approval/auth decision ‚Äî is accepted from *any* surface authenticated as the tree owner, not only the root's originating surface. This includes a child thread's console a human has directly opened (¬ß1, `activate(Human)`; PR5a/5b once shipped, but the underlying approval-interaction service has no console dependency ‚Äî it is exercisable as soon as any owner-authenticated surface can reach a `gate_ref`, ahead of full ThreadTree UI).

Why this is safe, not a hole in the never-auto-approve floor:
- **The approver must be a human, from any surface.** No LLM anywhere in the chain resolves a gate ‚Äî a subagent (at any depth) referencing its own or a sibling's pending gate still never resolves it. Only lifting this for owner-authenticated humans, never for automation, is what makes it safe.
- **The approval service was never root-scoped, only root-paged.** `crates/ironclaw_product_workflow/src/approval_interaction/service.rs` resolves against `gate_ref`/`run_id` with no thread affinity baked into the decision path ‚Äî root-surfacing was a paging convention layered on top, not a constraint the service itself enforced.
- **Two doors, one CAS'd lock.** Gate resolution is CAS-idempotent, single-winner, the same pattern as every other transition in this design (¬ß5.2, ¬ß5.5). Whichever surface's resolution attempt wins the CAS resolves the gate; the other observes the already-resolved state and no-ops. Two resolution paths onto one lock is exactly the shape CAS'd resolution already handles safely.

**Audit:** the resolution record captures *which* surface actually resolved the gate ‚Äî root or a specific non-root owner surface ‚Äî not just that it was resolved, so a human reviewing the tree afterward can see where the call was made.

## 10. Per-flavor configurability ‚Äî flavors are data, the loop is shared

All 4 launch flavors (`SubagentFlavorId::{General, Explorer, Coder, Planner}`, `crates/ironclaw_reborn/src/subagent/flavors.rs:9-16`) run the **same** loop machinery ‚Äî one `LoopFamily` (`LoopFamilyId::SUBAGENT`, `crates/ironclaw_agent_loop/src/families/subagent.rs`). **A flavor is config, not a fork.**

**(a) Prompt ‚Äî already file-sourced.** `.../subagent/directions/{general,explorer,coder,planner}.md` via `include_str!()` (`directions/mod.rs:22-25`). No change.

**(b) Capability allow/deny ‚Äî reuses existing machinery.** `SubagentCapabilitySurfaceResolver::resolve` (`.../subagent/capability_surface.rs:30-46`) computes `intersect_allow_sets(base, flavor_allowlist)`; deny layered outermost by `CapabilitySurfaceDenyFilter` (`ironclaw_loop_support/src/capability_surface_filter.rs:174-195`). Nothing new.

**(c) Budget ‚Äî the one real gap, currently family-wide.** `SUBAGENT_ITERATION_LIMIT = 16` (`.../families/subagent.rs:9`) baked into `DefaultBudgetStrategy` (`:37-41`). **Ruling:** add `iteration_limit: u32` to `SubagentFlavor` (default 16), read at `material_for_run` ‚Äî family default stays fallback (**P3.2**).

**(d) Model override ‚Äî same pattern as (c).** `resolved_model_route` already overrides `DefaultModelStrategy` per run. `SubagentFlavor.model_override: Option<ModelRouteId>` feeds the same resolution (**P3.3**).

## 11. WebUI scoping & extensibility

**PR5 WebUI.** `GET /api/webchat/v2/threads/{thread_id}/children` (lineage read projection, no new store) + `ThreadTree` sidebar. **Authorization:** the endpoint resolves the parent thread through the caller's authenticated scope, using the same owner-bound `TurnScope` resolution every other webchat v2 thread-read endpoint uses; an unauthorized or unknown `thread_id` collapses to the same not-found response as those reads (no existence oracle), and the returned children are filtered to that same resolved scope. Display rule: opening a child directly shows the raw transcript (¬ß1); a "what did my subagent do" parent-agent-framed view shows only the framed + byte-capped edge content. **Required test (P5.2) ‚Äî integration-tier** (`tests/integration/`; the endpoint is a real HTTP seam over live lineage data): the children endpoint returns the correct lineage projection for a thread with settled/drained/parked descendants; a cross-user request for another caller's thread returns not-found, with no lineage leaked.

Extensibility (unchanged): fork-on-extend, structured child‚Üíparent output schema, cross-agent shared memory, per-token budgets, tool attenuation at depth, `ARCHIVED` state + GC, file-discovered custom flavors ‚Äî named deferrals, no code today.

## 12. What this design deliberately does NOT do

- No SQL tables, ledger, reconciler phases, dual-dialect DDL.
- **No semantic injection scanning of child output** ‚Äî the equivalent-control posture is untrusted framing (`wrap_untrusted_subagent_text`) + the byte cap (¬ß5.4) + gate-driven approval-bubbling to a human (¬ß9) as the catastrophe backstop, not a scan. The unscanned-ingress gap this implies is platform-wide and pre-existing ‚Äî Reborn's `submit_turn` ingress has no `SafetyLayer` wiring today for any caller, not something this design introduces for subagents specifically. When Reborn-wide `SafetyLayer` ingress wiring lands, the child‚Üíparent drain write adopts that same scan point like every other model-bound input ‚Äî no bespoke subagent scanner ‚Äî and that follow-up's scope must explicitly list the drain path as one of the ingress points it covers.
- No auto-cancel of orphans.
- No new crate; no feature flag.
- **No separate live-index/archive layer for closed edges (¬ß2).**
- **No gate expiry timers.**
- **No multi-node/HA correctness claims.**
- **No signature change to `RootFilesystem::delete`** ‚Äî the CAS primitive is additive (`delete_if_version`, ¬ß4.0), avoiding a ~20-call-site blast radius.
- **No content queue for human-priority activation (¬ß6a)** ‚Äî the next-slot reservation stores a marker, never a payload; no ordering, injection, or budget semantics to design for.
- **Tripwires (flip these rulings if hit):** (1) #5466 turns out structural to CAS-over-libsql, not fixable at the RootFilesystem layer; (2) a scope's roster directory becomes pathological to `list_dir` ‚Äî shard by tenant; (3) any scope's *open* edge count grows large enough that admission-time `list_dir` is measurably slow ‚Äî one CAS'd count file; (4) depth cap raised above 1 and ¬ß8/¬ß6 need more than the limits knob.
