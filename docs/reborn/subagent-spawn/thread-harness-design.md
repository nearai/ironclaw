## 1. Inherited core

- Child = normal Turn thread + lineage header (`parent_run_id`, `tree_root_run_id`, `depth`). Inspect/human-attach/re-activate are ordinary thread operations.
- One await-edge file per parent-awaits-child; CAS `open ŌåÆ settled ŌåÆ drained`, `open ŌåÆ abandoned`.
- Terminal is per-run; threads park, never die. `activate(thread, input, provenance)` is the single re-activation primitive; `ThreadBusy` if a run is live.
- Consent-to-wake: an agent wakes only its own direct live child; orphaned descendants are human-wake-only.
- Parent resume via `TurnStatus::BlockedDependentRun` + `gate:subagent-await-<child_run_id>` + `resume_turn` with `ResumeTurnPrecondition::BlockedDependentRunGate` (verified: `crates/ironclaw_turns/src/request.rs:33,48` maps this to `required_status() == Some(BlockedDependentRun)`).
- Child run record is the result source of truth (no `CapabilityResultStore`).
- Safety: childŌåÆparent framed-as-untrusted + byte-capped via `wrap_untrusted_subagent_text`; childŌåÆhuman raw; gates bubble up lineage (generalized in ┬¦9).
- **No flag.** `disabled_capability_ids` gates the capability.

## 2. Closed edges are deleted

CLAUDE.md: *"LLM data is never deleted... 'Cleanup' means evicting from in-memory caches, never deleting database rows."* Scoped to **model I/O** ŌĆö transcripts, run records, events. The await-edge is control-plane delivery bookkeeping, not a record of what any LLM said or did. The child's run record, its lifecycle events, and the framed content the parent consumed (written at drain) are the durable facts ŌĆö the edge is scaffolding for one-time delivery; once delivered it has no further job.

**Ruling: on CAS to `drained`, and on `abandoned` finalization, the edge file is deleted ŌĆö via the new CAS-guarded `delete_if_version` (┬¦4.0), never a blind delete.** `RootFilesystem::delete(path)` has **no CAS precondition** today (`crates/ironclaw_filesystem/src/root.rs:158` ŌĆö no `CasExpectation` param, unlike `put`); ┬¦4.0 adds the missing primitive, closing the silent blind-delete gap this design corrects.

**Crash-ordering and self-healing.** Edge close is durable across **three** steps, every time ŌĆö not a special recovery-only path: (1) CAS the edge's own state to its terminal value, (2) drive `reservation_release` (┬¦5.5) `Unclaimed ŌåÆ Claimed ŌåÆ Released` and prune the tree's `released_children` entry for this child (┬¦5.5, round-7 ŌĆö prune strictly before delete, never after), (3) `delete_if_version(edge_path, released_version)`. **Order is fixed ŌĆö state transition, then release-and-prune, then delete** (deleting before the state CAS would make a live-looking edge vanish with no record of why; deleting before the release reaches `Released` would strand `SpawnTreeReservation` capacity with no edge left on disk to recover it from). A crash leaves either a **terminal-but-unreleased edge** (crashed between (1) and (2)) or a **terminal-and-released-but-undeleted edge** (crashed between (2) and (3)) on disk ŌĆö **recovery (boot pass ┬¦4.3/┬¦5.3, and the lazy backstop) must enumerate `drained`/`abandoned` edges the same way it enumerates open ones**, complete whichever of steps (2)/(3) is outstanding, in order, and only then move on; `NotFound`/`VersionMismatch` on the retried delete is itself benign (someone else finished it, or it's mid-handling) and reported `already_closed`. **This makes the orphan window self-healing by construction, not merely invisible** ŌĆö recovery cost stays bounded, now also covering both post-terminal windows, not just true open/settled state.

**Round-4 fix ŌĆö the delete token is always the version from the *last* successful CAS write, never the state-transition CAS from step (1).** Every CAS on the edge file ŌĆö the terminal-state transition in step (1), and each of the `Unclaimed ŌåÆ Claimed` / `Claimed ŌåÆ Released` transitions in step (2) ŌĆö bumps the same file's `RecordVersion`, because ┬¦5.5's tri-state lives on that same edge file, not a separate one. A naive reading of step (3) as "delete using the version step (1)'s CAS returned" is therefore always wrong by the time step (3) runs: step (2)'s `Released` CAS has already superseded it, so `delete_if_version(edge_path, <step-1's version>)` would deterministically fail `VersionMismatch` on every close, piling up terminal-but-undeleted edges instead of ever deleting them. **The correct rule, stated once here and consistent everywhere this sequence is named (┬¦4.0, ┬¦5.5): `released_version` is the `RecordVersion` returned by the caller's own `Claimed ŌåÆ Released` CAS in step (2) ŌĆö the freshest write the caller just made ŌĆö never a version captured earlier in the sequence.** Recovery follows the identical rule when completing an outstanding step: whichever of (2)/(3) it drives to completion, it uses the `RecordVersion` its own most recent successful write returned as the token for the next write, re-reading current state rather than reusing a version captured before it started its recovery pass.

**Consequences:** the per-parent live-index approach from earlier drafts is **deleted outright** (deletion gets there for free). No closed-root/archive language anywhere. Settled-but-undrained edges are the pending-delivery queue and are never deleted ŌĆö only terminal transitions delete. `NotFound` on any edge read is benign ŌĆö treat it like a completed settle-then-drain.

**Abandon is mode-scoped, not a blanket "parent terminal ŌåÆ abandoned" rule.** ┬¦1's inherited openŌåÆabandoned transition applies only to:
- **(a) Blocking edges whose resume became impossible** ŌĆö the parent run went `Cancelled`/`Failed` while parked on the gate/dependent-run wait, and was never resumed. This is the failure case `abandoned` exists to capture.
- **(b) Explicit tree teardown** ŌĆö an operator/human tears the tree down (`subagent_cancel`, ┬¦7 PR6) while the edge is still open.

**For background edges, parent-run completion with the edge open is the *normal* delivery case, not an abandonment trigger** ŌĆö see ┬¦8. Threads park, never die (┬¦1); a live parent thread finishing its run is not "parent gone," it's exactly the state ┬¦8's `System`-provenance activate exists to wake. The edge stays `open` (then `settled`) and drains the ordinary way, whether the parent run that opened it is still live or has already gone terminal.

## 4.0 CAS-guarded delete primitive

PR1 merge blocker: **P1.0b**.

**Gap:** `RootFilesystem::delete` (`root.rs:158`) takes no `CasExpectation`, unlike `put`. Every backend's delete is unconditional: `in_memory.rs:142` (`state.entries.remove(path)`, no version compare), `libsql.rs:875` (`DELETE ... WHERE path = ?1 OR path LIKE ?2`, no version predicate), `postgres.rs:664`/`postgres_delete_with_client` (`postgres.rs:1584`, same shape). ┬¦2's edge-close and ┬¦4.5's roster-prune both need TOCTOU-safe delete; neither exists today.

**Ruling: add `delete_if_version(&self, path: &VirtualPath, expected_version: RecordVersion) -> Result<(), FilesystemError>` as a new, additive `RootFilesystem` method ŌĆö not a signature change to `delete`.** No `CasExpectation` parameter: `Any` would be behaviorally identical to the existing `delete(path)` (offering it would duplicate a sibling method), and `Absent` has no meaning for a delete (nothing to compare against) ŌĆö both are unrepresentable in the parameter type rather than runtime-rejected branches. Every call site passes a concrete `RecordVersion` it already holds ŌĆö for the edge, that is always the version returned by the caller's own last successful CAS write (the `Claimed ŌåÆ Released` reservation-release transition, ┬¦5.5/┬¦2's round-4 fix ŌĆö never an earlier transition's version, since every CAS on the same file supersedes the one before it); for the roster, the marker's just-read version. `RootFilesystem::delete(path)` has ~20+ production call sites across unrelated subsystems (verified: `ironclaw_secrets`, `ironclaw_skills`, `ironclaw_outbound/src/filesystem_store.rs`, `ironclaw_threads/src/filesystem_service.rs`, `ironclaw_product_workflow/src/filesystem_ledger.rs`, `ironclaw_reborn_composition/{slack_setup,extension_lifecycle,bundled_skills,llm_admin/llm_key_store,product_auth_durable/*}.rs`, `ironclaw_resources/src/cas_snapshot.rs`, `ironclaw_reborn/src/subagent/goal_store.rs`, `ironclaw_approvals/src/capability_permission.rs`) ŌĆö none need CAS, none should have their blind-delete semantics touched. Forcing a `CasExpectation` onto all of them is exactly the blast-radius churn this design avoids (`P1.0b` is scoped to await-edges/roster only).

**Error taxonomy is NOT inherited from `put` ŌĆö it's a new, small diagnosis branch.** `put`'s `Version` arm never emits `NotFound` for an absent row: `postgres.rs`'s `diagnose_put_failure` (`postgres.rs:1461-1477`) and `libsql.rs`'s `Version(expected)` arm (`libsql.rs:1280-1287` ŌĆö not the `Absent`-arm conflict branch at :1243-1250) both collapse "row absent" and "row present at wrong version" into the same `VersionMismatch{found: None}`; the in-memory `check_cas` helper (`in_memory.rs:514-534`) does the same ŌĆö its `(CasExpectation::Version(expected), found)` catch-all arm returns `VersionMismatch{expected: Some(expected), found}` regardless of whether `found` is `None` (absent) or `Some` (wrong version). That collapse is correct for `put` (an absent row on write just means "create it"), but wrong for `delete_if_version`, where the caller needs to tell "already gone" (`NotFound`, benign per ┬¦2) apart from "gone stale, don't touch it" (`VersionMismatch`). **`delete_if_version` therefore implements its own two-branch diagnosis on the 0-rows/absent case, per backend: read current state first (or follow up a failed conditional delete with a read) ŌĆö row absent ŌåÆ `NotFound`; row present at a different version ŌåÆ `VersionMismatch`.** This is new logic sized to the one new method, not a reuse of `put`'s diagnosis. **Narrowed parameter simplifies this further** ŌĆö since `delete_if_version` never takes an `Any`, its diagnosis is always exactly these two branches, no third "unconditional" case to special-case: row absent ŌåÆ `NotFound`; row present at a version other than the caller's `expected_version` ŌåÆ `VersionMismatch{expected: Some(expected_version), found: Some(found_version)}`.

**Shape, per backend:**
- Default trait impl: `Unsupported`, same pattern as `put`'s default.
- `in_memory.rs` (new fn beside :134, the `delete` impl): look up the current version before removing ŌĆö `None` ŌåÆ `NotFound`; `Some(v)` where `v != expected_version` ŌåÆ `VersionMismatch{expected: Some(expected_version), found: Some(v)}`; matching version ŌåÆ remove, no mutation on either error branch. Cannot reuse `check_cas` as-is (its catch-all arm doesn't distinguish absent from wrong-version, per above) ŌĆö the two-branch check is written directly in the new fn.
- `libsql.rs` (new fn beside :871, the `delete` impl): `DELETE ... WHERE path = ?1 AND version = ?2` bound to the caller's `expected_version`; 0 rows ŌåÆ a follow-up `SELECT version` distinguishes `NotFound` (no row) from `VersionMismatch` (row exists at a different version) ŌĆö a new two-step diagnosis for this method, not the reused `put` shape. No unconditional-delete arm ŌĆö that's `delete(path)`, a separate existing method.
- `postgres.rs` (new fn beside :664): same `AND version = $2` addition bound to `expected_version`, with its own follow-up `SELECT version` on 0-rows to distinguish `NotFound` from `VersionMismatch` ŌĆö `version` column already exists, read by `get`/`query` (`postgres.rs:442,482-489,1009-1030`), but the diagnosis branch itself is new.
- `ScopedFilesystem` (new wrapper beside `delete` at `scoped.rs:526`): pure passthrough to `self.root.delete_if_version(path, expected_version)`.
- `CompositeRootFilesystem` (new fn beside `delete` at `catalog.rs:360`): `self.matching_mount(path)?.backend.delete_if_version(path, expected_version)` ŌĆö pure dispatch.
- `HsmBackend` (new fn beside `delete` at `hsm.rs:111`): pure delegation to the inner backend (`hsm.rs:115-122`) ŌĆö **in scope, not a placeholder skip.** `HsmBackend::declared_capabilities()` already advertises `Delete` + CAS; falling through to the trait default `Unsupported` would pass mount-time capability validation but fail at call time, so it must delegate like every other unified-entry-plane method on this wrapper.
- `MountScopedRootFilesystem` (`crates/ironclaw_host_runtime/src/invocation_services.rs:435`, beside its `delete` override at :430) and `SkillManagementRootFilesystem` (`crates/ironclaw_skills/src/management.rs:148`, beside its `delete` override at :144) ŌĆö both are decorators that forward `capabilities()` to their inner backend, so a CAS-capable mount under either one needs the forward for `delete_if_version` too, or it silently falls through to `Unsupported` despite the backend underneath supporting it. Pure passthrough, same pattern as `ScopedFilesystem`/`CompositeRootFilesystem` above.
- **Out of scope:** `LocalFilesystem::delete` (`local.rs:354`) ŌĆö `LocalFilesystem::put` already rejects `CasExpectation::Version` as `Unsupported` (`local.rs:201-206`, no per-path versioning), so it was never viable for CAS'd edges. `StorageTxn::delete` (`backend.rs:44`) is the separate multi-key-transaction mechanism; edges are single-key CAS-only ("Stores must always work with CAS... as the floor," `root.rs:33`) and never open a `StorageTxn` ŌĆö unaffected.

**Consequences:** (a) **Edge close** = CAS state-to-terminal, then drive `reservation_release` to `Released` (┬¦5.5), then prune the tree's `released_children` entry for this child (┬¦5.5, round-7 ŌĆö strictly before the delete), then `delete_if_version(edge_path, released_version)` ŌĆö `released_version` is the version returned by that `Released` CAS itself, per ┬¦2's round-4 fix; ordering in ┬¦2/┬¦5.5. (b) **Roster prune** (┬¦4.5) = the pruner ŌĆö boot's recovery pass, **or the close path after deleting a scope's last edge (┬¦4.5's round-7 liveness fix ŌĆö boot is not the only caller)** ŌĆö reads the roster marker's current version at the instant it observes the scope's open-edge dir empty, then `delete_if_version(roster_path, that_version)`. A concurrent spawn writing a new edge for that scope bumps the marker's version via its own idempotent-but-version-bumping upsert, so the prune's CAS fails with `VersionMismatch` and aborts ŌĆö no TOCTOU loss of a roster entry newly needed. (The residual-race prose below narrates boot as the pruner ŌĆö the original and most contended caller ŌĆö but the CAS protections are caller-agnostic, per ┬¦4.5's round-7 ruling.)

**Residual race ŌĆö round-4's two-check compensation had a crash-dependent gap, closed by round-5's CAS-based fix.** ┬¦4.5's roster-before-edge order is never violated by this race ŌĆö the roster marker for a scope is always written before that scope's first edge. The residual race is purely about **boot's read-then-delete window landing inside the gap between a scope's roster upsert and its first edge write**, for a scope currently in the middle of its very first spawn: (1) spawn upserts the roster marker (version bump) ŌĆö roster-before-edge holds; (2) boot's recovery pass lists that scope's open-edge dir and finds it still empty (the edge hasn't been written yet); (3) boot reads the roster marker's *current*, already-bumped version at that same moment; (4) spawn now writes its first edge for the scope; (5) boot's `delete_if_version(roster_path, expected_version = the version read in step 3)` **succeeds** ŌĆö nothing bumped the version again between steps 3 and 5, so the version boot read is still current. The result: a real edge now exists for the scope, but its roster marker is gone.

**Round-4's fix relied on ordering between two checks that could itself race: if boot crashes after its own successful delete but before its own post-delete re-list (check (ii) below), and spawn's self-heal (check (i)) had already run-and-no-op'd** *before* **the delete (observed the marker present, correctly, at that moment ŌĆö not stale, just early) ŌĆö nothing left restores the marker, and the surviving edge is invisible to the next boot pass (lazy-only heal).** This is a real sequence, not hypothetical: spawn-self-check (no-op, marker present) ŌåÆ boot delete succeeds ŌåÆ boot crashes before its post-delete re-list.

**Round-5 fix: spawn's self-heal becomes an unconditional, version-bumping re-put ("touch") ŌĆö not a get-then-put-if-absent check.** After writing its first edge for a scope, spawn always re-puts the roster marker, whatever it currently reads as: present at version `V` ŌåÆ re-put bumps it to a new version (same payload, fresh `RecordVersion` ŌĆö a touch, per ┬¦4.0(b)'s "idempotent-but-version-bumping upsert" pattern, now applied here too, not just to the pre-edge upsert); absent (`NotFound`) ŌåÆ re-put creates it fresh (`CasExpectation::Absent`), same as before. **This closes the window by CAS, not by hoping the checks run in a lucky order:** if spawn's touch runs before boot's delete attempt, it bumps the version boot already read in step (3), so boot's `delete_if_version(roster_path, expected_version = <stale>)` fails `VersionMismatch` and aborts ŌĆö the roster survives, unconditionally, with no dependency on boot ever reaching its post-delete re-list. If boot's delete runs first (spawn's touch hasn't fired yet), the touch still runs unconditionally right after ŌĆö sees `NotFound`, re-creates the marker ŌĆö so check (ii) below is no longer load-bearing for *this* ordering either; it is kept as a second, independent compensating check, not the load-bearing one. **(ii) boot prune self-check (unchanged from round-4):** immediately after a successful `delete_if_version` of a roster marker, boot re-lists that scope's open-edge dir; if it is now non-empty, boot restores the marker via the same idempotent re-put.

**Residual double-fault window, accepted.** If spawn crashes *after* writing its first edge but *before* executing its own touch, concurrent with a prune that has already read a stale (pre-touch) version, the touch never runs to invalidate boot's CAS ŌĆö the marker can still be deleted and, if boot also never reaches check (ii) (e.g., it too crashes), stranded. This is a genuine double-fault (two independent crashes overlapping one narrow window), not closed by this fix. It is healed lazily: any later edge-write style event that touches this scope's roster marker (per ┬¦5.3's lazy backstop) re-runs the same unconditional touch and restores it. **That heal is conditional, not clock-guaranteed: it requires a *subsequent spawn for this same scope*.** A scope whose parked parent has no independent reason to reactivate and that never spawns again stays outside boot's roster-driven discovery indefinitely ŌĆö the operator's own `subagent edges` command (┬¦5.4) also enumerates off the roster, so it cannot surface the stranded scope by omission either; only a direct, roster-independent query against an already-suspected scope (`list_parents_with_unclosed_edges(scope)`, ┬¦4.3) would find it. Documented here explicitly as an accepted residual on double-crash-probability grounds, not as a guaranteed eventual heal.

**Generalizes the ABA caller invariant (`docs/reborn/contracts/filesystem.md` ┬¦14.1).** Version tokens are not generation-stable ŌĆö a path's version restarts at 1 on a fresh `put` after a prior delete, so a token captured before a delete+recreate cycle can match a different incarnation of the same path (ABA). **Edge files are structurally immune**: a `(parent_run_id, child_run_id)` pair never recurs, so an edge path is never recreated and no token can go stale-in-a-way-that-matches. **Roster markers are recreated** (the same scope's marker is deleted and later re-put across its lifetime) ŌĆö exactly the case that invariant warns about. But check (ii) above is not a race-window bound that more concurrent activity could defeat; it is a **postcondition recheck**: the re-`list_dir` fires after *any* successful `delete_if_version` on the roster marker, unconditionally, regardless of how stale or fresh the token that authorized the delete was. That is what makes it converge across arbitrarily many intervening delete-recreate cycles for the same scope ŌĆö the check never trusts the delete's success as final; it always re-observes real state immediately after, covering ABA without needing a generation-tracking token.

**Leaf-key note.** Edge and roster files are leaf keys with no children, so `delete`'s cascade semantics (the `... OR path LIKE ...` subtree sweep visible in `libsql.rs:875` and mirrored in the in-memory/postgres backends) are immaterial here ŌĆö `delete_if_version` is pinned to single-key semantics only; it never needs subtree-delete behavior.

**Required tests (P1.0b) ŌĆö crate-tier** (`RootFilesystem`/backend primitive, no production caller to drive through yet): dual-backend parity: (1) correct-version delete succeeds; (2) stale version ŌåÆ `VersionMismatch`, no mutation; (3) missing path ŌåÆ `NotFound`; (4) roster-prune-vs-concurrent-spawn race, split into (4a) and (4b) per the residual-race ruling above ŌĆö (4a) version-mismatch abort: a concurrent version bump lands before prune's delete call ŌåÆ prune's CAS fails, roster entry survives untouched; (4b) CAS-delete-succeeds-but-edge-lands-anyway: for a scope's very first spawn, the roster upsert lands first (roster-before-edge preserved), then boot's `list_dir` observes the still-empty open-edge dir and reads the already-bumped marker version, then the spawn writes its first edge ŌĆö prune's `delete_if_version(roster_path, expected_version = the version read)` succeeds because nothing bumped the version again in between, stranding a roster marker a now-real edge needs; the test asserts convergence to a restored roster entry via either compensating path (integration-tier, since (4b) exercises both spawn and boot in the same race window). This test **is** the design's ABA regression ŌĆö the roster marker is the one recreated path type in this design, and (4b)'s convergence assertion is exactly the postcondition-recheck property the ABA-invariant generalization above depends on. **P1.0b also inherits `docs/reborn/contracts/filesystem.md` ┬¦15's `delete_if_version` cases** (default-trait-impl backend ŌåÆ `Unsupported`; denied when the mount lacks `delete` permission) ŌĆö not restated here to avoid drift between the two docs, but part of the same task's acceptance. **Until P1.0b lands, nothing else in PR1 merges** (sibling gate alongside P1.0, ┬¦4.4).

Deleted (via `delete_if_version`, ┬¦4.0) on drain/abandon-final (┬¦2).

**4.3 Boot-recovery driver: roster- and store-driven.** Enumerate the scope roster (┬¦4.5), then per scope `list_parents_with_unclosed_edges(scope)` ŌĆö a plain bounded, **scope-isolated** `list_dir` under that scope's **full-axis prefix** (┬¦4.2's `agents/{some/<agent>|none}/projects/{some/<project>|none}` segments, not just the tenant/user-mounted `/turns/subagent-await-edges` root) (┬¦2, ┬¦4.5a ŌĆö this only ever lists that one scope's mounted-**and**-axis-qualified subtree), **plus** the terminal-but-undeleted sweep from ┬¦2's crash-ordering ruling. No `TurnStateStore` active-run query.

**Recovery concurrency is bounded, not one-scope-at-a-time nor unbounded fan-out ŌĆö and the bound is shared across both recovery paths (round-4 fix).** A single global limiter, `BOOT_RECOVERY_MAX_CONCURRENT_SCOPES = 4`, gates scope-recovery work regardless of which path enqueued it: the boot pass walking the roster, and the lazy per-scope backstop (┬¦5.3) starting a recovery task for an unbooted scope on first touch. At most 4 scope-resolves run concurrently across both origins combined ŌĆö boot does not get its own separate pool, and a burst of lazy first-touches does not get an unbounded one. This bounds backend/connection load during a cold-boot recovery sweep *and* during a lazy-recovery burst, regardless of how many scopes are pending from either source.

**Fairness ŌĆö bounded pending queue, per-tenant cap (round-5 fix).** The 4-wide limiter above bounds *execution*, not admission to the queue behind it ŌĆö an unbounded pending set would let one tenant's cold-boot burst (hundreds of scopes) starve every other tenant out of recovery indefinitely. Two additions: (a) a fixed-size global pending set (`BOOT_RECOVERY_PENDING_QUEUE_CAPACITY`, sized independently of the 4 execution slots) queues scopes waiting for a slot; (b) a per-tenant in-flight cap ŌĆö at most 2 of the 4 executing slots may belong to one tenant at a time ŌĆö so no single tenant can occupy the whole limiter even when its scopes are first in line. Queue admission is deduped per scope by the existing `in_progress` guard (┬¦5.3) ŌĆö a scope already queued or executing is never queued twice. **When the pending set is full, a new lazy first-touch is dropped, not queued** ŌĆö the scope simply stays unbooted; the caller's admission attempt already returns `ScopeRecoveryInProgress` (or the equivalent retryable rejection for a scope that never got a recovery task started) either way, so the caller's contract is unaffected ŌĆö it retries on next touch, which re-attempts queue admission from scratch. A **lazy** first-touch scope that is dropped this way is picked up on that scope's own next lazy first-touch, same as any other roster-miss.

**Boot-pass drops are a distinct, weaker case ŌĆö accepted residual, stated explicitly (round-6 fix).** The retry story above is solid for a *lazy* first-touch (some future caller will eventually touch that scope again, triggering another admission attempt). It does **not** automatically apply to a scope the **boot walker itself** tries to admit and finds the pending queue full: boot's own roster walk (┬¦4.5) runs once per process start, not on a timer, so a scope it drops here is not retried later *by boot* ŌĆö recovery for that scope depends entirely on some future caller independently touching it (spawn, activate, or an operator's `subagent edges` query against it directly). For an orphaned scope with no live parent and nothing left to independently trigger a touch ŌĆö the same class of scope ┬¦4.0's "Residual double-fault window" and ┬¦4.5a already document as an accepted, non-guaranteed heal ŌĆö a boot-pass drop under queue saturation is a **second, independent path to the same accepted residual**, not a new failure mode, but it is worth naming explicitly rather than leaving the reader to infer it from the lazy-touch framing above: under a large cold-boot burst that saturates `BOOT_RECOVERY_PENDING_QUEUE_CAPACITY`, some fraction of the boot-discovered scopes may go undiscovered for the rest of that boot pass, with recovery depending on the same lazy-touch/operator-query paths ┬¦4.0/┬¦4.5a already accept as non-guaranteed for the double-fault case.

**Required test (fold into P1.9's extension), integration-tier:** with the pending queue full and the per-tenant cap saturated (2 of 4 executing slots belonging to tenant A), a new lazy first-touch for another of tenant A's scopes is dropped (stays unbooted, admissible again on its own next touch) while a first-touch for tenant B's scope is admitted into a remaining slot.

**4.5 Scope roster + scope-key encoding.** Boot/lazy recovery needs which `(tenant, user, agent, project)` scopes have unclosed edges without a global walk (why a global walk isn't available at all: ┬¦4.5a). At first edge write for a scope, spawn idempotently writes a roster marker under `ResourceScope::system()` (`crates/ironclaw_host_api/src/resource.rs:28,112`; used this way by `ironclaw_conversations/src/filesystem_store.rs:203,237`, `ironclaw_reborn_composition/src/llm_admin/llm_key_store.rs:116`). **This pre-edge upsert is itself a version-bumping upsert (round-5 consistency fix)** ŌĆö same as ┬¦4.0(b)'s round-4 fix already assumed ("a concurrent spawn writing a new edge for that scope bumps the marker's version via its own idempotent-but-version-bumping upsert") and the same as ┬¦4.0's round-5 self-heal touch ŌĆö all three roster writes (pre-edge upsert, self-heal touch, boot's restore-on-non-empty) bump `RecordVersion` on every call, never leaving it unchanged on a no-op path, so every reader's captured version is provably pre- or post- any given write, never ambiguous.

**Encoding ŌĆö flattened, single path segment (round-3 fix).** Round-2's `{some/<v>|none}` nested-directory encoding ŌĆö reusing `crates/ironclaw_reborn/src/local_trigger_access/filesystem.rs:403-419`'s `(agent_id, project_id, user_id)` key shape, one directory level per axis ŌĆö turns out to be the wrong shape for the roster specifically: the roster is the one artifact in this design that boot must enumerate independent of any particular scope (┬¦4.3), and nested markers put different scopes' entries behind different intermediate directories that a `list_dir` call at any fixed depth cannot see past its immediate children. A `list_dir(/turns/subagent-await-scopes/)` under the nested form would only discover scopes whose markers happen to be direct children of the roster root ŌĆö every scope nested one or more levels deeper (i.e. almost every real scope) is silently missed by boot recovery. **(At round-3, this scope-independent enumeration was a single `list_dir` call; round-5, below, further shards that same scope-independent walk into 256 fixed hash-prefix directories for memory-boundedness ŌĆö the flattening ruling here is what makes per-scope enumeration possible without nested-directory branching at all, independent of how many calls the walk itself takes.)**

**Ruling: the roster flattens to one file per scope directly under `/turns/subagent-await-scopes/`, filename = the full encoded scope key in one path segment.** The optional-axis encoding is preserved conceptually (`{some/<v>|none}` per axis) but adapted from nested directories into `__`-delimited filename segments: `/turns/subagent-await-scopes/<tenant_id>__<user_id>__<agent_segment>__<project_segment>.json`, where `<agent_segment>` is `agent-<agent_id>` when the scope carries an agent or the literal `agent-none` otherwise, and `<project_segment>` is `project-<project_id>` when the scope carries a project or the literal `project-none` otherwise. `tenant_id` stays in-path for the same reason as before ŌĆö the roster lives under the system sentinel (constant `__system__`/`__system__` target, verified `ironclaw_reborn_composition/src/lib.rs:718-760`), so one predictable global tree needs tenant encoded in the key itself. **The project axis matches ┬¦4.2's edge path** ŌĆö both the roster key and the edge path partition on the same four axes (`tenant`, `user`, `agent`, `project`); a roster entry that only encoded `agent` while the edge path also varied by `project` would let two different-project scopes for the same agent collide onto one roster marker.

**`__` is a delimiter, not a safe one, without escaping (round-4 fix).** Round-3's claim that "ids cannot themselves contain `__` by construction" is false: `validate_scope_id` (`crates/ironclaw_host_api/src/ids.rs:18-51`, the sole validator behind `TenantId`/`UserId`/`AgentId`/`ProjectId`) only rejects empty values, length > 256 bytes, the exact segments `.`/`..`, any `/` or `\`, and control characters ŌĆö it has no underscore restriction at all, so `_` and `__` are legal anywhere in an id (confirmed by the regression test at `crates/ironclaw_host_api/tests/host_api_contract.rs:200-223`, which never tests `__`). Naive `__`-joining is therefore genuinely ambiguous: `(tenant_id="a__b", user_id="c")` and `(tenant_id="a", user_id="b__c")` are two distinct scope tuples whose `tenant_id`/`user_id` pair both concatenate to the identical `a__b__c` prefix before the agent/project segments are even appended ŌĆö a real filename collision, not a hypothetical one.

**Ruling: each of the four filename components is percent-encoded independently before the `__` join.** Encoding: for each component (`tenant_id`, `user_id`, the agent segment, the project segment), every byte outside `[A-Za-z0-9.-]` ŌĆö which includes the underscore itself ŌĆö is replaced by `%` followed by its two-digit uppercase hex value (so a literal `_` in any id becomes `%5F`); the four escaped components are then joined with the literal, unescaped `__` exactly as before, so the on-disk shape (`<tenant_id>__<user_id>__<agent_segment>__<project_segment>.json`) is unchanged, only the byte content of each segment. Decoding reverses this: split the filename on `__` into exactly four parts ŌĆö safe now because `__` can never occur *inside* an escaped component, only *between* them ŌĆö then percent-decode each part back to its original id. This is the simplest reversible scheme available (no length-prefix framing, no new dependency) and it generalizes past `__` collisions to any non-path-safe byte `validate_scope_id` still permits (spaces, non-ASCII, `%` itself, etc.).

**Required test (fold into P1.6a/P1.6b), unit-tier**, `roster_key_encoding_disambiguates_naive_join_collision`: encode the two colliding tuples above ŌĆö `(tenant_id="a__b", user_id="c", agent=None, project=None)` and `(tenant_id="a", user_id="b__c", agent=None, project=None)` ŌĆö and assert the two produced filenames are distinct (`a%5F%5Fb__c__agent-none__project-none.json` vs `a__b%5F%5Fc__agent-none__project-none.json`), and that decoding each filename recovers its own original tuple exactly, not the other's.

**Sharded by a hash-prefix directory, one level, 256-way (round-5 fix).** The flat single-`list_dir` layout above still requires boot to enumerate the **entire** roster in one `list_dir` call before processing any of it: neither production backend's `list_dir` takes a row cap (`postgres.rs`'s `list_dir` issues no `LIMIT`; `libsql.rs`/`in_memory.rs` are the same shape), and `RootFilesystem::list_dir_bounded`'s default impl (`root.rs:94-102`) only truncates the result *after* the same unbounded call already materialized it ŌĆö none of the three production backends override it to cap server-side. Left as a flat directory, roster growth is unbounded boot-time memory growth with no cap and no natural back-pressure.

**Ruling: partition the roster into 256 fixed shard directories, keyed by the first byte of a stable hash of the already-encoded filename.** Path becomes `/turns/subagent-await-scopes/<xx>/<tenant_id>__<user_id>__<agent_segment>__<project_segment>.json`, where `<xx>` is the two-lowercase-hex-digit first byte of `blake3::hash(scope_key_filename.as_bytes()).to_hex()` and `scope_key_filename` is the round-3/round-4 percent-encoded, `__`-joined string above unchanged ŌĆö sharding is a directory layer on top of that encoding, not a replacement for it, so the collision-disambiguation test above is unaffected (two names that were already distinct strings before hashing cannot collide after it; two names that hash to the same 2-hex prefix simply share a shard directory, which is the intended fan-out, not a collision). `blake3` is already a workspace dependency used for exactly this stable-hex-from-key shape (`crates/ironclaw_filesystem/src/db.rs:196-198`, index-name collision suffixing) ŌĆö no new dependency. Boot's roster enumeration becomes **256 sequential `list_dir` calls**, one per shard (`00` through `ff`), each shard's entries processed and dropped before the next is listed ŌĆö peak memory is bounded by the **largest single shard's** scope count, not the deployment's total, and the bound holds regardless of how large the roster grows (fan-out is automatic, not a re-tuned constant). The lazy per-scope path (first-edge upsert, self-heal touch, admission's roster lookup) computes its shard directly from the scope key it already holds ŌĆö an O(1) hash, never a listing ŌĆö so sharding adds zero cost to any hot path, only restructuring the already-background, already-bounded-concurrency (┬¦4.3) boot sweep.

**Considered and rejected: `query()`+`Page` pagination instead of sharding.** `RootFilesystem::query(path, filter, page)` (`root.rs:107-114`) already exposes a `Page { offset, limit }` cursor (`index.rs:319-343`) with real `LIMIT`/`OFFSET` SQL on both DB backends (`postgres.rs:414-440`, translating a `Filter` over a `descendant_path_range`) ŌĆö a ready-made paginated alternative on its face. Rejected because `query` sits one layer above `list_dir`: it returns materialized `VersionedEntry` bodies (heavier than `list_dir`'s bare path+type `DirEntry`), and using it efficiently over the roster's flat directory means either declaring a new index via `ensure_index` for a struct that today needs none, or paginating a `Filter` that still full-scans the same underlying rows per page. **A sharper reason than weight: `Page{offset, limit}` is `OFFSET`-based (`postgres.rs:451`, `libsql.rs:508`, plain `ORDER BY path LIMIT ŌĆ” OFFSET ŌĆ”`), not cursor-based ŌĆö a boot sweep paging through a live, concurrently-mutating roster (spawns writing new markers mid-walk) can skip or double-visit entries as inserts shift later pages' offsets underneath it, the classic OFFSET-pagination hazard. Fixed 256-way sharding sidesteps this outright: each shard is addressed by its hash prefix alone, unaffected by concurrent inserts anywhere else in the roster.** Sharding needs no new backend capability and no index declaration ŌĆö it keeps the roster on the same `list_dir`/`put`/`delete_if_version` primitives every other piece of this design already uses. Rejected for that reason, not because paginated listing doesn't exist in the contract.

**Not chosen: materialize-with-tripwire.** Accepting the single-`list_dir` materialization now, with an operational tripwire to shard later "if it gets pathological" (the prior ┬¦12 tripwire 2), was rejected on review as deferring a known, structural unbounded-memory shape to an incident rather than closing it in the design. Hash-prefix sharding also **strictly dominates** that tripwire's own fallback ("shard by tenant"): hash-based fan-out is uniform regardless of tenant-count skew ŌĆö a single outsized tenant does not get its own oversized shard the way tenant-keyed sharding would ŌĆö so promoting sharding into the base design retires that tripwire outright (┬¦12) rather than leaving it as a future escalation.

**Required test (fold into P1.6a/P1.6b), unit-tier**, `roster_shard_prefix_is_deterministic_and_well_distributed`: the shard-prefix function is a pure function of the encoded filename (same input ŌåÆ same shard, always); assert it over a representative sample of encoded keys that the 256 buckets are not pathologically skewed (a smoke bound, not a statistical proof ŌĆö e.g. no single bucket owns an unreasonable share of a few thousand sample keys).

**Payload vs. filename.** The marker's **payload** still carries the full structured `(tenant_id, user_id, agent_id, project_id)` scope ŌĆö that payload, not the filename or shard, is the source of truth boot uses to construct the `ResourceScope`/`TurnScope` it mounts and walks. The filename (and its derived shard) exists purely as the enumeration key so the shard walk can discover every marker without parsing structure out of the name; parsing the filename back into components is a convenience for tooling/debugging only, never load-bearing for correctness.

**Asymmetry with edge paths, and why.** Await-edge paths (┬¦4.2) stay **nested** under their per-scope, per-axis directories ŌĆö unchanged, and correct ŌĆö because edges are always listed **per-scope** (`list_parents_with_unclosed_edges(scope)`, ┬¦4.3), never independent of a scope; only the roster is walked independent of any particular scope (now shard-by-shard, not in one call), which is exactly why it ŌĆö and only it ŌĆö needs this flattened-then-sharded filename scheme.

**Required test (fold into P1.6a/P1.6b), integration-tier:** write roster markers for at least three scopes differing across every axis (different tenant; different user, same tenant; same user, different agent; same agent, different project) ŌĆö assert the 256-shard sequential walk (`list_dir` per shard `00..=ff`, not a single global `list_dir`) enumerates all of them regardless of which shard each lands in (regression test for the round-2 nested-marker miss, extended through the round-5 shard indirection).

**Write ordering:** the roster marker is written **before** the first edge file for that scope, never after. A crash between the two leaves a roster **superset** ŌĆö harmless, since boot finds an empty open-edge dir and the CAS'd prune (┬¦4.0(b)) removes the stale entry next pass. The unsafe ordering (edge first, roster after) would let a crash hide a real open edge from boot recovery entirely ŌĆö never chosen. **Required test (fold into P1.6a), integration-tier** (drives boot's real roster walk + prune): the non-racing superset base case ŌĆö seed a roster marker whose scope has an empty open-edge dir (crash between upsert and first edge write), run the boot pass, assert the stale marker is pruned via `delete_if_version` and the post-delete re-list (┬¦4.0 check (ii)) restores nothing ŌĆö the "harmless superset" claim, pinned rather than assumed.

Boot enumerates the roster by walking its 256 shard directories sequentially (`list_dir` on `/turns/subagent-await-scopes/<xx>/` for each `xx` in `00..=ff`), not one `list_dir` on the roster root; lazy per-scope backstop (┬¦5.3) stays as roster-miss safety net. **Dual-backend:** `list_dir` verified on all three backends ŌĆö `postgres.rs:639`, `libsql.rs:830`, `in_memory.rs:197`.

**Prune is not boot-only (round-7 fix ŌĆö roster liveness).** As previously written, the only prune path was boot's recovery pass ŌĆö on a long-lived process the roster would only ever shrink at the next restart, silently contradicting the cardinality invariant below and eroding the point of sharding (bounding *live* per-shard scope counts). **Ruling: the close path runs the same CAS'd prune opportunistically.** After deleting an edge (┬¦2 step (3)), the closer re-lists the scope's open-edge dir; if empty, it runs exactly ┬¦4.0(b)'s sequence ŌĆö read the roster marker's current version, `delete_if_version(roster_path, that_version)`, then the post-delete re-list of check (ii) ŌĆö same code path, same protections (a concurrent spawn's version-bumping upsert/touch fails the CAS; check (ii) restores on late edges). No new race analysis is owed: ┬¦4.0's roster-prune race machinery is version-CAS-based, not boot-specific, so every interleaving argued there covers this caller identically. **Required test (fold into P1.6a), integration-tier:** closing a scope's last edge prunes its roster marker mid-uptime (no restart), while closing one of several leaves the marker in place; a spawn racing the close-path prune survives via the same ┬¦4.0 (4a)/(4b) convergence.

**Roster cardinality.** One marker per `(tenant, user, agent, project)` scope with at least one unclosed edge ŌĆö kept true *live*, not just across restarts, by the close-path prune above ŌĆö cardinality tracks **active-subagent scopes**, not edge count (a scope with 16 open edges is still one roster entry). Per-shard `list_dir` enumeration may additionally paginate per the backend's ordinary `list_dir` semantics within a shard ŌĆö sharding and per-call pagination are independent, stacking bounds, not alternatives. ┬¦12 tripwire 2 (previously "shard by tenant") is **retired by this fix** ŌĆö sharding is now baked into the base design, not a future escalation; its replacement tripwire (┬¦12) is widening the shard-prefix width if 256-way ever proves insufficient. The ┬¦4.3 concurrency bound is unaffected by sharding ŌĆö it bounds concurrent *scope recovery*, independent of how the roster listing that discovers those scopes is itself paginated.

**4.5a Scope-aware construction ŌĆö no fixed-view.** The await-edge store and the scope roster are both built on the **same single shared `ScopedFilesystem` handle**, constructed *once* via `crate::wrap_scoped(root)` ŌåÆ `ScopedFilesystem::new(root, invocation_mount_view)` (`crates/ironclaw_reborn_composition/src/lib.rs:848-852`; resolver `invocation_mount_view` at `:753-763`, alias table `PER_USER_ALIASES` ŌĆö including `/turns` ŌĆö at `:718-733`) ŌĆö **never** via `ScopedFilesystem::with_fixed_view`. Every op (`put`/`get`/`list_dir`/`delete_if_version`) takes the caller's live `ResourceScope` as an explicit argument (`scoped.rs:216-233,487,526`); the resolver recomputes the `MountView` for *that* scope on *that* call. This is the same pattern already load-bearing in production for `ironclaw_conversations::filesystem_store` (`.get(&scope, &path)` / `.put`, re-evaluating a fresh scope per call at `filesystem_store.rs:205,238`) and `ironclaw_reborn_composition::llm_admin::llm_key_store` (`store.put(scope(), ...)`, `scope()` re-evaluated per call at `llm_admin/llm_key_store.rs:42,115`) ŌĆö neither constructs a filesystem baked to one scope.

**Anti-pattern, named explicitly:** `owner_turn_state_filesystem` (`crates/ironclaw_reborn_composition/src/factory.rs:1927-1938`, used in production at `:2079,:4314`) builds a **`ScopedFilesystem::with_fixed_view`** baked to a single `owner_scope` at composition time ŌĆö a single-boot-owner view. That is the exact bug class behind #5720/#5721 (per-user `/turns` state collapses onto one owner on multi-user boxes). **New await-edge/roster code must not copy this constructor** ŌĆö it must be built the `wrap_scoped`/`invocation_mount_view` way: one process-wide handle, scope threaded per call, never per-scope-baked at construction. (The only other `with_fixed_view` hit under `subagent/` ŌĆö `factory.rs:5011`'s `subagent_goal_filesystem` ŌĆö is test-only fixture code inside `mod tests` opening at line 4647; production's goal store at `factory.rs:2232` is correctly built via `wrap_scoped`.)

**Consequence ŌĆö ┬¦4.2's canonical path is scope-relative, not scope-agnostic ŌĆö and now axis-relative too.** `ResourceScope` (`crates/ironclaw_host_api/src/resource.rs:31`) carries only `tenant_id`/`user_id` ŌĆö it has no `agent_id`/`project_id` field at all ŌĆö and `invocation_mount_view` (`ironclaw_reborn_composition/src/lib.rs:718-763`) rewrites `PER_USER_ALIASES` (including `/turns`) using only those two fields. **The mount alone cannot isolate two agents (or two projects) of the same user** ŌĆö that isolation has to come from the path itself, which is exactly what ┬¦4.2's `agents/{some/<agent>|none}/projects/{some/<project>|none}` in-path segments provide. `/turns/subagent-await-edges/agents/{some/<agent>|none}/projects/{some/<project>|none}/<parent_run_id>/<child_run_id>.json` is a `ScopedPath` alias; the physical `VirtualPath` it resolves to is `/tenants/<tenant>/users/<user>/turns/subagent-await-edges/agents/{some/<agent>|none}/projects/{some/<project>|none}/<parent_run_id>/<child_run_id>.json` (`/turns` is tenant/user-rewritten ŌĆö it is one of the `PER_USER_ALIASES`; the agent/project segments are ordinary path segments beneath the alias, untouched by that rewrite). Two different `TurnScope`s' edges for the *same* `parent_run_id`/`child_run_id` pair therefore land in physically distinct trees on the backend whenever **any** of tenant, user, agent, or project differs ŌĆö not just tenant/user. That is what makes `list_parents_with_unclosed_edges(scope)` (┬¦4.3) a plain, bounded, **scope-isolated** `list_dir` under `/turns/subagent-await-edges/agents/{...}/projects/{...}` for that scope's own agent/project axis values: it only ever lists that one scope's mounted-**and**-axis-qualified subtree, never a global one and never a sibling agent's or project's edges under the same user. The scope roster (┬¦4.5) exists precisely because that isolation means boot cannot discover *which* scopes have edges without a scope-agnostic index ŌĆö the roster marker (itself written under the system sentinel scope, so its own listing is scope-independent ŌĆö not scope-isolated like the edge tree, though walked via the round-5 256-shard scheme rather than one call ŌĆö and now carrying the same agent/project axes per ┬¦4.5) is what tells boot which real `(tenant, user, agent, project)` scopes to mount-and-walk one at a time.

## 5. Core delivery mechanics

The following are carried forward unchanged from the prior durability-design lineage (restated here so this doc is self-contained; ┬¦5.5 and ┬¦5.6 are new).

**5.1 Capacity: reservation is the cap, listing advisory.** Unchanged. `SpawnTreeReservation` (depth Ōēż 1, Ōēż 4 spawns/turn, Ōēż 16 descendants/tree) is the sole admission mechanism.

**Made explicit: the 16-descendant cap counts non-terminal children ŌĆö deliberately, including gate-parked ones.** A child sitting on `BlockedApproval`/`BlockedAuth` still holds its reservation slot. The cap bounds **outstanding side-effect obligations** ŌĆö 16 approval-parked children is 16 pending sensitive actions paging one human ŌĆö not compute. Compute concurrency is excluded on a separate, already-existing axis: the scheduler deschedules a blocked run (no worker slot held), so a tree full of gate-parked children costs zero live workers, only zero remaining reservation headroom. These are deliberately different budgets; this design does not collapse them.

**Never release-on-park; release only at terminal (┬¦5.5); never re-claim on ordinary resume.** A child resuming past its own gate (approval granted, auth completed) is not a re-admission event ŌĆö it already holds its slot from spawn, uninterrupted. **Ruling: resumption must never require re-admission.** An approved child that can't resume because the tree filled up while it waited would be a deadlock-by-design (the human who just unblocked it discovers the unblock didn't work) ŌĆö never chosen.

**Relief valves, when a tree does fill:** `subagent_inspect` (┬¦7 PR3) shows which children are blocked and why; `subagent_cancel` (┬¦7 PR6) reclaims slots by driving a child to terminal early (terminal ŌåÆ release, ┬¦5.5); a human resolving the gate queue directly (┬¦9.2) drains blocked children the ordinary way, which also releases their slots. `ironclaw-reborn subagent edges` (┬¦5.4) reports per-scope open-reservation counts alongside edge counts, so an operator sees a tree approaching the cap before it deadlocks.

**`subagent_extend` is the one re-claim path ŌĆö by design, not by exception.** Unlike ordinary resume, `subagent_extend` (┬¦6, ┬¦7 PR4) re-activates an *already-terminal* child (┬¦0 requirement 3, "no cold start"). A terminal child already released its slot (┬¦5.5: `Claimed ŌåÆ Released` on terminal). Re-activating it is therefore a **fresh outstanding obligation**, not a continuation ŌĆö `subagent_extend`'s `activate()` call re-claims a descendant-reservation slot as a normal admission check, and fails with the existing capacity error (the same one spawn uses) if the tree is already full. This is the only place in the design where activating a child re-runs admission.

Required tests (**P4.4**): extend on a full tree ŌåÆ capacity rejection (integration-tier ŌĆö drives extend through the real `activate()`/reservation admission path); extend releases its slot at the new run's terminal (crate-tier is sufficient for the reservation-accounting half).

**5.2 Two-layer exactly-once, `InvalidTransition` discriminator.** Layer 1: CAS single-winner per transition. Layer 2: `resume_turn`'s durable idempotency replay (`crates/ironclaw_turns/src/memory/mod.rs:187,1031-1046,2248-2269`, CAS'd in the same snapshot as every run-state transition). No `resume_dispatched_at` field needed.

**Boot/drain contract, matched on `from` ŌĆö no wildcard.** `resume_turn_once` (`memory/mod.rs:2616-2640`) sets `from: record.status.get()` when `BlockedDependentRunGate`'s required status (`BlockedDependentRun`, `request.rs:48`) mismatches, raising `InvalidTransition{from, to: Queued}`.

- **Benign `already_closed`:** `from Ōłł {Queued, Running, Completed}`.
- **`ResolveReport.failed`, surfaced:** everything else. No wildcard match.

Required tests (**P1.3**), crate-tier (`InvalidTransition` discriminator + CAS layer are pure state-machine logic, no external seam to drive through): (a) double-settle race ŌåÆ one resume; (b) crash-after-settle-before-resume ŌåÆ one resume via durable idempotency key; (c) crash-after-resume-before-drain ŌåÆ `InvalidTransition{from: Queued|Running}`-as-`already_closed` lets drain complete without a second resume.

**5.3 Boot enumeration: roster-driven + lazy backstop.** Eager: background boot task walks roster (┬¦4.5), resolves unclosed edges per scope, never blocks foreground. Lazy: a spawn/activate against an unbooted scope does **not** run scoped recovery inline on the caller's path ŌĆö it starts-or-joins the single in-flight scoped-recovery task for that scope (`in_progress` guard dedupes concurrent triggers onto one task) and immediately returns a retryable rejection, `ScopeRecoveryInProgress { retry_after_hint }` ŌĆö named and treated by callers exactly like `ThreadBusy` (retry/backoff, no special-casing at call sites). This preserves the foreground latency guarantee: no caller ever blocks on a scoped recovery walk. Once that scope's recovery task completes, the next spawn/activate against it is admitted normally.

**Fairness (round-5 fix, ┬¦4.3):** the shared limiter's pending queue is bounded and per-tenant-capped, not just execution-bounded ŌĆö a lazy first-touch that finds the pending queue full is dropped (scope stays unbooted, retried on next touch) rather than queued unboundedly. See ┬¦4.3 for the queue-capacity and per-tenant-cap rulings; this section's admission contract (retryable `ScopeRecoveryInProgress`, no caller-side special-casing) is unchanged by that addition.

**Distinct from `ThreadBusy`/`ThreadReserved` (┬¦6a) in granularity, not just name.** `ScopeRecoveryInProgress` is a **scope-wide** rejection ŌĆö it can affect every thread/caller under that scope at once, tied to the background recovery task's lifetime, not to any single thread's run state ŌĆö whereas `ThreadBusy` and `ThreadReserved` are **per-thread**. Callers still retry/back off identically for all three (no special-casing at call sites), so this doesn't change handling; it only means a human reading logs should not conflate "this one thread is busy" with "this whole scope is still recovering." **Lazy per-scope recovery tasks triggered by first-touch admission share the same bounded limiter as the boot pass (round-4 fix)** ŌĆö `BOOT_RECOVERY_MAX_CONCURRENT_SCOPES = 4` (┬¦4.3) governs the background scope-recovery work regardless of which path enqueued it, one global limiter for both. A burst of first-touch calls across many distinct unbooted scopes still dedupes to one recovery task per scope via the `in_progress` guard, and each such task competes for the same 4-wide bound boot uses ŌĆö it never gets a separate or unbounded pool. **The bound applies only to the background recovery work, never to the foreground admission call**: `activate()`/spawn admission against a scope with an in-flight or now-queued recovery task still returns `ScopeRecoveryInProgress` immediately (the opening ruling above) ŌĆö the caller is never made to wait on the limiter itself; only the recovery task it triggered queues behind whatever else is already occupying the 4 slots.

**Required test (P1.9 extension), integration-tier** (drives spawn/activate admission through the real lazy-backstop path against an unbooted scope): admission attempted against a scope with an in-flight scoped recovery ŌåÆ `ScopeRecoveryInProgress` retryable rejection; retried after the recovery task completes ŌåÆ admitted normally. **Extend the same test (round-4 fix) for the shared limiter:** with `BOOT_RECOVERY_MAX_CONCURRENT_SCOPES` slots already saturated by a mix of boot-triggered and lazy-triggered recovery tasks, a new lazy first-touch against a distinct unbooted scope still returns `ScopeRecoveryInProgress` immediately (never blocks on the limiter) while its recovery task queues behind the others; once a slot frees, the queued task runs and that scope becomes admissible in turn.

**5.4 Observability.** `ResolveReport { resumed, drained, abandoned, already_closed, failed }` per scope, `debug!` + counters. Under ┬¦2, `drained`/`abandoned` now mean "resumed/finalized **and the edge file deleted**." `failed > 0` is actionable: inspect the named edges via `subagent edges --scope`, then the gating child's run record ŌĆö entries here are resume/drain errors that recovery retries next pass, so a value that persists across passes (not a one-shot blip) is the escalation signal. `ironclaw-reborn subagent edges [--scope ŌĆ”]` lists unclosed edges off the roster (a host-level trusted operator command, deliberately cross-tenant like the rest of the `ironclaw-reborn` CLI ŌĆö not a tenant-scoped surface). **`VersionMismatch` disposition, unified:** occasional `VersionMismatch` on close/prune/touch paths is expected CAS contention ŌĆö benign, self-healing, no action (┬¦2/┬¦4.0 treat it as such case-by-case). A *sustained per-close* `VersionMismatch` rate ŌĆö every close failing ŌĆö is the round-4 stale-token bug class reintroduced, and is alert-worthy, not benign. **Terminal byte-length accounting:** today's `AwaitedChildState.terminal_byte_len` (`gate_resolution.rs:27`) is set by `record_terminal_byte_len` (`:186-198`) as a best-effort **second** write *after* `mark_terminal_result_written` in `write_terminal_result` (`completion_observer.rs:497-518`) ŌĆö **moves onto the edge as an additive `terminal_byte_len: Option<u64>` field, set in the same CAS write that transitions the edge to terminal.** Strictly better than today (folds two mutations that could drift into one atomic write). Still feeds `ByteCapStrategy` observability only, never durability ŌĆö a missing value is logged at `debug!` and never blocks delivery.

## 5.5 Descendant-reservation release tri-state on the edge

**Gap:** the single-winner release guard for `SpawnTreeReservation` capacity lives entirely on the dying store ŌĆö `descendant_reservation_released` / `descendant_reservation_release_claimed` fields plus `claim_descendant_reservation_release`/`mark_descendant_reservation_released`/`release_descendant_reservation_claim` (`gate_resolution.rs:334-379`, exact range). ┬¦3's replacement inventory lists this as "surviving" logic without saying where its state lives once the store dies ŌĆö this section closes that gap.

**Ruling: the tri-state moves onto the `AwaitEdge`'s own state payload, CAS'd like every other transition ŌĆö no separate store, no new file.** Add `reservation_release: ReservationReleaseState` (`Unclaimed | Claimed | Released`):
- `Unclaimed ŌåÆ Claimed`: CAS'd (`CasExpectation::Version(edge_version)`) exactly like today's in-memory check-then-set; only the CAS winner releases `SpawnTreeReservation` capacity.
- `Claimed ŌåÆ Released`: CAS'd after the capacity-release call succeeds, mirroring `mark_descendant_reservation_released`.
- A failed release attempt CAS's back `Claimed ŌåÆ Unclaimed`, mirroring `release_descendant_reservation_claim`'s retry-unlock, so a transient failure doesn't permanently strand the reservation.

Lives on the **same edge file** as settle/drain state ŌĆö one more field, one more CAS'd path, not a second file ŌĆö so it's deleted with the edge on drain/abandon (┬¦2).

**Boot recovery also owns the claim-but-not-yet-released window ŌĆö and that window has two sub-phases, not one (round-5 fix).** `Claimed` is entered by winning the `Unclaimed ŌåÆ Claimed` CAS, but reaching `Released` additionally requires (a) the capacity-release call (`release_tree_descendants`, `completion_observer.rs:436-444`) to succeed and (b) the `Claimed ŌåÆ Released` CAS to commit ŌĆö two separate steps, only the second of which is CAS'd. A crash **before** (a) leaves an edge in `Claimed` with the release call never made ŌĆö safe to retry, as originally stated. A crash **between** (a) and (b) ŌĆö the release call already succeeded, only the recording CAS didn't commit ŌĆö leaves the *same* `Claimed` state on disk, indistinguishable from the first case. Recovery cannot tell these apart from `reservation_release` alone, and `release_tree_descendants` (`crates/ironclaw_turns/src/memory/mod.rs:1692-1719` and its `filesystem_store.rs`/`lifecycle.rs` siblings) is a **plain shared-counter decrement** keyed on `(scope, root_run_id)`, not on the individual child ŌĆö it has no per-child idempotency of its own, so a naive "recovery just retries the call whenever it finds `Claimed`" (as this section previously said) double-decrements the tree's shared descendant counter for that child. Because the counter only hard-errors on *under*flow ("Reject over-release loudly," `memory/mod.rs:1716-1720`), a double-release below that floor doesn't error ŌĆö it silently frees one extra slot, eroding ┬¦5.1's 16-descendant cap upward by one per crash landing in this narrow window, with nothing to ever re-derive true capacity from ground truth.

**Ruling: `release_tree_descendants` gains an additive `idempotency_key: TurnRunId` parameter (the child's own run id) ŌĆö the store checks-and-records the key atomically before decrementing, so a retried call for the same child is a no-op, not a second decrement.** Child run ids are ABA-immune for exactly the reason round-4 already established for edge paths (┬¦4.0's "Generalizes the ABA caller invariant" ŌĆö a `child_run_id` never recurs), so a dedup key on it can never collide across two different children or be revalidated against a stale prior incarnation. This is an additive trait-signature change on `TurnStateStore::release_tree_descendants` (`crates/ironclaw_turns/src/store.rs:116`), with a small, named blast radius: two production call sites (`completion_observer.rs:438`, this design's own terminal-release path, keyed on the child's `TurnRunId` it already has in hand; `ironclaw_loop_support/src/subagent_spawn_port.rs:458`, a synchronous spawn-rollback unwind, not retried by recovery and thus not at risk of *this* double-release window, but still needs a key for signature compatibility ŌĆö **its own `compensation.submitted_child_run`'s `run_id` serves that role (round-6 correction: this field is set in the same statement pair as `submitted_child_tree`, `subagent_spawn_port.rs:999-1000`, so a real child `TurnRunId` is already in scope at the rollback call site ŌĆö no new token type needs inventing, and an earlier draft of this ruling wrongly claimed one did**) and the store's few backend impls (`filesystem_store.rs:824,1204`, `lifecycle.rs:592`, `memory/mod.rs:1692`, `filesystem_store/row_store/traits.rs:284`). With this fix, recovery finding an edge in `Claimed` simply retries the (now-idempotent) release call unconditionally and then CASes to `Released` ŌĆö the original "just another CAS attempt, no special-cased recovery path" framing is restored, now actually true because the underlying call, not just the edge's own state, is safe to repeat.

**Where the per-child dedup record lives, and how it stays bounded (round-6 fix).** The obvious-looking shortcut ŌĆö reuse the existing generic idempotency-record cache (`crates/ironclaw_turns/src/memory/mod.rs`'s `idempotency_records`, LRU-capped at `max_idempotency_records * 4`, the same mechanism ┬¦5.2 already cites for `resume_turn`'s durable idempotency replay) ŌĆö is **rejected here**: that cache is bounded by evicting the *oldest* record under unrelated load, and an edge that sits unrecovered for a long time under heavy churn from *other* trees could have its dedup key evicted before recovery ever retries it, silently reopening this exact double-release race. **Ruling: the dedup record lives on the `SpawnTreeReservation`'s own durable record, not a separate cache ŌĆö a small `released_children: BTreeSet<TurnRunId>` field alongside its `(scope, root_run_id) ŌåÆ descendant_count` map**, checked-and-inserted in the same store call that performs the decrement (present ŌåÆ no-op, absent ŌåÆ decrement and insert). **That check-insert-decrement is one atomic mutation of the single `SpawnTreeReservation` record** ŌĆö a read-modify-CAS-write on one durable record, the same single-record discipline every other transition in this design uses; no backend needs anything beyond the CAS floor it already provides, and no cross-record atomicity is assumed anywhere in this mechanism.

**Round-7 fix ŌĆö the entry is pruned *before* the edge is deleted, as its own ordered step, not "in the same step."** An earlier draft said the entry is "removed in the same step that deletes the edge" ŌĆö but the entry lives on the reservation record and the edge is a separate file, and this design provides no transaction spanning two records (┬¦4.0's out-of-scope ruling: edges are single-key CAS-only and never open a `StorageTxn`). Read as delete-then-prune, a crash between the two would orphan the entry permanently: with the edge gone, no recovery sweep ever revisits that `child_run_id`, and each such crash would leak one entry ŌĆö falsifying the boundedness claim below. **Ruling: the close sequence prunes first ŌĆö `Claimed ŌåÆ Released` CAS, then prune the tree's `released_children` entry for this child, then `delete_if_version(edge_path, released_version)`.** Crash between prune and delete: the edge survives on disk at `Released`, so recovery's terminal-but-undeleted sweep (┬¦2) finds it, re-runs the prune (a no-op on an absent entry), and completes the delete ŌĆö and it never re-invokes the release call itself, because recovery only retries the release on `Claimed`, never on `Released`, so the early prune cannot re-open the double-decrement. Crash before the prune: entry present, edge still on disk at `Released` ŌĆö same sweep, same completion. Either way the entry is always removed while its edge still exists to drive recovery back to it, so no entry can outlive its edge. With that ordering fixed, the set is genuinely bounded: an entry exists only between its child's release and that same child's edge close ŌĆö at most one entry per currently-open-or-in-flight edge, bounded by ┬¦5.1's 16-descendant reservation cap ŌĆö never a function of the tree's cumulative lifetime child count.

**Accepted residual, named (round-7): the stalled in-flight releaser.** An actor that observed the edge at `Claimed`, then stalled *before issuing its release store call* for the entire duration of a competing actor's release ŌåÆ `Released` CAS ŌåÆ prune sequence, issues its call against an already-pruned entry and decrements a second time. This is not closed by the ordering above ŌĆö no ordering of prune vs. delete closes it, because the vulnerable call was in flight before either. It is accepted on narrowness grounds, same disclosure style as ┬¦4.0's double-fault window: this design makes no multi-node claims (┬¦12), and in one process a claimant that dies with the process never issues its call after restart ŌĆö the only reachable overlap is a live in-process claimant racing the same process's boot/lazy recovery sweep inside the startup window, and the `in_progress` guard (┬¦5.3) already caps recovery to one task per scope, so the racing pair is at most {live claimant, one recovery task}. Worst case is one silently-freed slot per occurrence while the tree is non-empty; the store's existing over-release floor ("Reject over-release loudly," `memory/mod.rs:1716-1720`) still hard-stops erosion at zero. **Invariant that keeps the residual two-party, stated not assumed:** resuming a `Claimed` edge that you did not personally CAS from `Unclaimed` is exclusively the recovery sweep's (boot/lazy, ┬¦4.3/┬¦5.3) prerogative ŌĆö the live path only ever calls release after winning its own `Unclaimed ŌåÆ Claimed` CAS, and no foreground path may "helpfully" resume a `Claimed` edge it didn't claim. Any future path that adopts recovery's permissive behavior widens this residual from two-party to N-party and voids the narrowness argument above ŌĆö P1.9's race test (a) pins the two-party shape. **Detection story (it is not invisible):** the drift is a countable invariant violation ŌĆö a tree's `SpawnTreeReservation` count lower than its live non-terminal-children count (`children_of`, ┬¦1's lineage queries). the `subagent edges` command (┬¦5.1's relief-valve note; the command's home is ┬¦5.4) reports per-scope open-reservation counts alongside edge counts precisely so an operator can compare the two; a persistent negative gap for a tree is this residual's signature, distinguishable from transient in-flight settles by re-querying after the scope quiesces.

**Required test (fold into P1.9's extension), crate-tier:** seed an edge at `Claimed` and call `release_tree_descendants` twice with the same `idempotency_key` (simulating recovery retrying a call that already succeeded once) ŌĆö assert the tree's descendant counter is decremented exactly once, not twice, and the second call returns success (a no-op) rather than an over-release error or a silent second decrement. **Required test (fold into P1.2/P1.9), crate-tier ŌĆö round-7, crash-injected, not happy-path** (crash-injection between two named store writes; no external seam can stage this window, same justification as P1.3): (a) crash injected between the `Released` CAS and the prune ŌåÆ recovery re-runs prune + delete, the entry does not outlive the edge, and the counter shows exactly one decrement across the whole history; (b) crash injected between the prune and `delete_if_version` ŌåÆ recovery completes the delete **without** re-invoking the release call (it observes `Released`, not `Claimed`), counter still decremented exactly once.

**Ordering with edge deletion.** An edge may be deleted (┬¦2, ┬¦4.0) **only after** its `reservation_release` field has reached `Released` ŌĆö never while `Unclaimed` or `Claimed`. The full terminal sequence is: settle ŌåÆ resume/drain ŌåÆ drive `reservation_release` to `Released` ŌåÆ prune the `released_children` entry (round-7 fix above) ŌåÆ `delete_if_version`. Recovery's terminal-but-undeleted sweep (┬¦2's crash-ordering ruling) must honor this same sequencing: on finding a terminal-but-undeleted edge, it first completes the release (retry `Claimed ŌåÆ Released`, or the failure-path retry-unlock to `Unclaimed` and re-attempt), re-runs the prune (no-op if already pruned), and only *then* calls `delete_if_version` ŌĆö a terminal edge whose `reservation_release` isn't yet `Released` is never deleted, so capacity accounting is never lost with no edge left to recover it from.

**The version token for that `delete_if_version` call is the `Released` CAS's own return value (round-4 fix).** Because `reservation_release` lives on the same edge file as every other field, the `Claimed ŌåÆ Released` CAS in this sequence bumps the file's `RecordVersion` past whatever version the earlier terminal-state CAS (┬¦2 step (1)) returned ŌĆö that earlier version is stale by the time deletion runs and would fail `VersionMismatch` every time if used. The caller must carry forward the version *this* CAS returns, not any version read or returned earlier in the sequence; recovery, which may itself be the one performing the `Claimed ŌåÆ Released` transition, uses the version its own write just returned for the same reason ŌĆö it never reuses a version it captured before starting its recovery pass.

**Required test (P1.9 extension), crate-tier:** (a) double-release race, **staged at the dangerous point (round-7 reframe): both racers already past `Unclaimed ŌåÆ Claimed`** ŌĆö two concurrent resolver instances (crash-retry racing a fresh recovery pass) both observe the edge at `Claimed` and race the release store call itself; assert capacity is released **exactly once** ŌĆö the second call finds the `released_children` entry present and no-ops. (The `Unclaimed ŌåÆ Claimed` CAS having a single winner is a distinct, already-safe property ŌĆö assert it as a sub-case, but it is not the race the idempotency key exists for); (b) boot-recovery-retries-`Claimed` ŌĆö seed an edge already at `Claimed` (simulating the crash window above), run a recovery pass, and assert **both**: it drives the edge to `Released` (or retry-unlocks to `Unclaimed` on a simulated release failure) rather than leaving it stuck, **and** ŌĆö extending the same seeded case rather than a separate test ŌĆö that the edge is deleted only *after* reaching `Released`, never before (the release-before-delete ordering); **plus a third seed (round-7): a terminal edge still at `Unclaimed`** (crash before any claim was attempted) ŌĆö recovery walks it `Unclaimed ŌåÆ Claimed ŌåÆ Released ŌåÆ prune ŌåÆ delete` from scratch, covering ┬¦2's first crash window explicitly rather than by assumption.

## 6. Run-budget floor (derived, not stored)

Cap: **8 consecutive `ParentAgent`-provenance activations per thread.** `Human` resets to 0, never capped. 9th consecutive `ParentAgent` ŌåÆ `subagent_extend_budget_exhausted`, sanitized, no internal identifiers cross the model-visible boundary.

**Counter ŌĆö derived at `activate()`, not stored.** `SessionThreadRecord.metadata_json` (`crates/ironclaw_threads/src/contract.rs:184`) has **no mutation path**. Instead: `TurnRunRecord` gets one additive field, `subagent_activation_provenance: Option<ActivationProvenance>` (`ActivationProvenance { Human, ParentAgent, System }` ŌĆö `System` is ┬¦8's background-wake provenance) set once at run creation, immutable thereafter. **Staging note (round-7): the field itself lands with PR2 (P2.4), not PR4** ŌĆö ┬¦8's `System`-provenance tagging and ┬¦8.3's streak cap both read it before this section's extend budget ever ships; P4.2 adds only the ┬¦6 windowing logic on top of the already-present field. Without this pull-forward, PR2 would depend on a schema field two PRs away ŌĆö a staging inversion, named and closed here.

`activate()` derives the cap via a **bounded** query, not an open-ended walk: fetch at most the newest **8** non-`System`-provenance run records for the thread (`K = cap = 8`, one indexed/ordered query, `LIMIT K`) ŌĆö enough to decide the cap outright for the **pending** activation being admitted, without ever materializing a 9th `ParentAgent` record: if a `Human` run appears anywhere in that 8-record window, or history has fewer than 8 non-`System` records before its start, the pending activation is admitted (the streak resets or hasn't reached the cap); if all `K` fetched records are `ParentAgent` with no `Human` among them, the pending activation would be the 9th consecutive `ParentAgent` and is refused. `System` runs are excluded from the fetch itself (not filtered post-fetch), so they never count against the `K`-record budget. The constant is tied to the cap by construction (`K = cap`, not `cap + 1` ŌĆö the cap is enforced at admission time on the *next* activation, so history itself never holds more than `cap` consecutive `ParentAgent` records to begin with) ŌĆö raising the 8-activation cap means raising `K` in lockstep, never decoupling them (distinct from ┬¦12 tripwire 4, which is about the separate depth cap).

**Required tests (P4.2), crate-tier** (bounded `LIMIT`-8 query is pure state-machine logic over run records, no external seam to drive through yet): one test row, four assertions ŌĆö (a) under-8 consecutive `ParentAgent` activations are admitted; (b) exactly 8 consecutive `ParentAgent` activations admitted, the 9th refused with `subagent_extend_budget_exhausted`; (c) a `Human` activation anywhere in the window resets the streak to 0; (d) `System` runs are excluded from the fetch and never count toward, nor are ever blocked by, the cap.

Unchanged: per-child SUBAGENT budget (`iteration_limit` 16, made per-flavor-overridable in ┬¦10); `SpawnTreeReservation` cap (16), depth cap (1, depth-agnostic per ┬¦9/┬¦10).

## 6a. Human input priority ŌĆö no content queue

Two mechanisms give a human priority over `ParentAgent`/`System` activation traffic on a thread. **The no-queue non-goal stands**: neither mechanism stores any payload, so none of a content queue's ordering, injection, or budget questions apply ŌĆö a reservation is a marker, not a mailbox.

**(i) Next-slot reservation.** A human whose `activate()` hits `ThreadBusy` may set a `human_waiting` reservation marker on the thread ŌĆö **owner-gated: only a human authenticated as the thread's owner (`owner_user_id`, uniform across the tree per README ┬¦8 item 2, "Approval ownership") may set, renew, or abandon it.** A non-owner cannot reserve ŌĆö and thereby block ŌĆö another user's thread; the marker would otherwise be a 15-minute interference vector against the owner's own `ParentAgent`/`System` deliveries. Operator intervention, when needed, is the console cancel path ((ii) below), not this marker. While set, `activate()` admission refuses `ParentAgent` and `System` provenance with a new `ThreadReserved` outcome ŌĆö callers treat it exactly like `ThreadBusy` (retry/backoff, no special-casing needed at call sites). The marker clears when the human activates (their `activate()` succeeds) or when they explicitly abandon the reservation.

**The reservation is a lease, not an indefinite hold.** A disconnected client that never activates and never abandons would otherwise block `ParentAgent` **and** `System` (┬¦8's settle-wake delivery) forever ŌĆö a starvation bug, not a feature. **Ruling:** the marker carries `owner` + `expires_at` (default **15 minutes**, `HUMAN_RESERVATION_LEASE_TTL`); admission treats an expired lease as absent via a lazy, read-time check (`now > expires_at` at the point `activate()` reads the marker) ŌĆö no timer task, no dedicated background reaper. No sweep over this per-thread marker is needed for correctness: a live `activate()` call re-checking an expired marker is sufficient on its own, so a stale marker left after a client disconnects is inert ŌĆö it simply never blocks anything again once its `expires_at` has passed, whether or not anything ever cleans up the file itself. Human re-activation or explicit abandon still clears the marker immediately, ahead of expiry; renewing the reservation is a re-`put` with a fresh `expires_at`.

**This does not contradict ┬¦12's "no gate expiry timers" non-goal.** That non-goal is about **security-decision gates** (approval/auth) ŌĆö they guard a sensitive action and must park forever until a human decides, never auto-resolve on a clock. The `human_waiting` reservation carries no security content at all ŌĆö it is a UX courtesy hold on scheduling priority, and letting it lapse only returns the thread to ordinary `ParentAgent`/`System` admission, exactly the state it would be in had the human never reserved it. Lapsing a courtesy hold is safe by construction; auto-resolving a gate would not be.

Deferred `System` settle-wakes are not lost while a reservation holds `ParentAgent`/`System` off the thread ŌĆö the existing run-start sweep and boot pass (┬¦8.2, triggers 2 and 3) already heal any settle-wake that loses to `ThreadBusy`/`ThreadReserved`; the settled edge just waits its turn. **Liveness framing, honest (round-7):** unlike a `ThreadBusy` deferral (bounded by the live run finishing on its own), a reservation-based deferral is bounded only by the 15-minute lease plus whenever the thread next actually runs ŌĆö and it composes with ┬¦8.3's streak cap without stranding anything (the cap gates only the reactive wake; trigger 3 drains at the resolver layer regardless, and trigger 2 is not `System`-provenance-gated), but neither mechanism *initiates* a retry when the lease lapses. Delivery latency here is human-paced by design, the same posture as ┬¦8.3's liveness bound.

**Marker storage:** a small marker file, scoped and CAS'd the same way as every other piece of state in this design (┬¦4.0/┬¦4.5's pattern) ŌĆö not a mutation onto `SessionThreadRecord.metadata_json`, which ┬¦6 already establishes has **no mutation path**. Reusing the filesystem-marker pattern this design already relies on for edges and the scope roster is simpler than opening a first mutation path onto `metadata_json` for a feature that doesn't need anything else `metadata_json` carries. Payload: `{ owner: UserId, expires_at: Timestamp }` ŌĆö `owner` identifies the reserving human for audit/UX, `expires_at` backs the lease check above.

Ships with the extend PR (**P4.3**) ŌĆö `activate()` admission is where `ThreadBusy` already lives, so `ThreadReserved` is a sibling admission outcome in the same place. **Required tests ŌĆö integration-tier**: (a) reservation set ŌåÆ parent `subagent_extend` refused with `ThreadReserved` ŌåÆ human `activate()` succeeds and clears the marker ŌåÆ a subsequent parent extend is admitted normally; (b) reservation set with an expired `expires_at` ŌåÆ a `ParentAgent` `activate()` is admitted (expired lease treated as absent), with no timer task or reaper involved; (c) a non-owner's attempt to set (or abandon) the reservation is rejected and leaves admission behavior unchanged.

**(ii) Interrupt & take over.** Console-only compose of two existing primitives ŌĆö the underlying run-cancel mechanism (the same one `subagent_cancel`, ┬¦7 PR6, later wraps as a model-facing tool) and `activate(Human)` (┬¦1) ŌĆö no new state, no new admission outcome, and no dependency on PR6's tool-level capability landing first. A human on a child (or parent) console cancels the live run and activates in its place. Ships with the console UI PR (**P5.4**).

**Ruling: `drain_settled` is NEWLY IMPLEMENTED by an owned PR2 task (P2.4) wiring it to `AwaitEdgeResolver` ŌĆö list settled edges for the live parent, drain each. This supersedes the #4474 stub contract; no `LoopBackgroundChildPort` is ever built.**
- **Live parent:** `drain_settled` drains each iteration (P2.4).
- **Parked/completed parent:** on child settle, resolver calls `activate(parent_thread, input, provenance=System)`. `ThreadBusy` is a benign no-op ŌĆö edge stays `settled`; ┬¦8.2's retry set picks it back up. `System` activations are exempt from ┬¦6's cap, bounded to **one** attempt per settled child ŌĆö the edge's `settled` state is the dedupe.

**Batched drain, not per-edge rescan.** When `drain_settled` (or a recovery/boot pass) finds **multiple** settled edges for the same parent, it must batch all their `result_ref`/`safe_summary` transcript updates into **one** thread-snapshot read + **one** CAS write pass ŌĆö never one rescan-and-CAS-retry cycle per edge. `update_parent_result_reference` ŌåÆ `update_tool_result_reference` (┬¦8.1) is the right primitive for a *single* settled edge's CAS-retry closure; the batch caller accumulates the full set of settled `(result_ref, safe_summary)` pairs for the parent first, then invokes the transcript-update helper **once** over that whole set ŌĆö one snapshot read, every matched message rewritten in place, one CAS write ŌĆö rather than calling it `E` times against a thread with `M` messages. This keeps a multi-edge drain O(E+M) instead of O(E├ŚM); the O(E├ŚM) shape is exactly what a naive per-edge loop over `update_parent_result_reference` would produce on a run-start sweep or boot pass that finds several children settled at once.

**8.1 Drain-append idempotency.** The framed-content write into the parent ŌĆö `update_parent_result_reference` ŌåÆ `update_tool_result_reference` (`crates/ironclaw_threads/src/filesystem_service.rs:1965-2013`) ŌĆö is a **CAS-guarded in-place field update on an already-existing message**, not an append: it rescans for the tool-result-reference message (`matches_tool_result_reference`) then rewrites its `content` via `apply_message_update`'s CAS-retry closure. Calling it twice with the same `result_ref`/`safe_summary` reproduces the same `content` ŌĆö **already idempotent by construction**, verified not asserted. The payload write it follows (`update_capability_result`, from `write_terminal_result`, `completion_observer.rs:494-518`) is guarded today by the in-memory `terminal_result_written` flag; under the edge design that guard **becomes the edge's own CAS state** (write payload once per `settled`ŌåÆ`drained` transition ŌĆö a crash before that CAS just retries the whole flow, safe precisely because the transcript write is an overwrite, not an append). **Required test ŌĆö integration-tier** (`tests/integration/`; drives the drain path through the real `AwaitEdgeResolver` + thread service, asserting at the transcript-message seam per `.claude/rules/testing.md`, part of P2.4's acceptance): crash after the transcript field-write but before the edge's CAS to `drained` ŌåÆ recovery replays the equivalent write ŌåÆ assert exactly one tool-result-reference message for that `result_ref`, content unchanged by the replay.

**Required test (P2.4 extension) ŌĆö integration-tier:** drain of N (Ōēź2) settled edges for the same parent in one pass performs exactly one thread-snapshot read and one CAS write ŌĆö not N of each ŌĆö asserted via a write-count seam (e.g. a counting wrapper around the thread service's snapshot-read/CAS-write calls, or an equivalent call-count assertion).

**8.2 System-activate retry set.** A settled edge for a parked/live parent is drained by exactly three independent triggers:
1. **Settle-time** `activate(..., System)` attempt (above) ŌĆö may lose to `ThreadBusy` mid-run.
2. **Every run-start sweep** ŌĆö `PostCapabilityStage::process` runs on every `Continue`, including a fresh run's first iteration, so the *next* time this thread runs for **any** reason, `drain_settled` (P2.4) picks up every still-settled edge as a side effect of that run happening at all. **Scope note (round-7):** `process` fires on `TurnCompletedStep::Continue` iterations only ŌĆö a degenerate run that exits on its very first step without a single `Continue` never fires this trigger for that activation; such a run leans on triggers 1 and 3 (and on the threadŌĆÖs next non-degenerate run), which is why the invariant below names the trigger set, not ŌĆ£any run whatsoever.ŌĆØ
3. **Boot pass** (┬¦4.3/┬¦5.3) ŌĆö roster-driven, independent of any thread activating: the pass drains settled edges **directly at the resolver layer** (the transcript write + edge close of ┬¦8's batched-drain rule ŌĆö `update_parent_result_reference` is a thread-service call, not a run), so it needs no `activate()` and is never gated by ┬¦8.3's streak cap; only the *reactive* `System` wake is.

**Invariant:** a `ThreadBusy` at settle-time (trigger 1) is always healed by trigger 2 or 3 ŌĆö a settled edge can never go permanently undrained while its parent thread ever runs again or a boot pass ever occurs. **Required test ŌĆö integration-tier** (`tests/integration/`, same P2.4 acceptance as ┬¦8.1, driven through the live parent thread + boot-pass harness, not the resolver in isolation), named `settled_edge_threadbusy_is_healed_by_run_start_and_boot_pass`: **both** scenarios below are asserted in the same test, not either/or ŌĆö (a) settle a child mid-parent-run (forcing `ThreadBusy` on trigger 1), assert no further `System` attempt (dedup by `settled` state), then let the live run's next iteration drain it (trigger 2) and assert the edge closes; and, as a separate scenario, (b) settle a child mid-parent-run the same way, then instead run a boot pass and assert the roster sweep drains it (trigger 3). **Make the parent-completed precondition explicit** (┬¦2's mode-scoping): the parent run reaching its own terminal state (`Completed`) while the background child is still running and its edge is `open` is not a special case of the above ŌĆö the test asserts the edge stays `open` (never `abandoned`, per ┬¦2) across the parent's own terminal transition, and that the child's later settle still wakes the parked parent thread via trigger 1 (or trigger 3, if the thread never runs again first) and drains normally.

**8.3 Autonomous-wake containment ŌĆö a second derived streak cap (round-7).** ┬¦8's `System` wake is deduped per settled edge and ┬¦5.1 bounds *outstanding* children ŌĆö but nothing above bounds the *cumulative* spawn ŌåÆ settle ŌåÆ wake ŌåÆ spawn cycle: a parent that spawns one fresh child on every `System` wake sustains an indefinite autonomous loop under every existing cap, with no human in it. The prior draft bounded this with a dedicated `AutonomousContinuationBudget` component (README ┬¦7.4/┬¦8) ŌĆö never built; only its production-readiness field exists, retired in ┬¦3. **Ruling: same mechanism as ┬¦6, second axis ŌĆö a derived consecutive-`System`-activation cap, no stored counter, no new component.** `activate()` admission fetches the newest `K = SYSTEM_WAKE_STREAK_CAP = 16` run records of `Human`/`System` provenance for the thread (`ParentAgent` excluded from the fetch, mirroring how ┬¦6's fetch excludes `System`): if all `K` are `System` with no `Human` among them, the pending `System` activation is refused; if a `Human` appears anywhere in that window, **or history holds fewer than `K` `Human`/`System` records before its start (the young-thread case, same admit rule as ┬¦6's under-`K` branch)**, the wake is admitted. `Human` resets the streak. A refusal loses nothing **durable**: the settled edge stays `settled`, drained by trigger 2 on the thread's next admitted run or by trigger 3's boot/recovery pass ŌĆö which drains directly at the resolver layer without activating the parent (┬¦8.2 trigger 3), so the cap never gates delivery itself, only the reactive wake. **The honest liveness bound:** for a human-abandoned thread with a saturated streak on a long-lived process, no wall-clock promptness is guaranteed ŌĆö the parent does not *react* to those results until the next `Human` activation or the next process restart's boot pass. That is the cap working as intended (autonomous work parked pending human attention), not data loss. The two caps compose: ┬¦6's fetch is blind to `System` and this cap's fetch is blind to `ParentAgent`, so any human-free activation sequence ŌĆö however interleaved ŌĆö admits at most 8 `ParentAgent` + 16 `System` activations before the thread parks until a human returns. The old sketch's per-time-window rate quota is deliberately dropped: every `System` wake requires a real child run's full lifecycle to produce a settled edge, so with a hard count cap a time window adds tuning surface without adding containment. (The value 16 here coincides with ┬¦5.1's descendant cap and ┬¦10c's iteration limit ŌĆö three unrelated budgets that must remain independently named constants in code, never merged by a refactor.) **Named trade-off, mirroring ┬¦5.1's posture: the streak budget is deliberately thread-wide, not per-child.** A noisy subtree that burns 16 back-to-back wakes exhausts the parent's autonomous attention for its quieter siblings too ŌĆö their settled edges still drain losslessly via triggers 2/3, and a thread consuming its autonomous budget that fast is precisely the thread that should wait for its human. Per-child-lineage streaks were considered and rejected: they reopen unbounded total wakes through child fan-out (N children ├Ś per-child budgets), defeating the cap's purpose. Observability: a streak-cap refusal increments a counter and logs at `debug!` (┬¦5.4's discipline) ŌĆö the successor to the sketch's `AutonomousContinuationStopped` event. **Required test (fold into P2.4's acceptance), crate-tier** (pure run-record windowing, same seam as ┬¦6's P4.2): one row, four assertions ŌĆö (a) under-16 consecutive `System` wakes admitted; (b) the 17th consecutive `System` wake refused, the edge stays `settled`, drained by the next admitted run; (c) an interleaved `ParentAgent` activation neither resets nor counts toward the streak (excluded from the fetch); (d) a `Human` activation resets it.

## 9. Gate propagation ŌĆö approval and auth, always to the tree root

Any gate from any descendant, any depth, bubbles to the **tree root's** originating human surface. Covers **auth/credential gates** (`BlockedAuth`) too, not just approvals. **Surfacing is always the tree root's originating surface** ŌĆö one deterministic paging point, unchanged. **Resolution is not the same thing as surfacing**: it is accepted from any surface authenticated as the tree owner, including a child thread's console a human has open directly ŌĆö ┬¦9.2 lifts the old blanket restriction for owner-authenticated humans specifically; it still holds, without exception, for any LLM in the chain.

**Both kinds, one shape ŌĆö no new gate representation.** `TurnBlockedGateMetadata` (`crates/ironclaw_turns/src/events.rs:59-66`) already carries `gate_kind: TurnBlockedGateKind` (`Approval`, `Auth`, `Resource`, `AwaitDependentRun`, `ExternalTool`) plus `credential_requirements`. The walk is generic over `gate_kind`.

**9.1 Root-delivery-surface resolution, field-by-field.** The root's `source_binding_ref` is **not** unconditionally the delivery surface ŌĆö this breaks for triggered origins. Resolution chain, reusing existing production machinery instead of inventing a fallback: `ironclaw_outbound`'s `resolve_run_notification_context` (`crates/ironclaw_outbound/src/resolution_engine.rs:59-95`) already dispatches on `RunNotificationOrigin` (`.../delivery_resolution.rs:137-151`):
- **`LiveSourceRoute { source_route }`** ŌĆö **dispatches on `gate_kind`, matching its sibling origin arms (round-3 ruling).** It was, until this ruling, the only arm of `resolve_run_notification_context` that ignored `gate_kind` entirely: `Triggered` and `TriggeredFromSourceRoute` already route `ApprovalPrompt`/`AuthPrompt` through `load_preference_target` (owner-scoped, with the existing approval/authŌåÆfinal_reply fallback) rather than the raw binding ref. **`LiveSourceRoute` now does the same:** `ApprovalPrompt`/`AuthPrompt` resolve via `load_preference_target(scope, actor, kind)`; `FinalReply`/`ProgressUpdate`/`DeliveryStatus` keep `source_route.reply_target_binding_ref` verbatim, unchanged (`:73-75`) ŌĆö still covering **interactive chat and live Slack** for those kinds, where the root's `TurnRunRecord.reply_target_binding_ref` (`ironclaw_turns/src/store.rs:160,171`) genuinely identifies a real conversation. **Rationale:** authority-bearing prompts (an approve button, an auth URL) raised anywhere in a tree rooted in a shared live conversation must not be paged into that shared surface by default ŌĆö the same posture PR #4953 enforces at send time for OAuth URLs, applied here at the resolution layer, where candidate selection belongs. **This narrows paging only, not resolvability**: ┬¦9.2 already accepts resolution from any owner-authenticated surface, so a human watching the shared channel or a child console can still resolve the gate ŌĆö they are just not paged there by default. Where the owner's preference happens to resolve to the same private conversation as the source route, delivery is unchanged.
- **`Triggered { trigger }`** ŌåÆ `resolve_triggered_target(scope, actor, kind)` (`:76,119-137`), which for `ApprovalPrompt` calls `load_preference_target(scope, actor, PreferenceTargetKind::ApprovalPrompt)` ŌĆö resolved from `TurnScope.explicit_owner_user_id()` (the trigger creator) via the owner's outbound preference, **bypassing `reply_target_binding_ref` entirely.** Every trigger fire's binding is synthesized against the canonical `TRIGGER_TRUSTED_ADAPTER_KIND = "trigger"` adapter (`crates/ironclaw_triggers/src/trusted_submit.rs:3,33-41`; predicate `is_trusted_trigger_adapter_kind`, "the trigger-owned authority," lines 7-13) ŌĆö never a live channel ŌĆö so a triggered root's binding refs are unconditionally non-actionable, not "sometimes." (Note: `SourceBindingRef::new("trusted-trigger-outcome-source")` at `ironclaw_conversations/src/inbound.rs:1053` is a **unit-test fixture** inside `#[cfg(test)] mod tests` opening at line 529, not a production value ŌĆö the real non-actionable value is the `"trigger"` adapter binding above.)
- **`TriggeredFromSourceRoute { trigger, source_route }`** ŌåÆ hybrid path for a triggered fire with a genuine source route too (`:78-86`).
- **`SystemEvent { reason }`** ŌåÆ `NoDelivery` (`:87-89`).

**Origin classification** reuses the existing predicate `is_automation_trigger_thread` (`crates/ironclaw_product_workflow/src/reborn_services.rs:4727-4737`, parses `SessionThreadRecord.metadata_json`) ŌĆö no new classifier.

**Ruling:** the walk's root-delivery step builds a `RunNotificationContext`/`CommunicationDeliveryIntent::RunNotification` for the tree root (root's `TurnScope` + classification above) and delegates to this **existing** engine ŌĆö it does not read `source_binding_ref` directly, and invents no second fallback.

**Named integration gap (not hand-waved):** today's triggered-run delivery poller (`SlackFinalReplyDeliveryServices`/`deliver_triggered_run`, `crates/ironclaw_reborn_composition/src/slack_delivery.rs:2033`, feature `slack-v2-host-beta`) watches the **root run's own** status for `BlockedApproval`/`BlockedAuth` (`notification_for_actionable_state`, `:395-460`). Under this design a gate parks the **descendant**, not the root ŌĆö the root sits at `BlockedDependentRun` (┬¦5.2), so this poller would miss the gate entirely. **P2.5/P2.6 (┬¦7) own extending it (or a successor) to accept the walk's resolved gating-run id/kind/gate_ref, not just observe the root run it already tracks** ŌĆö a real required wiring change, not a pre-existing behavior to lean on unmodified.

**Auth identity ŌĆö no new resolver.** `RuntimeCredentialAuthRequirement.requester_extension: ExtensionId` (`crates/ironclaw_host_api/src/decision.rs:106-113`) is already typed on the gate metadata the walk forwards verbatim.

**Required tests (P2.5/P2.6) ŌĆö integration-tier** (`tests/integration/`; the walk, root-routing, and the Slack poller extension above all cross the descendantŌåÆrootŌåÆdelivery-surface seam, not a single-function unit): the depth-agnostic walk resolving a descendant gate to the tree root's delivery surface for each `RunNotificationOrigin` branch (┬¦9.1), and the extended triggered-run delivery poller (`deliver_triggered_run`, above) picking up the walk's resolved gating-run id/kind/gate_ref instead of only the root run's own status. **Add (┬¦9.2):** resolution submitted from a child-console surface resolves the gating run exactly once ŌĆö the second resolution attempt observes the already-resolved CAS state and no-ops; **and** a resolution attempt from an authenticated **non-owner** surface is rejected with no gate existence leaked (the same not-found collapse as ┬¦11's cross-user read). **Add (┬¦9.1, round-3 `LiveSourceRoute` kind-dispatch):** `LiveSourceRoute` + `ApprovalPrompt`/`AuthPrompt` for a tree rooted in a shared-channel conversation ŌåÆ resolves to the owner's preference target, not the shared channel.

**Mechanics ŌĆö depth-agnostic:**
1. The gating run parks. No ancestor run touched.
2. Resolver walks `parent_run_id` upward: skip terminal runs, stop at the run whose `parent_run_id` is `None` ŌĆö the root.
3. ┬¦9.1's resolution engine picks the delivery target from the root's scope + origin.
4. Resolution flows back down: the surface's decision resolves the **gating run itself** ŌĆö not the root run.
5. An `abandoned` (deleted, ┬¦2) edge along the chain doesn't break the walk ŌĆö lineage lives on the immutable run record.

**9.2 Resolution accepted from any owner-authenticated surface.** Surfacing and resolution are decoupled. Surfacing (┬¦9's opening ruling) stays exactly as designed ŌĆö the tree root's originating surface is the sole paging point, unconditionally. What changes: the gate **resolution** itself ŌĆö the actual approval/auth decision ŌĆö is accepted from *any* surface authenticated as the tree owner, not only the root's originating surface. This includes a child thread's console a human has directly opened (┬¦1, `activate(Human)`; PR5a/5b once shipped, but the underlying approval-interaction service has no console dependency ŌĆö it is exercisable as soon as any owner-authenticated surface can reach a `gate_ref`, ahead of full ThreadTree UI).

Why this is safe, not a hole in the never-auto-approve floor:
- **The approver must be a human, from any surface.** No LLM anywhere in the chain resolves a gate ŌĆö a subagent (at any depth) referencing its own or a sibling's pending gate still never resolves it. Only lifting this for owner-authenticated humans, never for automation, is what makes it safe.
- **The approval service was never root-scoped, only root-paged.** `crates/ironclaw_product_workflow/src/approval_interaction/service.rs` resolves against `gate_ref`/`run_id` with no thread affinity baked into the decision path ŌĆö root-surfacing was a paging convention layered on top, not a constraint the service itself enforced. **Which is exactly why the owner check lives at the surface layer, named here as a requirement, not assumed:** every surface that exposes or accepts a `gate_ref` (root chat, child console, ┬¦11's endpoints) must validate the authenticated caller against the tree's owner ŌĆö `caller.user_id == root.owner_user_id`, the same owner every descendant inherits verbatim (README ┬¦8 item 2's spawn-code enforcement makes owner uniform across the tree) ŌĆö before the gate is either shown or resolvable, the same owner-bound scope resolution ┬¦11 specifies for the children endpoint. The service's affinity-free resolution is safe *only* behind that surface-layer check; P2.5/P2.6 own wiring and testing it.
- **Two doors, one CAS'd lock.** Gate resolution is CAS-idempotent, single-winner, the same pattern as every other transition in this design (┬¦5.2, ┬¦5.5). Whichever surface's resolution attempt wins the CAS resolves the gate; the other observes the already-resolved state and no-ops. Two resolution paths onto one lock is exactly the shape CAS'd resolution already handles safely.

**Audit:** the resolution record captures *which* surface actually resolved the gate ŌĆö root or a specific non-root owner surface ŌĆö not just that it was resolved, so a human reviewing the tree afterward can see where the call was made.

## 10. Per-flavor configurability ŌĆö flavors are data, the loop is shared

## 11. WebUI scoping & extensibility

**PR5 WebUI.** `GET /api/webchat/v2/threads/{thread_id}/children` (lineage read projection, no new store) + `ThreadTree` sidebar. **Authorization:** the endpoint resolves the parent thread through the caller's authenticated scope, using the same owner-bound `TurnScope` resolution every other webchat v2 thread-read endpoint uses; an unauthorized or unknown `thread_id` collapses to the same not-found response as those reads (no existence oracle), and the returned children are filtered to that same resolved scope. Display rule: opening a child directly shows the raw transcript (┬¦1); a "what did my subagent do" parent-agent-framed view shows only the framed + byte-capped edge content. **Required test (P5.2) ŌĆö integration-tier** (`tests/integration/`; the endpoint is a real HTTP seam over live lineage data): the children endpoint returns the correct lineage projection for a thread with settled/drained/parked descendants; a cross-user request for another caller's thread returns not-found, with no lineage leaked.

Extensibility (unchanged): fork-on-extend, structured childŌåÆparent output schema, cross-agent shared memory, per-token budgets, tool attenuation at depth, `ARCHIVED` state + GC, file-discovered custom flavors ŌĆö named deferrals, no code today.

## 12. What this design deliberately does NOT do

- No SQL tables, ledger, reconciler phases, dual-dialect DDL.
- **No semantic injection scanning of child output during the harness/dark-launch phase** ŌĆö the equivalent-control posture is untrusted framing (`wrap_untrusted_subagent_text`) + the byte cap (┬¦5.4) + gate-driven approval-bubbling to a human (┬¦9) as the catastrophe backstop, not a scan. The unscanned-ingress gap this implies is platform-wide and pre-existing ŌĆö Reborn's `submit_turn` ingress has no `SafetyLayer` wiring today for any caller, not something this design introduces for subagents specifically. **This is a phase-scoped non-goal, not a standing one: ┬¦7's prod-enable gate (round-5 fix) makes the drain write's scan coverage a hard PR2 merge blocker** ŌĆö either the platform-wide `SafetyLayer` ingress wiring lands and the drain adopts that scan point like every other model-bound input, or (if platform wiring is not yet live) a drain-scoped scan call is added directly at the single drain write site. Either way, `spawn_subagent` is never cleared from `disabled_capability_ids` with the drain unscanned.
- No auto-cancel of orphans.
- No new crate; no feature flag.
- **No separate live-index/archive layer for closed edges (┬¦2).**
- **No gate expiry timers.**
- **No multi-node/HA correctness claims.**
- **No signature change to `RootFilesystem::delete`** ŌĆö the CAS primitive is additive (`delete_if_version`, ┬¦4.0), avoiding a ~20-call-site blast radius.
- **No content queue for human-priority activation (┬¦6a)** ŌĆö the next-slot reservation stores a marker, never a payload; no ordering, injection, or budget semantics to design for.
- **No per-time-window wake rate quota (┬¦8.3)** ŌĆö autonomous-wake containment is a count cap only; every `System` wake requires a real child run's full lifecycle, so a rate window would add tuning surface without adding containment.
- **Tripwires (flip these rulings if hit):** (1) #5466 turns out structural to CAS-over-libsql, not fixable at the RootFilesystem layer; (2) **retired by round-5's roster sharding (┬¦4.5)** ŌĆö was "a scope's roster directory becomes pathological to `list_dir` ŌĆö shard by tenant," now baked into the base design as 256-way hash-prefix sharding; its replacement escalation is widening the shard-prefix width (e.g. 3 hex chars, 4096-way) if a single shard's own scope count grows pathological, same hash function, no other change; (3) any scope's *open* edge count grows large enough that admission-time `list_dir` is measurably slow ŌĆö one CAS'd count file; (4) depth cap raised above 1 and ┬¦8/┬¦6 need more than the limits knob.
