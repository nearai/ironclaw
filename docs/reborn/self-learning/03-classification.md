# 3 — Learn What to Remember

Status: engineering design, proposed. Third increment of the [self-learning design](00-overview.md); it builds directly on the signals, provenance, scope, expiry, and lifecycle substrate from increment 2. The difference is that now the system learns things the user *didn't* explicitly ask it to — and that means, for the first time, something has to make a judgment about what a moment in a turn actually represents.

## The problem this slice solves

Most of what's worth remembering is never said as an explicit "remember this." The user mentions in passing that the team uses pnpm, not npm. A document the agent read says this repo releases on Fridays. A correction — "no, I meant the staging environment" — reveals a standing preference. The content of these moments arrives unlabeled: nobody hands the system a tag that says "this is a project convention." The job of this increment is to notice such moments, decide whether there's a durable fact in them and what *kind* of fact it is, and store it through the same scoped, attributed, expiring machinery built in increment 2 — without letting the model that noticed it also decide to commit it.

The reason "what kind" matters is that the kind determines everything downstream: a stable personal fact, a project convention, and a fact lifted out of a shared document have completely different scopes, lifetimes, and permission rules. Get the kind wrong and you either lose the fact or leak it.

## The approach

A signal still triggers everything, but now the interesting signals are the implicit ones — a task outcome, a user correction, a source document changing. There's a subtlety here worth being precise about, because it's the difference between a slogan and a working system. A moment in a turn does not announce itself; the host has to *detect* the moment, cheaply and deterministically, before anything else happens. That detection produces a coarse **signal kind** — not the durable fact kind, just the rough shape of what just occurred. A correction signal, for instance, fires when the current turn contains a negation or contradiction pattern aimed at the agent's own prior assertion ("no, I meant…", "actually it's…"); a source-changed signal fires when a tracked document's hash moves; a profile-changed signal fires when a settings write lands. These detectors are pattern-matchers, not classifiers — they are allowed to over-fire, because everything downstream is conservative. So the thing the router receives is never raw content; it's a typed signal whose *kind* was stamped at emit time by one of these coarse host detectors. The durable question — is the *content* a personal fact, a project convention, or a document-derived fact — is a separate, later question that only the model gets to read.

The signal flows from there through a four-step pipeline, and the entire safety story is in how little of it is a model. Three of the four steps use no model at all; the one that does uses a single, tightly constrained call, and even its output is only a proposal that a deterministic step then disposes of.

## The four steps, and where the model is

**A router sorts the signal into a rough lane — a lookup, no model.** Because the signal's *kind* was set deterministically by the host detector that emitted it, the router can map it to a family with no judgment: a changed document is heading toward a document-derived memory; a stable setting change is a profile fact; a user correction is a preference-or-rule that gets pinned down in a moment. The router is classifying the coarse host signal kind, not reading the content — it commits to nothing expensive; it just narrows the lane.

**An evidence pack is rebuilt from the source references — deterministic, no model.** Following the signal's references back to the actual material (the message window, the document excerpt), it re-redacts and caps everything to a small bundle. Redaction here preserves relative-time phrases like "until Friday" or "for two weeks" verbatim, because a later deterministic step needs to read them. If the references don't resolve, the candidate dies here: without a resolvable source it can never be permission-checked, and a memory that can't be permission-checked can never be safely surfaced. This is the same provenance backstop from increment 2, now load-bearing.

**A single constrained model call reads the evidence and proposes a content label — and this is the only model in the pipeline.** It's here because the job genuinely needs reading comprehension: telling a durable personal fact ("has two kids") from a project convention ("releases on Fridays") from a fact lifted out of a document is a reading task, and a rule can't do it reliably. The call reads the redacted evidence and returns a **fact kind** from a fixed list, a one-line summary, and a suggested trigger. This "model proposes a typed item" pattern is well-trodden — mem0 extracts facts this way, Letta organizes memory into typed blocks, A-MEM writes structured notes, and the underlying split between declarative facts and procedural know-how is the old distinction from the cognitive-architectures literature (Soar and ACT-R, declarative versus procedural memory). What we deliberately *don't* let the model do is the entire difference: it proposes only the content label, the summary, and the trigger. It does not choose the operation, the scope, the expiry, the risk, or whether the thing goes live. mem0 lets the model pick the operation — add, update, delete, no-op — and that convenience is exactly where an unverified self-edit does damage; we don't cross that line. The suggested trigger is also constrained at the boundary: it may only name entities that actually appear in the evidence pack, because a trigger that reaches for entities the source never mentioned is a softer form of the same scope leak the policy step exists to prevent.

**A policy step disposes of the proposal — deterministic, no model.** This is where the safety actually lives, and it runs in a fixed order. First it runs deterministic personal-content detection over the evidence — PII patterns, private-aside markers, anything that reads as personal — *before* it considers any scope at all. Anything that trips that detector is floored to the owner's user scope and stays there regardless of what the model labeled it. Only then does it consider scope at all: it starts from the user-scope floor and narrows to the smallest of the model's proposed kind and what the source itself permits. Widening past the user floor to project or shared scope is allowed *only* on an independent deterministic signal — the source was itself derived from a project document, or a project-config keyword is present — never on the model's label alone. Scope, in other words, can only shrink from the source's permission, and can only grow past the user floor on evidence the host can check for itself. The step then computes the expiry — a per-kind category default, optionally tightened by a deterministic temporal grammar that reads the explicit phrases redaction preserved ("for two weeks" → two weeks, "until Friday" → that date) — with no model call anywhere in the expiry path. It assigns a risk level from a checklist — anything touching external writes, broad or global scope, private sources, or PII-like content is high. It sends high-risk or low-confidence proposals to human review (more on what "low-confidence" means below). On a duplicate it merges; on a conflict with something already active it does *not* overwrite — it writes the new item as a candidate and routes the pair to review, leaving the active item in place. And it rejects outright when there's nothing safe to keep. The model's proposal is just an input here; this step is what actually writes a candidate.

## What "low-confidence" means, and the salience score

Two small mechanisms deserve to be pinned down, because both are places a careless design would quietly hand a safety decision back to the model.

The first is confidence. Policy routes "low-confidence" proposals to human review, so confidence is safety-bearing — which is exactly why the *model* doesn't get to report it. If the model emitted its own confidence number, it would be self-certifying a routing decision, and a confidently-wrong read is precisely the failure we're guarding against. So "low-confidence" is defined as a deterministic host signal, not a model output: a proposal is low-confidence when the model's content label disagrees with the family the router derived from the host signal kind, or when the evidence falls below a specificity floor (too short, too vague, no concrete entity to anchor on). The property that matters is that this is strictly one-way — a low-confidence verdict can only push a proposal *toward* more review, never let one skip it. Confidence can add a gate; it can never remove one.

The second is salience. There's a small deterministic salience score available — importance computed from a category base plus a few high-signal keywords like "always," "never," or "policy" — and like everything numeric here it's computed by the host, not trusted from the model. (Generative Agents, by contrast, asks the model to score importance directly — fine for a simulation, but we'd rather keep a model out of a numeric, safety-adjacent loop when a heuristic will do.) Honesty requires a caveat, though: nothing in this increment actually *consumes* the salience number yet. It's recorded on the candidate, but the consumer is curation, which lands in increment 6; until then it's an inert field, and the real-world lesson from agents that record a signal and never wire it in is that an unwired field is a liability, not a feature. So salience is specified here and deferred there, deliberately, rather than pretended into use early.

## Why this split, and the failure it prevents

This is the increment where "model proposes, host disposes" stops being a slogan and becomes the actual code path, so it's worth stating the failure it prevents and being honest about exactly how far the guarantee reaches. If the model both labeled the fact *and* decided its scope, a single confidently-wrong read could write a private aside — "my therapist appointment is Tuesdays," dropped into a conversation whose audience happens to include the project — at project scope, and start surfacing it to everyone on the project. The earlier draft of this design called that leak "impossible by construction," and that was an over-claim: an intersection that lets the model's label select the scope family will still leak a mislabeled private fact up to whatever the source audience permits.

The honest, and still strong, claim is narrower in two pieces. First, splitting the content label (model) from the disposition (deterministic policy) bounds the blast radius to what the source audience already permits — a fact can never be surfaced more widely than the conversation it came from already allowed, no matter how the model labels it. Second, and this is the part that closes the leak above, personal content is floored to user scope regardless of the model's label, because the deterministic personal-content detector runs *before* any non-user scope is assigned and a non-user scope can only be reached on an independent signal the host verifies itself. Those two together are a real, checkable guarantee. "Impossible by construction" was not.

## A note on degradation

Because only one step uses a model, the pipeline degrades gracefully when the classifier model is unavailable — but "gracefully" has to mean something precise about what survives, because the whole point of the increment is that some kinds need a reading and some don't. The deterministic families flow through untouched: a revocation is applied, a profile change is written, because neither needs the model to read anything. A changed source is the interesting case — its candidate is *built* (evidence packed, redacted) and held, but it is **not** surfaced, because nothing has labeled its content yet and an unlabeled document excerpt could carry anything. It waits in the queue as a redacted candidate until the model comes back and labels it. So no turn is affected, nothing is lost, and nothing unlabeled is ever shown — the signal is durable until it's classified.

## What this sets up

After this increment the system learns durable facts on its own, not just when asked, and the classification pipeline — host detection, router, evidence, constrained model proposal, deterministic disposition — exists as a reusable shape. The next increment reuses it almost wholesale: a recurring *failure* is just another signal kind the host detects and the router recognizes, and a corrective *rule* is just another thing the constrained model proposes and policy disposes of. What the next increment adds isn't new classification machinery; it's a real verification gate, because a rule changes the model's behavior and an explicit fact does not.

## Open questions for this increment

The exact set of fact kinds the classifier may return is the central one — too few and distinct kinds get conflated, too many and the model gets unreliable. The deterministic specificity floor that gates "low-confidence" needs tuning against real traffic: set it too low and vague asides slip past review, too high and everything routes to a human. And while the conservative interim rule above — auto-merge true duplicates, route every active-item conflict to review — is enough to ship this increment safely, the full merge-versus-supersede policy, where a constrained model may *propose* a supersession that a deterministic step disposes of, lands later with curation, once contradictions become common enough to be worth handling well.
