# Reborn Self-Learning — Overview & Roadmap

Status: engineering design, proposed.

This is the index for a set of incremental design docs. Each one corresponds to roughly one shippable PR (or a tight stack of them), and they build on each other — from the simplest useful version to the full system. You can read them in order; each later doc assumes the ones before it have landed. The individual docs are:

1. **Make memory reach the model** — wiring stored memory into the prompt: the context planner, hybrid search, filtering, ranking, packing.
2. **Remember & forget** — the simplest write path (explicit "remember this / forget that"), which lays the substrate: signals, provenance, scope, lifetime, and the candidate→active lifecycle.
3. **Learn what to remember** — learning facts the user didn't explicitly state, by classifying what happened in a turn.
4. **Never make the same mistake twice** — turning recurring failures into verified behavioral rules. This is where verification becomes essential.
5. **Reusable procedures** — distilling repeated successes into runnable skills, verified by execution.
6. **Keep it healthy and honest** — curation, measurement, and the human controls that keep the whole thing from rotting and prove it actually helps.

## The problem

Reborn handles a user's requests one turn at a time, and it's good at any single turn — but it doesn't carry much forward. The same tool fails the same way on Monday and again on Thursday, and nothing remembers the first time. A user says "keep your summaries short" and it holds for the conversation, then evaporates. A three-step procedure that worked last week gets reinvented from scratch this week. The agent has a memory store it can search, but searching a store is not the same as learning: the store is passive, and nothing curates it, scopes it, or decides when last week's note is worth surfacing today.

The obvious move — let the agent write to its own memory and prompt as it works — is exactly the move that gets self-improving agents into trouble. Now the model is editing the instructions it runs under, in the middle of running them, with nothing checking that the edit is correct, still in scope, still permitted, or actually an improvement rather than a confidently-wrong guess. Add a second user and it gets worse: one person's private fact quietly leaks into another's context because nothing tracked where it came from or who's allowed to see it. The research points the same way: unbounded, un-curated, self-verified memory tends to make agents *worse*, not better. That's the regime we're avoiding — none of it says memory itself is a bad idea, only that this particular shape of it is.

What we want is narrower and harder: an agent that reliably doesn't make the same mistake twice, that picks up durable preferences and reusable procedures, and that does all of it without ever letting the model rewrite its own future on a whim, without leaking across users, and without drowning the prompt in stale notes.

## The end state, in one breath

The system splits in two. A **fast path** handles any agent execution — an interactive turn, but equally a background job, a routine, or a heartbeat — and never blocks on learning; while it runs it just emits small **signals** — "this tool failed with an auth error," "the user said remember this." A **slow path** consumes those signals out of band and turns the useful ones into **learned artifacts** — memories (facts), rules (behavioral guardrails), and skills (reusable procedures) — each scoped to who it belongs to, attributed to the source it came from, and given a bounded lifetime: an expiry for things that go stale, or "until changed or revoked" for facts that stay true until something contradicts them.

The gate that follows differs by what the artifact does. Anything that *guides a decision* or *runs as a procedure* — a rule, a skill — has to clear an external signal before it goes live: replayed against past runs to prove it wires in, then re-run for real to prove it actually changes the outcome. A plain fact that's backed by a resolvable source clears that same gate trivially — its "verification" is just that its source still exists and still permits it — so the strong invariant is not that the model never asserts anything until proven, but that nothing the model *would have us act on* changes behavior until an external signal says it should. Memory reaches the model only as a small, budgeted bundle the context planner assembles per request, filtered by permission and freshness at the moment of use.

This out-of-band shape is the same one the better memory systems converge on, and it's worth being precise about which half each one demonstrates. mem0 keeps a fast write path but defers its background **summary** refresh — consolidating raw memories into a running summary off the critical path — which is the slow-path move we're borrowing. Letta runs background "sleep-time" agents that revise memory while no turn is waiting. On the in-turn side, MemGPT's context paging swaps memory in and out *during* the turn under a budget — that's the budgeted-bundle half, the read path, not the out-of-band learner — and Generative Agents reflect periodically but *in the loop*, synthesizing higher-level observations as the simulation runs rather than on a decoupled plane. We take the slow-path discipline from the first group and the budgeted-read discipline from the second.

## Two principles that run through every increment

**The model proposes, the host disposes.** A turn can flag that something is worth learning, but a deterministic, host-owned pipeline — not the model — decides what actually gets stored, what scope it's allowed, how long it lives, and whether it's ever used. The model appears only where a decision genuinely needs reading comprehension — is *this* the same failure as *that* one, does *this* sentence mean "for two weeks" — and never where it would make a trust- or safety-bearing decision about *activation*: whether a learned thing goes live, what audience it reaches, when it's surfaced. (Measurement is the deliberate, bounded exception — increment 6 lets a model *grade* an open-ended output when success can't be checked mechanically, but only as a yardstick, never as the gate that promotes anything.) This is the single thing most self-learning designs get wrong: mem0, for instance, lets the same model both extract a fact *and* decide to add, update, or delete it — convenient, but exactly where an unverified self-write does its damage. We take the operation and the activation away from the model on purpose. It's the same lesson the self-correction literature keeps landing on — that a model asked to check its own work tends to ratify it — which is why this design routes verification through an external signal rather than the model's own judgment.

**No persistent, cross-user learning during the turn.** Everything expensive — figuring out what a signal means, whether it's worth keeping, whether acting on it actually helps — happens after the turn, out of band, where it can take its time and spend a few tokens without anyone waiting. The turn itself only emits cheap signals, with one deliberate exception: a "forget that" is honored synchronously at end-of-turn, because a revocation can't wait for the slow path to catch a leak. The point of the rule isn't that *no* learning may ever touch a live conversation — it's that nothing learned in one user's turn may persist or cross into another user's context without going through the out-of-band pipeline first. (Whether a strictly ephemeral, never-persisted, never-shared *session* scratch tier — the thing that would catch "you made that same mistake two messages ago" within a single conversation — is worth adding is a real open question; it's the highest-frequency win and the safest place to relax the rule, and the team will decide it on its own rather than fold it into this sequence. We don't build it here; we just stop pretending it's categorically unsafe.) If learning is slow, disabled, or broken, the user's turn is unaffected either way.

This is, in spirit, the Reflexion idea — an agent that learns from its own failures by storing reflections it can consult later — but with its known weakness designed out. Reflexion's reflections are only as good as the signal that produced them; a reflection written off a noisy or self-graded outcome is a confidently-wrong lesson stored as fact. So we keep the "remember your failures" instinct and replace the self-evaluation with an external check before anything goes live.

## Why this order

The sequence is value-first and risk-last, and the order of the docs above already states *what* each increment is; what's worth pinning here is *why* each handoff is shaped the way it is.

Retrieval comes first because memory has to reach the model before anything else matters, and it pays off immediately against the memory that already exists, with no learning machinery at all. The simplest write — honoring an explicit "remember this" — comes next not just because it's a real feature on its own, but because it's the cheapest place to build the substrate everything later leans on. Two substrates get laid here. The first is the signal-and-lifecycle plumbing: signals, provenance, scope, lifetime, and the candidate→active lifecycle, all at low risk because an explicit, source-backed fact needs almost no verification to trust. The second, quieter one is a minimal store-side floor — a per-scope size cap and a staleness floor — so the increments between this and full curation aren't shipping *pure* un-curated accumulation, which the system itself treats as net-negative.

Only then do we let the system learn things the user didn't ask it to, the first point where classification has to make a judgment call. The headline capability — never making the same mistake twice — comes after that, because it's the first time a learned thing changes the *model's own behavior*, and so the first time real verification is non-negotiable. That's where the *second major* substrate arrives: a verification plane — a replay harness to prove a rule wires in, a budgeted live re-run to prove it actually changes the outcome, a control set to prove it doesn't fire where it shouldn't, and background amortization so all of this fits a fixed budget. Skills follow naturally and are the easy case for verification, because runnable code can be checked by running it — CodeT's trick of trusting code that passes independently-generated tests rather than the model's say-so is the same instinct. Voyager, the canonical reusable-skill library, is the warning attached: it grew a genuinely useful skill library, but it gated additions behind a *live LLM critic* — a model judging the model — and this design deliberately departs from that by gating behind external verification instead. Finally, the health-and-honesty layer — curation and measurement — comes last not because it's optional but because it has the most to curate and measure once the rest exists; un-curated memory is the thing the research shows degrades agents, so this is the increment that keeps the whole thing earning its place.

Each increment is independently shippable, and — because the store-side floor from increment 2 protects the gap — each leaves the system in a state worth running rather than the un-curated heap full curation later cleans up. But it's not true that none of them changes the shape the others assume: increment 4 introduces the verification plane, a second substrate the earlier docs don't presuppose. Increments 1–3 build and consume the signal-and-lifecycle substrate; from 4 on, rules and skills also stand on verification. The order is what lets each substrate land where it's first genuinely needed, instead of paying for both up front.
