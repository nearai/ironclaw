# 1 — Make Memory Reach the Model

Status: engineering design, proposed. First increment of the [self-learning design](00-overview.md); read the overview for the whole arc. This one has no learning in it at all — it makes the memory that already exists actually reach the model, well. Everything later depends on this read path working.

## The problem this slice solves

Reborn already has a memory store, and the model can search it with a tool when it thinks to. But that's the catch — memory only shows up when the model explicitly goes looking, and a model mid-task often doesn't think to look. There's a budgeted prompt-assembly layer in the runtime that *could* carry memory into the prompt automatically, but the path that fills it is currently unwired, so in practice proactive recall doesn't happen. The first job is to wire that path and do retrieval properly: get the *right* handful of memories into the prompt before each model call, and not a hundred wrong ones.

That "not a hundred wrong ones" is the whole difficulty. Dumping everything even vaguely relevant into the prompt is the naive version, and it backfires — it crowds out the actual task, and in the regime where the raw history still fits the window (the LOCOMO benchmark sits around 16-26K tokens), a naive memory dump loses to simply pasting in the raw history. That comparison flips once the history *doesn't* fit: on a long-context benchmark like LongMemEval-S, where full sessions run past 100K tokens, dumping everything degrades and curated retrieval is what wins. Either way retrieval is a selection problem, and it belongs to one component that owns the decision.

## The approach

A **context planner** runs before each model call and assembles the small set of context that belongs in the prompt. It uses no generative model to do this; it's a deterministic pipeline, which keeps it fast and reproducible (the only LLM involvement in the whole read path is an optional mid-turn top-up, described at the end). It builds a short structured description of the moment — who the turn is for, the conversation, the active project or skill, the literal task text, and the concrete things named in the turn (tool names, error strings, ids) pulled out with plain pattern-matching. Then it gathers candidates from a few different places, filters them, ranks them, and packs the top of them into the prompt within a token budget.

The reason it's several places and not one search is that "relevant memory" isn't one kind of lookup, and treating it as one is how you get the hundred wrong items.

## How retrieval actually works

**Some context is loaded by key, not searched.** Your timezone, your locale, a standing "keep it terse" preference — a tiny, stable, always-on slot read directly by key. The honest caveat is that these aren't memory documents; timezone and locale are account settings, and the standing preference is the kind of thing increment 2's explicit-remember path will write. For this increment that slot is best treated as a named pull from settings/profile rather than from the memory store — or, if the source isn't wired yet, an explicit placeholder. Either way the discipline against it turning into a junk drawer can't be vibes: it's a fixed allowlist of keys (or a hard cap on the slot), so nothing accretes into it by accident.

**Some context is relevant only because of where you are in the work.** A note about this repo's release process should appear when you're doing a release and stay out of the way otherwise. So memory items carry tags for where they apply — a skill, a project, a capability — and the planner pulls the ones whose tags match what's active right now. This is a metadata filter, not a similarity search: "show me notes for the thing I'm literally doing," not "show me things that vaguely resemble it."

**Genuine recall from the store uses hybrid search**, which has become the standard answer for good reason. A keyword search is unbeatable at exact tokens — a function name, an error string, a ticket id, a path — and useless at paraphrase. A vector search, where the query is embedded and compared by similarity, is the opposite: great at "the same thing in different words," hopeless at exact strings. You want both, so you run both and merge the two ranked lists. There are two common ways to merge. *Reciprocal rank fusion* scores each result by one over a constant plus its rank, summed across the lists; it's nice because it doesn't care that the two searches produce scores on totally different scales. In Elasticsearch it's an opt-in `rrf` retriever rather than the default, while Azure AI Search uses it as the default for hybrid queries; agents like praktor reach for it too — usually with the constant near 60, which matches what ironclaw_memory already defaults to. The alternative is to normalize the two scores and take a weighted blend, which is what openclaw does (a 0.7/0.3 weighting, then MMR to spread out near-duplicates) and what Weaviate defaults to; it's more tunable but assumes you trust the magnitudes. The design treats the fusion method as a knob, not a commitment — and it's a knob the store already exposes, since ironclaw_memory's hybrid search supports both an RRF strategy and a weighted one.

What it does *not* treat as optional is graceful degradation, and that has two halves. On the store side: if embeddings aren't available, keyword search alone has to be a sufficient fallback (it's a famously strong baseline), and if even that isn't available the store must report "I can't search" rather than return an empty result. That last point sounds pedantic but it's a genuine failure mode — a search that silently returns nothing looks exactly like a search that found nothing, and the agent will act on the difference. The planner side has to honor the distinction the store just made: when recall comes back unavailable, the planner keeps every other lane working and packs a deterministic marker that says *recall was unavailable this turn*, so the model can tell "I looked and nothing matched" apart from "I couldn't look." Without that marker the failure collapses back into silence at the one layer that's supposed to prevent it.

**Recall over past conversations is a substrate we don't have yet, so it's deferred.** The trick that would make it work is to index not just the prose of past turns but the tool names, arguments, and error strings — so "how did we fix this auth thing last time" matches on the actual tool and error tokens, not just the chatter, returning a small anchored window (the goal, the resolution, a few surrounding messages) rather than a whole transcript. The problem is that nothing in the system does this today: ironclaw_memory's hybrid search runs over memory-document chunks, and nothing writes a per-turn trace — `tool_name`, args, error strings — into anything searchable. Standing this up honestly is a whole subsystem, not a lane in this increment: a write-time turn/trace indexer, an owning store for those traces, an anchored-window storage scheme with its own scope, and — because raw transcripts contain exactly the PII the rest of the design is careful about — transcript redaction, which would otherwise be forced a full increment ahead of increment 2's redaction substrate. Rather than smuggle all of that in here, session recall gets its own later increment; this increment ships the memory-document lanes and leaves a clean seam for it.

**Everything these lanes return is a candidate, not yet context.** Before anything goes near the prompt it passes a set of plain yes/no checks. For this increment those checks are exactly two things: is the run within this item's scope (does the item's tenant/user/agent/project contain the current one), and is the item fresh and permitted by settings. No model; just boolean comparisons, and either can drop a candidate however relevant it looked. It's worth being precise about what these checks *don't* do yet, because the comparator below has the same property. There is no audience check here, because audience doesn't exist yet — memory scope today is exactly tenant/user/agent/project, with no per-message ACL; audience is defined as new structure in increment 2 (the set of thread participants at send time, captured at the channel/turn layer). And there is no "is it active" check, because there's no candidate/active state until increment 2 either. So the increment-1 filter is scope-containment plus freshness/settings, and nothing more; the audience and active/candidate checks slot into this same gate once their substrate lands.

**What survives is ranked by a fixed comparator, not a model.** The order of tie-breakers encodes the priorities: an item that matched a precise trigger and is narrowly scoped beats one that merely scored well on similarity; then the search score; then how reliable the source is; then how useful the item has been before; then recency; then sheer compactness so a shorter item wins a tie and protects the budget. Two of those keys are inert no-ops in this increment, the same way the audience filter is — and for the same reason, so it's worth saying plainly rather than letting them read as live. "How reliable the source is" needs provenance, which arrives in increment 2; until then every item scores identically on it. "How useful before" needs usage data, which the curation increment (6) produces; until then it's a constant too. They're in the comparator now so the ordering is stable as the data lands, but for increment 1 the real discriminators are trigger-match-and-scope, then search score, then recency, then compactness.

It's tempting to use an LLM to rerank, and some smaller agents do, but a generative chat model is the wrong tool *for that job*: a model call on every turn, different answers to the same inputs, and the evidence is that chat models aren't reliable relevance judges anyway — openclaw, one of the more disciplined systems, pointedly uses a deterministic multi-signal score for exactly this. That argument is specifically against *generative* reranking; it is not an argument against reranking as such. A deterministic cross-encoder — a small fixed-weight model that scores a (query, candidate) pair — is a different animal: given its weights it returns the same score for the same inputs, it runs in single-digit milliseconds, it's cacheable, and it satisfies every property the fixed comparator is chosen for. So an optional cross-encoder rerank over the top-K survivors, just before packing and behind the same graceful-degradation fallback as the embeddings, is a reasonable enhancement to reach for if the fixed comparator's ordering proves too coarse. It stays optional and it stays deterministic; the "no LLM reranker" rule means no chat-model-in-the-loop, not no learned scorer at all.

**Finally the planner packs the winners slot by slot**, each slot with its own token budget, skipping near-duplicates and recording *why* anything was dropped. That omission record is worth being honest about, because it's easy to wave off as nearly free and it isn't. It's a first-class, permission-sensitive, deferred-write artifact: it stores a reference to the item and a reason from a fixed enum (over budget, filtered, duplicate, low rank) — never the withheld content, because "dropped: audience mismatch" against a memory you weren't allowed to see would itself leak that a forbidden memory exists. So it's written off the hot path, it carries retention, and reading it back is itself permission-checked. The reason to pay for all that is increment 6: "why wasn't this memory used?" is exactly the question the curation-and-measurement layer hard-depends on, and there's no way to answer it after the fact unless the read path recorded it at the moment of the decision.

**And if the model needs more mid-turn, it asks through a retrieval tool.** It states what it's looking for and the host runs this very same pipeline again with a smaller budget. The model never reaches into the store; it only asks — and the host answers under the same scope (and, once it exists, audience) gate as the deterministic path, which means an out-of-scope ask is *refused*, not merely ranked low. The host also excludes anything already packed this turn, so the model doesn't burn its mid-turn budget re-reading what it can already see. This is the one non-deterministic entry into an otherwise fully deterministic read path: the model chooses *when* and *what* to ask, but the host still disposes of the answer. The "retrieve as a tool the model calls" pattern is what MemGPT and Letta do with their memory tools and what the agentic-RAG line of work formalized, and the nuance everyone converges on — which we follow — is that it's not either/or: you preload a little always-on context and expose a tool for the long tail.

One thing to be clear-eyed about: this whole pipeline — gather, filter, rank, pack — runs on *every* loop iteration, not once per user turn, and each run does two backend queries (keyword and vector) plus a fusion. Within a single turn the moment barely changes from iteration to iteration, so this is a real and avoidable cost. The mitigation is caching across iterations within a turn, keyed on a fingerprint of the planner's structured description of the moment; when the fingerprint is unchanged the previous candidate set is reused and only the budget-packing is redone. That keeps the per-iteration cadence from turning into per-iteration query load.

## What this sets up

After this increment, the agent's existing memory genuinely helps — it surfaces at the right time without the model having to go looking — and there's a single, well-understood place (the planner) where every "what does the model see" decision is made. That matters because every later increment produces things the planner has to choose among. Learned facts, behavioral rules, and skills will all flow through these same lanes, the same filters, and the same ranking. Build retrieval well now and the rest of the system gets to assume it.

There's one boundary this increment should flag even though it doesn't act on it: the memory that exists today was written before provenance existed, so once increment 2 makes "no resolvable source → never surfaced" a rule, that legacy memory needs somewhere to land. Increment 2 owns the answer (grandfather pre-provenance items into a narrow owner-only scope with a synthetic source marker so nothing is silently dropped); increment 1 just needs to know the seam is there, because the filters it builds are exactly what will start enforcing the new rule.

## Open questions for this increment

The exact per-slot token budgets, the rank tie-break weights where ties are frequent, and the fusion method and its constant per provider are all tuning, not architecture. The mid-turn retrieval budget — how much the model is allowed to pull when it asks for more — is a policy choice. The initial set of slots (how finely to divide "profile" from "task" from "retrieved memory") is worth pinning down against real prompts before freezing it. The optional cross-encoder rerank is a build-if-needed enhancement, not a commitment, and whether it earns its place is an open question for after the fixed comparator is measured.

Two contracts are under-specified enough to call out on their own. The first is **entity extraction** — the "concrete things named in the turn" the planner pulls out with pattern-matching. That needs pinning before it ships: which text surface it reads (the literal task text and the latest turn, not the whole conversation), a bounded set of entity classes with concrete patterns (tool names against the known tool registry, error codes/identifiers, ticket-style ids, file paths), an explicit does-*not*-extract list so it doesn't drown the keyword query in noise (bare UUIDs, long hex blobs, generic paths), an input-length cap with non-backtracking matching so a wall of pasted text can't ReDoS the fast path, and a graceful fallback to using the literal task text as the query when extraction yields nothing. The second is the **omission record's persistence and permission shape** — its retention window, where it lives, and the exact permission check on read-back — since increment 6 reads it and a leak through it is a real risk. Both want to be settled before the read path is frozen, since later increments build directly on top of them.
