# 4 — Never Make the Same Mistake Twice

Status: engineering design, proposed. Fourth increment of the [self-learning design](00-overview.md). This is the headline capability — an agent that stops repeating a known failure — and it's also the first increment where a learned thing changes the *model's own behavior*, which is the first time real verification is non-negotiable. It reuses the classification pipeline from increment 3 almost unchanged; the new substance is the failure-to-rule path and, above all, verification.

## The problem this slice solves

The most visible way an agent looks dumb is repeating a mistake it already made: retrying a call that fails for a reason retrying can't fix, asking for a missing artifact only after it was needed, reaching for the wrong tool in a situation it's seen before. A memory of a *fact* doesn't fix this; what's needed is a small change in how the agent *decides* — a guardrail like "after this capability returns this auth error, stop retrying and ask the user to reconnect." That's a **rule**: compact behavioral guidance, scoped to where it applies, that the planner drops into the prompt only when its trigger is active.

The danger is equally specific. A rule is, functionally, the model editing the instructions it runs under. A wrong rule doesn't just fail to help — it actively steers the agent wrong, every time its trigger fires, until something catches it. So the bar for letting a rule go live is much higher than for a fact, and meeting that bar is most of this increment.

A note on scope before going further. This increment builds **reactive** guardrails only: a rule is keyed on a *(capability, error-category)* pair and fires after that failure has been observed, to keep the agent from walking into it again. The preventive cases above — "ask before acting," "you're reaching for the wrong tool" — are real and worth catching, but they need a richer *situation* trigger that fires on the shape of a request rather than the wake of a failure, and that's a different mechanism. We list them here because they motivate the feature, but the retrieval key this increment ships matches them only when the wrong-tool reach actually *failed* and left an error category behind; a genuinely preventive trigger is deferred to a later increment. Keeping the motivation and the trigger key in step matters, because a rule that promises to fire on a situation it can't actually detect is worse than no rule.

## The approach

Two new pieces sit on top of the existing pipeline. First, recurring failures are condensed into a **failure signature** and, past a threshold, proposed as a rule — which is mostly the increment-3 pipeline with a new family. Second, and this is the real work, a **verification** step proves the rule actually changes behavior in the intended way and doesn't misfire, before it's ever trusted. Nothing about a rule reaching the model is allowed to rest on the model's own opinion that the rule is good.

## From repeated failure to a candidate rule

When a capability call fails, the executor already knows the capability, a normalized error category, and the auth/approval state — it emits a failure signal straight from those facts, no model. The idempotency key from increment 2 now earns its keep: identical failures collapse onto one record and *increment a count* rather than piling up. That count is the whole trick for keeping this affordable — it turns "a failure" into "a failure that has now happened five times," and only once it crosses a threshold does the system spend anything on it.

That counter is shared state mutated from the slow path, so it has to be handled like one. The increment happens DB-side and atomically, so two failures landing at once can't both read-modify-write a stale value; redeliveries of the same underlying failure — the same end-of-turn signal arriving twice because something retried — are deduplicated on the idempotency key before they touch the count, so the number tracks *distinct* failure events and not delivery noise. The point of the threshold is to spend effort on a genuinely recurring problem, and that only holds if the count means what it says.

Past the threshold, the classifier proposes the rule's text — "on this capability and this auth state, stop retrying and ask the user to reconnect" — the same constrained-model-proposes step from increment 3, and a deterministic policy step disposes: it sets the rule's trigger key to the capability and failure family, assigns its risk, and marks it verification-gated. Crucially, the rule also inherits **owner and audience scope** from the provenance of the runs that produced the failure — capability and failure family are only the *trigger* (the question of *when* the rule fires), never a statement of *who* it may fire for. A failure that recurred entirely within one user's private workspace yields a rule scoped to that user; it does not become a guardrail that reshapes everyone's behavior just because the capability is shared. It's now a candidate rule: stored, never used, waiting on verification.

Retrieval already knows how to surface it. Rules ride the same tag-and-metadata lane the context planner from increment 1 uses for everything it fetches — but a rule is never matched by similarity. It's fetched by *exact* match on a new key this increment builds: the active capability plus the live failure family. A guardrail that fires when something merely *resembles* its trigger is worse than no guardrail, so similarity has no place here. Candidate rules are excluded from that fetch entirely — only verified, active rules are reachable — and retrieval applies the same scope and audience checks it applies to any learned item before injecting one, so a rule scoped to one user is never dropped into another's prompt even if the trigger key matches.

## Verification: the part everyone gets wrong

Here is the single most important design stance in the whole system, and it's worth stating bluntly: **the thing that decides whether a rule goes live is never the model's opinion of the rule.** This isn't a preference; it's where the evidence in the area is strongest, and it comes from two distinct places that are easy to run together.

The first is that *verification matters at all* — that gating learned behavior on a real check, rather than letting it in ungated, is what separates a skill library that improves an agent from one that degrades it. Voyager is the cleanest demonstration: it grew a genuinely useful library of reusable skills, and when its authors ablated the feedback the system learned from, capability collapsed. The largest drop came from removing its automatic curriculum (about -93%); removing its self-verification step — the check that decided whether a freshly written skill was good enough to keep — cost about -73%, the most important of its three *feedback signals*. That -73% is the number people cite for "verification matters," and it does. But it's worth being precise about what Voyager's verifier actually *was*: a GPT-4 critic, a live model call judging the model's own output. So Voyager argues *for verification* — and, read honestly, it's an example of verification done with a model in the loop, not against one.

The second place — and the one that says the check must *not* be the model grading itself — is the self-correction literature, which the overview already leans on: a line of results (notably arXiv 2310.01798 and 2402.08115) showing that a model asked to check or correct its own reasoning, without an external signal, tends to ratify what it already produced. That, not the Voyager number, is what forbids self-certification here. Reflexion (Shinn et al., arXiv 2303.11366) is the canonical precedent for the *good* half of this — an agent that learns from its own failures by storing reflections it consults on later attempts — and also for the cautionary half: a Reflexion reflection is only ever as good as the evaluator signal behind it, so a reflection written off a noisy or self-graded outcome is a confidently-wrong lesson stored as fact. Taking the "remember your failures" instinct while replacing the self-evaluation with an external check is the whole move. So this design's verification is deliberately *not* Voyager's: where a rule's verification needs a behavioral judgment it uses an execution outcome, not a critic model, and for the skill case in the next increment the same instinct shows up as CodeT's trick — trusting generated code that passes independently-generated tests (a reported +18.8% on pass@1) rather than the model's say-so about its own code.

The cautionary contemporary case is Hermes, otherwise one of the more capable self-learning agents in the comparison set. What it shows plainly is what *not* to ship: it lands learned behavior with its approval gate **off by default**, and the only thing standing between a learned change and production is a regex scanner — which catches some obvious bad strings but is not, and was never meant to be, an approval gate. Learned behavior reaches the agent unverified. We don't do that; the gate below is the answer to it. (Hermes is also not a model to copy on cost discipline — it ships without real cost controls — so nothing about *how* we afford verification leans on it.)

A rule is hard to verify precisely because it changes a *model's* behavior, and the only way to know if it worked is to watch the model behave. So a rule gets two checks, in order of cost.

The first is a **replay** of the recorded failing run with the rule injected. This is deterministic and effectively free, and it's tempting to read a behavioral result into it — but it proves only one thing: that the *wiring* is right, that the rule is retrieved, matched, and dropped into the prompt at the moment its trigger is live. It proves nothing about behavior, because a replay reuses the canned model responses recorded the first time; the model in a replay literally cannot react to a rule that wasn't there when those responses were captured. So replay is a regression lock — it catches the day someone breaks retrieval so the rule stops reaching the prompt — and not evidence that the rule changes what the agent does. Every behavioral claim has to come from the second check.

The second is an actual **live re-run** of the failing task with the rule applied, at a fixed temperature, and this is the only thing that speaks to behavior. It isn't a single run, because one sample of a stochastic model proves little — it's a small fixed-N multi-sample pass-rate, so an accept is "the rule moved behavior reliably," not "it worked once." And it asserts two things at once: that the case which used to fail now goes right, *and* that a set of unrelated **control cases** still behave, so the rule isn't over-firing into situations it shouldn't touch.

What "goes right" means is checked structurally on the recorded action trace, not by a model grading the transcript — and it differs by what the rule is for. For the wrong-tool kind of failure, the win is that **the task now succeeds**: the recorded output checks that used to fail now pass. For the auth kind, the win is narrower and more interesting — it's a **decision change**, asserted on the trace itself: the agent no longer issues the futile retry it used to, and the reconnect prompt is present. The point isn't that the auth task suddenly completes (it can't — the credential is genuinely missing); it's that the agent now makes the *intended decision* in the face of that, which is exactly what the rule was supposed to install. Reading that off the structure of the action trace, rather than asking a model whether the transcript "looks better," is what keeps the gate honest.

The control set is what makes "doesn't over-fire" falsifiable, so its construction belongs in the design and not the open questions. By default it's assembled from two pools: recent *successful* runs of the same capability that did **not** hit this failure family, and runs of sibling capabilities. Before any live re-run is even scheduled, a deterministic check asserts that the rule's trigger does **not** fire on any control case — and a trigger firing on a control is an automatic failure, no re-run needed. That ordering is deliberate: most of the ways a rule misbehaves are visible from the trigger alone and can be caught before spending a single model call. Only the controls that survive that deterministic screen, alongside the formerly-failing case, go to the live multi-sample re-run. The decision to accept is that external pass-or-fail across both halves — never the model's say-so.

All of this runs in the background on a budget, and it's worth being honest about where the cost actually sits, because a live re-run is **output-bound**: most of its expense is the model generating fresh decisions, and prompt prefix caching trims the *input* side only, so it helps but it isn't the lever. The real levers are upstream of the model. First, the deterministic trigger screen above eliminates most control cases *before* any model call — a rule that fails on a control never reaches a re-run at all. Second, the recurrence counter rations live re-runs to the failures that have actually recurred enough to be worth a real call, which is what it's been quietly setting up since the first failure signal was emitted; a re-run after every turn would be ruinously expensive, and nothing of the sort happens. What's left — the genuinely-recurring candidates that pass the deterministic screen — is batched and amortized on the background plane against a fixed budget. That amortization is this design's own discipline, not borrowed from anyone in the comparison set.

## Lifecycle, now with a real gate

The candidate→active lifecycle from increment 2 finally uses its verification gate for something real, and a candidate that fails doesn't simply vanish — it lands in a specific state with the reason recorded, and the state determines what happens next.

A candidate rule becomes **active** when the live re-run passes and the rule's risk is low. Risk is judged by criteria that are computable at proposal time, not vibes: a rule that would **apply broadly** is one whose trigger matches more than one capability or uses a wildcard failure family; a rule that would **expand what the agent can do** is one whose prescribed action invokes a tool that wasn't in the failing run's allowed set; a rule that touches **external writes** is one scoped to a write-capable capability. Any of those routes the candidate to a human before it can go live, because those are exactly the rules whose blast radius is too large to promote on an automated pass alone.

When verification doesn't pass, the state machine is explicit. A re-run that fails for a **transient** reason — the live call itself errored, a dependency was briefly unavailable — leaves the rule a *candidate* and lets it retry, up to a bounded maximum attempts, so a flaky moment doesn't condemn a sound rule. A rule that **fires but doesn't change the outcome** — the wiring is right, the trigger matches, but the multi-sample re-run shows behavior didn't actually move — goes to **needs-repair**: the rule text is the suspect and the proposal is sent back to be reworked, not retried as-is. A rule that **over-fires on a control case** is **suppressed** outright, because firing where it shouldn't is the one failure mode there's no safe version of. The word **demote** is reserved for one transition and one only: an *active* rule that a later regression catches misbehaving in the wild is demoted back to *candidate* for repair. Keeping that vocabulary tight matters, because "demote" and "suppress" and "needs-repair" are different fates and conflating them hides which one actually happened.

One subtlety the verification plane forces on the lifecycle: a live re-run takes real wall-clock time, and in that window the candidate is still subject to the normal decay and expiry that keep stale learned items from accumulating. Letting a candidate quietly expire *while its own verification is still in flight* would be a bug — the system would discard a rule it was in the middle of proving out. So decay and expiry are **paused for any item that has a pending verification candidate**, and resume (or the item's lifetime extends on a fresh recurrence) once verification resolves. Verification latency and the staleness clock have to be reconciled deliberately, not left to race.

Staging behavioral changes behind a gate like this is what the careful agents do and the careless ones skip. A picoclaw fork we looked at, for instance, moves a learned change through a real ladder — a learning record becomes a draft, a draft becomes active only after it clears a success-ratio bar with enough similar records behind it, and an unused one eventually goes cold and is archived — with the agent itself running in observe, draft, or apply modes depending on how much it's allowed to act on what it's learned. (OpenClaw's Skill Workshop is the cleaner public reference for the staging idea if picoclaw's internals are hard to pin down.) The names differ; the discipline — nothing learned goes live until it's earned it — is the part worth copying.

## What this sets up

After this increment the agent demonstrably stops repeating known failures, and — just as important — the verification machinery exists: replay as a free regression lock that proves a rule wires in, a budgeted live multi-sample re-run as the real behavioral gate, a control set that proves a rule doesn't fire where it shouldn't, and the principle, now enforced in code, that a model never certifies its own learning. The next increment, skills, gets to reuse this gate — and discovers its job splits. A skill that's a runnable snippet can be verified the cheap way, by executing it against its own checks in a sandbox with no live model in the loop; a skill that's really a prose procedure changing how the agent acts has to be re-run for real, the same cost class as a rule. "Free" there will mean free of model *tokens*, not free of compute, and it'll run on this same background plane against this same budget.

## Open questions for this increment

The recurrence threshold (how many distinct failures before a rule is even proposed) and the live-verification budget per owner are the load-bearing knobs — too eager and it's expensive and noisy, too conservative and the agent keeps repeating mistakes longer than it should. The control-set construction has a default here, but its *parameters* — how many control cases is enough, how far afield the sibling capabilities should reach — want tuning against real traffic, since "doesn't over-fire" is only ever as good as the cases you check it against. And the auto-promotion thresholds by risk level — which rules may go live on a passing re-run alone versus which always need a human — are a product decision as much as an engineering one. The preventive, situation-triggered guardrails this increment deliberately leaves out are the obvious next frontier once the reactive ones are proven.
