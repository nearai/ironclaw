# 2 — Remember & Forget

Status: engineering design, proposed. Second increment of the [self-learning design](00-overview.md); it assumes retrieval (increment 1) is wired. This is the simplest possible write path — honoring an explicit "remember this" or "forget that" — and it's chosen deliberately, because building it forces us to build the substrate every later kind of learning depends on: signals, provenance, scope, expiry, revocation, the ledger, and the candidate→active lifecycle. But it does all of that at low risk, because a fact the user explicitly stated, attributed to the message they said it in, needs almost no verification to trust.

## The problem this slice solves

When a user says "remember that I prefer metric units" or "I'll be in London for two weeks" or "forget what I said about the old API key," that should *stick* — durably, scoped to them, and (where it makes sense) with an expiry — and "forget" should actually stop the thing from reaching future prompts. Today none of that survives the conversation. The hard part isn't storing a string; it's storing it *safely*: tied to where it came from so permissions and revocation work, scoped so it doesn't leak to another user, and with a lifetime so "visiting London" doesn't haunt the agent a year later. And "forget" hides a second hard part — *forget what?* — that the clean-sounding version quietly skips.

The closest thing shipping in a real product is ChatGPT's Memory (and Gemini's Saved Info): a user tells the assistant to remember a preference, it persists across sessions, and the user can ask it to forget. Two things about how those products actually behave are worth stealing up front. They run on a *small fixed budget* — on the order of a thousand-odd words of saved memory, not an unbounded store — which is a tacit admission that more remembered text is not strictly better. And their deletion is *eventually consistent*: OpenAI's own guidance is that a deleted memory may keep being referenced for a while before it fully stops. We'll lean on both of those facts rather than pretend we can do better.

## The approach

The turn doesn't write memory; it emits a **signal**, and a background pipeline turns that signal into a stored, scoped, expiring memory. This is the two-plane split the overview sets up: the turn stays cheap, and everything that needs time or judgment happens out of band. For explicit remember/forget that pipeline is, for the explicit case, *genuinely* deterministic — the user already told us what to remember, the expiry comes from fixed rules over the user's own words, and no model sets scope, lifetime, or activation — which makes it the perfect place to stand up the machinery without much risk.

There is exactly one deliberate exception to "the turn only emits cheap signals," and it's worth naming now because it recurs across the design: redaction runs inline, on the turn, before any signal is queued. Stripping a pasted secret cannot wait for a background plane, because the whole point is to keep the secret out of durable storage in the first place. So redaction is the one piece of real work the fast path does on purpose; everything else it defers.

## Signals: how the turn says something happened

The first principle is that signals come from the host's own code, never from the model and never from a guess. Detecting "the user said remember/forget" is a phrase-and-intent match on the user's *own words* — an imperative "remember…" or "forget…" — not a model interpreting mood. When the match fires, the runtime emits a signal carrying enough to act on later without dragging the whole turn along: its kind (explicit-remember or explicit-forget), the trusted scope it happened in, a reference to the exact message it came from, the audience that message had, a short redacted summary, a timestamp, and a key. Throughout these docs "signal kind" means this small closed set — explicit-remember, explicit-forget, and later the failure and success signals — and it's kept distinct from a memory's *fact kind* (personal, project, document-derived), which is a property of the stored item, not of the event that produced it.

That key is doing quiet but important work, and it's worth being precise about what it does and doesn't do, because later increments lean on it. The key is built from stable facts about the *delivery* of the event — who emitted it, its kind, the owner, and the source message it points at — so the same underlying message, even delivered twice (a retried webhook, a double-tap), produces the same key. This is *delivery* idempotency, nothing more: it stops the same source event from being written twice. It is emphatically *not* semantic dedup — noticing that "I prefer metric" said in two different conversations is the same fact is a content-level merge, and that belongs to increment 3, not here. And it is not how forget finds its target; forget is matched by resolving what the user wants forgotten, described below. Keeping these three jobs separate — same-message collapse, same-fact merge, forget-target resolution — is what stops the key from quietly becoming a dumping ground for "things that feel related."

Concretely, "collapse" on a key collision is an idempotent upsert: a second arrival of an already-seen key updates the existing ledger entry's last-seen timestamp and bumps a seen-count, and does not create a second item. (For explicit memory the seen-count is mostly inert; in increment 4 the same atomic increment is how "a failure" becomes "a failure that's happened five times.") Because the background plane can process signals concurrently, every such upsert is keyed by the idempotency key and applied as an atomic compare-and-set per `(owner, source)`, so two workers racing on the same key can't both insert. It's the same idempotency-key discipline any event-sourced system uses.

One thing happens before a signal is ever stored, and it's the inline exception named above: redaction. Even an explicit "remember" can carry a secret the user pasted; raw transcripts, secrets, personal data, and paths are stripped to a short safe summary by pattern rules before anything is queued. nullclaw does a strict version of this, scrubbing secrets and PII at the boundary with a one-way hash. The cheap online step is also the safe step. Two constraints on it matter downstream. First, redaction must *preserve relative-time phrases* — "for two weeks," "until Friday" — because expiry is computed from exactly those words and nothing else; a redactor that ate "two weeks" along with the secrets would silently break freshness. Second, redaction fails *closed*: when a span looks like it might be a secret but the classifier can't say so confidently, the safe move is to drop that span from the summary, or — if dropping it would gut the fact — to persist the item with a quarantine flag that the lifecycle refuses to auto-activate, routing it to a human instead. A maybe-secret never sails through on optimism.

## Provenance: where the memory came from is the permission model

This is the load-bearing idea of the whole increment, so it's worth being emphatic: a learned memory stores a reference to its source — here, the message the user said it in, with its version and its audience — and that reference is *not* for audit. It is how permissions work. The reasoning is the one enterprise search settled on years ago: when you derive something from a source, the derived thing has to inherit the source's access control or it leaks. Azure's AI Search calls this "security trimming" and projects document permissions onto every derived chunk; Pinecone supports the same idea by letting you filter on source-ACL ids carried in chunk metadata; Supabase enforces it with row-level security; Glean's whole pitch is inheriting the permission graph of whatever it indexes. We do the same for a remembered fact.

That source reference has two parts that do permission work, and one of them is new structure this increment must build. The *owner* is straightforward: the memory is keyed to the owner of the run, not to whoever happened to send the message. The *audience* is the part that doesn't exist yet. Reborn today has no per-message ACL — a memory's scope is exactly one of tenant, user, agent, or project, and a message carries no audience field at all. The only thread-level identity the system tracks is the participant set on a conversation thread (the `add_thread_participant` path in `ironclaw_conversations`, which maintains a `participants` set per thread). So this increment *defines* audience as new structure: for a chat message, the audience is the set of thread participants at the moment the message was sent, captured at the channel/turn layer and frozen into the source reference. It is a point-in-time membership snapshot, recorded where the message enters the system, and it is the thing a remembered fact inherits. Docs that come later treat audience as "defined in increment 2"; it is not assumed to pre-exist anywhere.

Scope defaults to the narrowest thing that makes sense — this user — because the asymmetry that governs everything is that narrowing scope is always safe and widening it is always dangerous. zeroclaw bakes this in by intersecting allowlists so a scope can never accidentally grow; we adopt the same stance.

Owner-keying and audience-bounding then have to *compose* correctly, and the place that's easy to get backwards is a shared agent where a non-owner contributes a fact. Suppose Bob, a participant in Alice's run, says something worth remembering. The fact is keyed to the run's owner — Alice — and *also* bounded by the audience of Bob's message, which is the thread Alice and Bob were both in. It is not narrowed to "Alice, private." That distinction is the whole game: storing Bob's contributed fact under Alice's scope *narrowed to user-private* would turn Bob's statement into Alice's private memory that Bob can never see surfaced and that escapes the thread it was said in — precisely the cross-participant leak the overview warns about. Owner-keying decides *whose store it lives in*; audience-bounding decides *who it may ever be shown to*. A contributed fact lives in the owner's store but stays clamped to the audience that heard it, so it can resurface inside that shared context and nowhere narrower or wider.

At use time — and "use time" means the precise moment the increment-1 planner is assembling a prompt for a specific run, not some earlier derivation step — the permission checks become real, and they re-read live state rather than trusting a snapshot. The run's scope must sit inside the item's allowed scope, and the people in the conversation *right now* must sit inside the audience the source allows. Crucially, that audience is re-evaluated against the source's *current* membership at use time, not against whatever was true when the memory was first derived — the same security-trimming discipline Glean and Azure apply, where a permission revoked yesterday takes effect today. If the source can't be resolved at all, the item fails closed and is simply omitted, and a separate source-health sweep triages whatever's rotting. A perfectly relevant remembered fact is withheld when the conversation is broader than the place it was said.

## Expiry: almost nothing lives forever, and the clock has no model in it

"Remember I prefer metric" is durable; "I'll be in London for two weeks" obviously isn't. So expiry is *computed deterministically*, and there is no model call anywhere in it. Each memory kind has a fixed category default — a personal preference defaults to "until changed or revoked," a stated travel window to something short — and that default can be refined only by a deterministic temporal grammar that reads explicit phrases out of the (redaction-preserved) text: "for N days" sets N days, "until Friday" resolves to that date. If the text names no horizon, the kind's default stands. No model reads the page and proposes a lifetime; a parser does. This is exactly why redaction has to leave "two weeks" intact, and exactly why the redaction-before-anything ordering raises no model-ordering question at all here — nothing downstream of redaction is a model.

This is deliberately unglamorous, and that's the point: per-kind category TTLs are established practice, not a novel mechanism, and the temporal grammar is a parser, not a judgment. The supporting prior art is recency decay and validity intervals — Generative Agents rank retrieval by a blend in which recency is *one of three factors* (alongside relevance and importance), and Zep stamps facts with validity intervals marking when they became and stopped being true. We borrow the deterministic spirit of both without borrowing a model.

Two clocks enforce expiry, and neither involves a model. The first is a decay weight that quietly sinks older items in the ranking without deleting them. This is a *new multiplier* layered onto retrieval, and it must be reconciled with increment 1's frozen comparator: increment 1 already lists "recency" as a low tie-break key, and this increment does not redefine that key or reorder the comparator — it adds a separate freshness *decay* as a multiplier applied alongside it, so the comparator's existing shape stays exactly as 1 froze it. The second clock is a hard expiry checked at use time, which simply stops surfacing an expired item; cleanup is deferred to a later archival sweep, and it is never a hard delete, because the standing rule is that learned data is hidden and aged out, not destroyed. Durable facts get "until changed or revoked" rather than a fixed date.

## Candidate, then active — even here

Every learned item is born a *candidate*: stored, but not yet eligible to be used. It becomes *active* only after verification. For an explicit, source-backed fact the gate is trivial, but trivial does not mean tautological — the checks have to have teeth. The auto-pass conditions are: the source reference resolves to a real message; redaction actually ran and succeeded on that content (a quarantined-as-maybe-secret item does *not* auto-pass); and the chain of scopes nests correctly — the item's scope is within what the emitter was permitted, which is within what the source's audience permits. If all three hold, the item activates automatically; if any fails, or the source is ambiguous, it goes to a human. Two checks that an earlier sketch of this gate listed are pointedly *absent*: we do not check that "nothing already contradicts it," because contradiction detection is comprehension and lands in increment 6 — last-writer-wins is acceptable here — and we do not check the tautological "it really was explicit," because the host emitted the signal from the user's own imperative, so re-asserting it proves nothing.

That sounds like ceremony for a simple feature, and for this increment it nearly is. But standing the lifecycle up now, with the trivial gate, is what lets increment 4 slot a *real* verification gate into the exact same place without reshaping anything.

## The ledger: every item's life, recorded

The store that tracks all of this is the **ledger** — and it's worth defining precisely here because later increments read it back as if it always existed. The ledger is append-only and *transition-recording*: it does not just hold each item's current state, it records every transition an item makes (candidate→active, active→revoked, active→expired, candidate→review), and stamps each transition with a *reason* and a *timestamp*. It is kept separate from the memory documents themselves; the documents are the content, the ledger is the history of how each came to be used or not used. This is the thing increment 6 means when it reads back "why was this memory surfaced, skipped, or retired" — that answer is a payoff of the ledger existing from this increment, not a new structure 6 has to invent. Throughout the design this is "the ledger," not a "log" or a "trail."

Because the ledger is shared state that the concurrent background plane mutates, every write to it is keyed by the item's idempotency key and applied as an atomic upsert, processed in order per `(owner, source)`. That ordering is what makes the next section's concurrency guarantee hold.

## Forget, and what it actually does

"Forget that" raises the question the clean-sounding framing skips: *forget what?* The user's headline — "forget what I said about the old API key" — names a fact descriptively, and the host has no model-free way to know which stored item that refers to. So forget-target resolution is, honestly, a *use-time retrieve-and-confirm step*, and it is the one allowed bit of comprehension at this boundary: the host runs the same retrieval pipeline against the user's forget phrase to find candidate items, then *confirms the match with the user in the same turn before revoking anything*. This is reading-comprehension used to *locate*, then a deterministic host action to *revoke* — the model proposes the target, the host disposes of the item — not a model-free host signal pretending to resolve descriptive language on its own.

The disambiguation contract is explicit. Zero matches: tell the user nothing matched, revoke nothing. More than one plausible match: list them and ask which (or all), and revoke only what the user confirms in-turn. Exactly one confident match: still surface it for confirmation before revoking, because an erroneous forget is itself a small data-loss event. For *this* increment we can ship the unambiguous case cheaply — a positional "forget that," referring to the fact the user just stated or the assistant just surfaced, resolves to a specific item with no descriptive search at all. Full *descriptive* forget ("the thing about the old API key," referring to something said long ago) rides the retrieve-and-confirm path; if we choose to land only positional forget in this increment, we say so plainly and defer descriptive forget to a fast-follow, rather than shipping a descriptive forget that silently revokes the wrong item.

Once a target is resolved and confirmed, revocation is where honesty matters, because it's the part everyone underestimates. The clean-sounding version — "delete the derived data" — is genuinely hard: caches, logs, and indexes all have to be reached, and with embeddings you often can't even tell which vectors a piece of source influenced; OWASP and the vector-database vendors are blunt that true erasure is difficult and rarely verified. ChatGPT's own deletion behavior — references can linger for a while — is the shipping admission of exactly this. So forget does *not* claim instant global erasure. What it does is mark the item revoked with a tombstone, and that one transition is processed *promptly — synchronously, inline at end-of-turn* — which is the single deliberate exception to "learning happens out of band," the revocation twin of the redaction exception. Because the tombstone is committed before the next turn begins, and the increment-1 planner re-checks every item's status at use time, a forgotten fact stops reaching the model on the very next turn. That is the true claim — "caught by the next use-time check," not "erased everywhere instantly." Physically scrubbing the content from every index is a separate, eventually-consistent follow-up, and for revoked user PII specifically it runs under a *bounded SLA* so "eventually" has a ceiling. (That bounded purge is distinct from the system's standing never-delete-LLM-data value, which covers non-PII learned state; revoked PII is the carve-out that does get physically removed, on a clock.)

One concurrency hazard has to be closed here, because forget and remember can race on the background plane: a forget must never be silently *un*-done by a later remember of the same fact. The guarantee is that a revocation is a *persisted tombstone*, not a one-shot delete — if a matching item is created afterward (a re-paste, a delayed duplicate signal), the tombstone is re-applied and the new item lands already revoked. Combined with the in-order, per-`(owner, source)` ledger writes above, this means forget-before-remember always wins regardless of which signal the workers happen to process first. The backstop for the whole scheme stays as before: a memory with no resolvable source can never become permissioned memory at all, so it's never surfaced.

## What happens to memory written before provenance existed

There is one boundary this increment has to handle that no later one does: memory that already exists from before provenance was a thing. Increment 1 wired retrieval against whatever was in the store, and none of those items carry a source reference, an owner-keyed scope, or an audience. The rule introduced here — "no resolvable source ⇒ never surfaced" — would, applied naively, silently erase all of that legacy memory from view the moment this increment lands. That's not acceptable, so legacy items are *grandfathered*: each pre-provenance item is migrated into a narrow **owner-only** scope and given a synthetic source marker that resolves (to "pre-provenance import") rather than failing the resolvability check. They keep working, scoped tightly to their owner and to nobody else, until they naturally age out or are revoked. Increment 1's doc flags this boundary in a sentence; this is where the policy lives.

## A floor on accumulation, so later increments don't ship a junk drawer

Increments 3 through 5 will pour far more into this store than explicit remember/forget does, and if this increment shipped a store with no bounds, those increments would be shipping pure un-curated accumulation — which the research is clear makes agents *worse*. Full curation is increment 6's job and stays there. But this increment installs the minimal floor that keeps the store honest in the meantime: a per-scope *size cap* (a bound on how many active items a single scope may hold, evicting the stalest when exceeded) and a *staleness floor* (items past a generous age with no use are dropped out of the active set). These are deliberately crude — a cap and a floor, not real curation — and they exist so that the standalone value of every increment between here and 6 is "a bounded store that helps," never "an unbounded pile that grows."

## What this sets up

After this increment, explicit memory works end to end — remembered, scoped, attributed, expiring, retrievable, and revocable — and, more importantly, the substrate exists: signals with delivery-idempotency keys and inline redaction, provenance (owner plus the newly-defined audience) as the permission model, narrow-by-default scope with a live use-time re-check, deterministic per-kind expiry on two model-free clocks, an append-only transition-recording ledger, a candidate→active lifecycle with a (currently trivial but teethed) verification gate, prompt synchronous revocation by tombstone, a legacy-grandfather migration, and a minimal store-side cap and staleness floor. Every later increment is a matter of feeding richer signals into this same machinery and making the classification and verification steps do more.

## Open questions for this increment

The per-kind lifetime defaults, the decay rate, and the per-scope cap and staleness-floor numbers all need real values against real prompts. The exact shape of a source reference — how much version and audience detail to capture for a plain chat message versus a document, and how to snapshot audience cheaply at the channel layer for channels whose participant set isn't a tidy thread — wants to be settled now, since everything later reuses it. The phrase/intent matching for "remember/forget" needs a real grammar so it fires reliably without tripping on every casual "remember when…," and the temporal grammar for expiry needs its own coverage list of the phrases it promises to parse (and a defined fallback to the category default for anything outside it). And the descriptive-forget retrieve-and-confirm path — its match threshold, and how aggressively to widen the search before giving up — is the piece most worth prototyping against real "forget the thing about X" phrasings before committing to it.
