# 5 — Reusable Procedures (Skills)

Status: engineering design, proposed. Fifth increment of the [self-learning design](00-overview.md). It reuses the recurrence-counting, classification, and verification machinery from increments 3 and 4, and turns repeated *successes* into reusable procedures. It's placed after rules because it's the same shape: a workflow recognized by recurrence, distilled by a constrained model, and held behind a gate until an external signal clears it — and for the procedures whose checks are mechanical, that signal can come from running them instead of re-running the model.

## The problem this slice solves

Some valuable things the agent does aren't a fact or a guardrail — they're a *procedure*: the specific sequence of steps that reliably produces a release note, reconciles two reports, or sets up a new project the way this team likes. When the agent rediscovers that sequence from scratch every time, it's slower, more error-prone, and more expensive than it needs to be. A **skill** captures that procedure once so it can be offered back when the same situation recurs, so the agent stops paying to re-derive a plan it has already found.

The risk here is different from rules. A rule quietly bends the model's judgment; a skill is closer to *something the agent might run*, so the dangers are the ones you'd expect — it might do the wrong thing, touch something it shouldn't, or depend on something that's since changed. There's a subtler risk too, specific to how skills get made: a skill is written by a model from the transcripts of past runs, so its body and any helper code it carries can quote a file path, an account id, or a secret that appeared in those runs. That generated body has to be redacted before the skill is stored, not just the pointer back to where it came from. The upside is that for the procedures whose success is mechanically checkable, you can verify the honest way — execute it and check the result — instead of asking a model whether its own work is good.

## What exists today, and what this changes

Reborn already has a v2 skills subsystem, and it is exactly the un-gated shape this design is built to retrofit. Today a single successful thread is enough: when a run finishes cleanly and is long enough to look like a real procedure, the engine (`mission.rs`, the skill-extraction trigger) fires a distillation that turns that *one* transcript into a skill and installs it as immediately selectable. There is no recurrence counter — one success is the whole bar. There's a name-and-similarity dedup against existing skills and an optional merge step, but no *candidate* state: nothing holds the new skill back from being offered into a turn while it's still unproven. The skill selector (`selector.rs`) only soft-de-ranks a low-confidence extracted skill — it halves the score, it never refuses to surface it — so there's no hard gate between "just learned" and "live." Skill *repair* (`mission.rs`'s repair path, writing through `skill_tracker.rs`) takes a model's revised skill body and applies it as a new version with an optimistic version check and no verification of any kind. And the metadata a skill carries (`V2SkillMetadata` — name, version, activation, source, trust, requirements, code snippets, lifetime usage/success/failure counts, revision and repair history) has **no field for machine-checkable success checks at all**; distillation emits prose plus code snippets and nothing else.

So this increment is a behavior-changing retrofit, and it's worth naming the four moves it makes. It recurrence-gates the extraction trigger so one lucky run no longer mints a skill. It adds a candidate/lifecycle state and teaches the selector to read it as a hard pre-scoring filter, so an unproven skill is unreachable rather than merely down-weighted. It gates the repair write behind the same verification a fresh skill must pass. And it adds a new typed success-check structure to the skill, populated from an external source rather than authored by the distilling model — the piece the keystone "verify by execution" depends on, and the piece that does not exist today. Because the change makes already-extracted skills suddenly under-gated relative to the new bar, the increment also has to migrate them: skills already written as selectable get reclassified to candidates and re-enter the gate, rather than being grandfathered live.

## The approach

The path mirrors the rule path. A workflow that keeps succeeding the same way is recognized by recurrence, distilled into a candidate skill, and held until verified — and for the procedures whose contract is mechanical, verification leans on **execution** rather than a live model re-run, which is cheaper and more trustworthy, reserving human review for the genuinely risky ones. We don't try to build skill distillation from first principles; the shape is well-established. Voyager is the canonical example of an agent that grew a reusable skill library from successful experience — but it's worth being precise about how Voyager *verified* those skills, because this design departs from it on exactly that point. Voyager gated each new skill behind a live GPT-4 critic: a model judging whether the model's own code worked. When it ablated that self-verification step, its capability dropped by nearly three-quarters — so the lesson Voyager teaches is that *verification matters*, not that a model is a fit judge of its own work. This design replaces the critic with execution wherever it can. For "trust generated code because it passes independent checks, not because a model approved it," the cleaner precedent is CodeT, which accepts code that agrees with independently-generated test cases rather than the model's say-so, and reports an 18.8-point pass@1 gain doing so.

## From repeated success to a candidate skill

When a multi-step workflow completes successfully, the runner emits a success signal carrying the active skill or workflow id, a summary of the steps taken, and any deterministic output checks that passed. The same idempotency-and-counting mechanism from increment 4 applies: the system doesn't propose a skill the first time something works — it waits until the *same* sequence has succeeded enough times that it's clearly a procedure and not a one-off, with each distinct underlying success counted by the atomic, idempotency-keyed increment from increment 4.

What counts as "the same sequence" needs a real mechanism, and it's strictly harder than the failure key in increment 4 — a failure collapses to a `(capability, error-category)` pair the executor already knows, while a success is a whole trajectory. The signature here is the *ordered set of tool names* the run invoked — a normalized action skeleton with the arguments dropped — so two runs that called the same tools in the same order against different inputs count as one procedure, while a run that took a different route counts as a different one. Argument-level variation is tolerated by construction (the arguments are exactly what's dropped); order and tool-identity are not. This is deliberately coarse, and tuning how much reordering or how many extra steps still counts as "the same" is one of the open questions below.

Only once the signature recurs past the threshold does the constrained-model step distill those steps into a candidate skill, and only the things that are safe to retain are kept. Provenance points back at the runs it was learned from. But provenance alone is not the whole exposure: the distilled *body* of the skill, and any helper snippets in it, were written by a model from those transcripts and can carry paths, ids, or secrets verbatim. So the body and snippets are redacted before the skill is stored — the same deterministic redaction the rest of the system applies before anything model-authored is persisted — and the learning-provenance pointer is stripped from the operational artifact on top of that, so the runnable skill carries no trace of the specific transcripts behind it. (A picoclaw fork we looked at strips learning provenance off a skill before it ships; the staging idea is more cleanly attributable to OpenClaw's Skill Workshop, which holds a learned skill in a staging area before it goes live.)

A candidate skill is not selectable. This is where the retrofit bites: the lifecycle state lives in the skill's own `MemoryDoc.metadata` — the same place v2 skills already live, alongside the version and metrics, not in increment 2's ledger — and the selector reads that state as a **hard pre-scoring filter**, dropping every candidate before any keyword or trigger scoring happens. That's the change from today's soft de-rank, where a low-confidence skill is merely down-weighted but still reachable. Nothing the agent learns is reachable until it has passed the gate, exactly as the planner ignores candidate rules.

## Verification by execution — the easy case, and only where it's earned

This is where skills can be genuinely simpler than rules — but only for the skills whose success is mechanically checkable, and only if the check is grounded in something other than the model that wrote the skill.

Start with the check itself, because the whole story rests on it and it doesn't exist today. A skill's success checks are a **new typed structure**, and the one rule that makes them trustworthy is that they are **externally grounded, never authored by the distilling model**. A model that writes both the procedure and the test the procedure must pass is grading its own homework, and will write a test its own output passes — the reward-hacking failure this whole design exists to avoid. So the checks are derived from facts outside the model: the *recorded deterministic output checks that already passed in the source runs* — the artifact the run was asserted to produce, the values it was asserted to return — plus the *declared tool and permission preconditions* the runs actually required. A check must also survive a **discrimination test**: it has to *fail* when run against a known-bad trajectory of the same procedure, because a check that passes everything verifies nothing. A skill that has no externally-grounded check — because nothing mechanical was ever asserted about its source runs — is not auto-passed; it is **permanently unverified**, and treated like any other unverified learned thing.

With a real check in hand, the cost of verifying a skill depends entirely on what kind of skill it is, and the blanket "no model call needed" is wrong as stated. Most skills are **pure prose** — a procedure the model reads and follows during a live turn — and a procedure made of prose can only be verified the way a rule is: a live behavioral re-run, watching the model actually carry out the steps and asserting the externally-grounded checks on what it produced. That's the same cost class as verifying a rule, and there's no escaping it for prose. The cheap case is the **deterministic-snippet** skill, whose code snippets are Python helpers the model can call rather than reason through — those can be exercised directly in a sandbox against the checks, with no model in the loop at all. So "free" here means free of model *tokens*, not free of compute, and even then only for the snippet case; the prose case spends real tokens. Either way, skill verification runs on the **same background plane and the same budget** as rule verification from increment 4 — it's rationed to the procedures that recurred enough to be worth it, and it amortizes the live re-runs the same way, so it doesn't get its own parallel budget.

Execution-as-verification, where it applies, also catches the failure modes that matter for skills specifically: the procedure is run against its declared checks (does it produce the expected artifact?), its dependencies are confirmed to still exist, and the permissions and sandbox requirements it needs are checked. A skill that quietly depends on something that's drifted away fails its check and doesn't promote.

Two cases still need more than a passing check. A **high-risk** skill — one that gains a new capability, reaches outside the system, makes external writes, or applies broadly — goes to human review on top of passing its checks, because "it ran and produced output" isn't sufficient assurance when the blast radius is large. And because a thin check could pass a bad skill, contract strength has to be *defined*, not left to intuition, or "a thin contract earns scrutiny" is circular. A check is strong enough to trust an execution pass only if it carries at least a minimum number of assertions, none of which is trivially true (an assertion that holds for any output, like "the result is a string," counts for nothing), and the whole set survives the discrimination test above — it fails the known-bad run. A skill whose checks don't clear that bar is treated as unverified, not as passed. Execution verifies a skill against its contract; it can't verify that the contract was worth checking, so a measurably-thin contract gets the same fate as no contract.

## Lifecycle, demotion, and drift

Skills move through the same candidate→active lifecycle as everything else. The careful agents stage learned behavior this way and the careless ones skip it; picoclaw, for instance, moves a learned skill from a learning record to draft, then active, and eventually cold and archived if it stops being used — gated on a success ratio and a minimum number of similar records, not on a model's approval. Two transitions matter especially for skills.

**Demotion.** A skill that starts failing in the wild should be pulled back and repaired — but "failure rate climbs past a threshold" has to be reconciled with what the metrics actually are, because today they're lifetime cumulative counts (total usages, total successes, total failures) that never cross a clean line: a skill that worked a thousand times and then breaks has a tiny lifetime failure *rate* for a long while even as it fails every recent run. So demotion keys on a **windowed** failure rate computed over a recent slice of those same counts — reusing the existing metrics, not standing up a parallel counter — and a skill whose recent failure rate crosses the threshold is demoted back to candidate and re-enters repair, the same canary idea as rules applied to a runnable thing. The repair write itself is now gated: a repaired skill is not a fresh selectable version the moment the model produces it (which is what happens today), it's a candidate that must clear the same verification as any other candidate before it goes live again.

**Drift.** Because skills depend on tools and capabilities that change underneath them, a periodic re-check (part of curation, increment 6) re-runs a skill's contract and demotes it if its dependencies have moved. A skill is never assumed to keep working just because it worked once; it earns its place by continuing to pass.

All of these background passes — recurrence counting, demotion, repair, drift re-checks — mutate the same skill metadata, and the existing optimistic version check in `skill_tracker.rs` isn't enough on its own, because it just rejects a write whose version moved and leaves the underlying races unresolved: a drift pass demoting a skill while a repair pass is mid-write, a late `record_usage` from a long-running turn landing after a demotion and resurrecting the skill's confidence, two passes trying to advance the same skill at once. So each skill has a **single writer at a time** for lifecycle transitions, conflicts resolve by re-reading and re-deciding against current state rather than retrying a stale write, late usage metrics are applied additively (they adjust counts, they don't reverse a lifecycle decision the pass already made), and every pass is idempotent per `(skill, pass)` so a retried or duplicated pass is a no-op. The version check stays — it's the cheap optimistic guard — but the concurrency story is the single-writer discipline around it, not the check alone.

## What this sets up

After this increment the agent reuses its own verified procedures, and the verification story is complete across all three kinds of learning: deterministic checks for facts, a live re-run for behavioral rules, the same live re-run for prose skills, and execution for the procedures whose contract is mechanical — none of them resting on a model's self-assessment. The one thing we explicitly *rejected* was the obvious shortcut of trusting a model's own review of what it produced, which is the lesson increment 4 paid for. What's left is the layer that keeps all of this from rotting and proves it's actually helping, which is the final increment.

## Open questions for this increment

The recurrence threshold for promoting a procedure to a candidate skill is the first knob, and "the same sequence" is the second: the ordered-tool-name signature above is a starting mechanism, but how much reordering, or how many extra or missing steps, still counts as one procedure is a real tuning problem and harder than increment 4's exact error key. The minimum strength of a skill's success checks is pinned above as a definition — a discrimination test plus a non-trivial assertion floor — but the exact floor (how many assertions, how aggressively to reject near-trivial ones) needs calibration against real distilled checks, since the whole verification story rests on it. And the boundary between "auto-promote on a passing execution" and "require human review" for capability-bearing skills is a product decision that should be set conservatively at first.
