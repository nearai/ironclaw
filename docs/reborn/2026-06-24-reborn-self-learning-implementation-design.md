# Reborn Self-Learning — Implementation Design

Status: engineering design, proposed.

## The problem

Reborn handles a user's requests one turn at a time, and it's good at any single turn — but it doesn't carry much forward. The same tool fails the same way on Monday and again on Thursday, and nothing remembers the first time. A user says "keep your summaries short" and it holds for the conversation, then evaporates. A three-step procedure that worked perfectly last week gets reinvented from scratch this week. The agent has a memory store it can search, but searching a store is not the same as learning: the store is passive, and nothing curates it, scopes it, or decides when last week's note is worth surfacing today.

The obvious move — let the agent write to its own memory and prompt as it works — is exactly the move that gets self-improving agents into trouble. Now the model is editing the instructions it runs under, in the middle of running them, with nothing checking that the edit is correct, still in scope, still permitted, or actually an improvement rather than a confidently-wrong guess. Add a second user and it gets worse: one person's private fact quietly leaks into another's context because nothing tracked where it came from or who's allowed to see it. The research is consistent on this — unbounded, un-curated, self-verified memory tends to make agents *worse*, not better — so "just let it remember things" is not a plan.

What we want is narrower and harder: an agent that reliably doesn't make the same mistake twice, that picks up durable preferences and reusable procedures, and that does all of it without ever letting the model rewrite its own future on a whim, without leaking across users, and without drowning the prompt in stale notes.

## The shape of the solution

The core idea is to split the system in two. There's a fast path that handles the user's turn and a slow path that learns from it afterward, and they never block each other. During a turn the agent answers the user as quickly as it can; the only thing it does for learning is emit small *signals* — little records that say "this tool just failed with an auth error" or "the user just said remember this." Emitting a signal is cheap and immediate. Everything expensive — working out what the signal means, whether it's worth keeping, and whether acting on it actually helps — happens later, out of band, where it can take its time and spend a few tokens without anyone waiting.

This out-of-band framing isn't novel; it's how the better memory systems already work. MemGPT framed the context window as RAM and external storage as disk, with a controller paging things in and out. Letta turned that into "sleep-time" agents that consolidate memory in the background while the main agent is idle. mem0 runs an asynchronous module that refreshes memory off the critical path. The original Generative Agents paper ran its "reflection" pass periodically rather than inline. We're squarely in that lineage: learn afterward, never during.

The second idea is the one that keeps it safe, and it's worth stating plainly because it's the thing most self-learning designs get wrong: **the model proposes, the host disposes.** A turn can flag that something is worth learning, but it is a deterministic, host-owned pipeline — not the model — that decides what actually gets stored, what scope it's allowed, how long it lives, and whether it's ever used. The model shows up only in the few places that genuinely need reading comprehension, and nowhere that needs to be trustworthy or reproducible. That line — model for judgment, machinery for safety — runs through every part of the design below. mem0, for example, lets the same model both extract a fact *and* decide whether to add, update, or delete it; that's convenient, but it's also exactly where an unverified self-write does its damage. We take the operation and the activation away from the model on purpose.

The rest of this document walks the two paths piece by piece — how memory reaches the model during a turn, how a signal becomes a stored thing, how the system decides what kind of thing it is, how it stays scoped and permissioned and fresh, how a learned change is verified before anyone trusts it, and how the whole pile is kept from rotting. For each, it's worth being concrete about how we'd build it and why that way rather than the alternatives.

## Getting memory into a turn

Start with the fast path, because it's where value is actually delivered: when the agent is mid-turn and needs to remember something, how does the *right* memory end up in the prompt — and not a hundred wrong ones?

The answer is a planning step that runs before each model call and owns this decision entirely. In Reborn this seam half-exists today — there's a budgeted prompt-assembly layer, but the path that pulls memory into it is currently unwired, so in practice memory only reaches the model when the model explicitly searches for it. Wiring that path is part of this work. The planner builds a small structured description of the moment — who the turn is for, the conversation, the active project or skill, the literal task text, and the concrete things named in the turn like tool names, error strings, and ids, which it pulls out with plain pattern-matching rather than a model call so it stays fast and reproducible. Then it gathers candidate context from a few different places, because "relevant memory" isn't one kind of lookup.

Some context is just loaded by key — your timezone, your locale, a standing "keep it terse" preference. There's no searching here and there shouldn't be; this is the always-on, tiny, stable slot, and the only real discipline is keeping it from turning into a junk drawer.

Some context is relevant only because of *where you are* in the work. A note about this repo's release process should surface when you're doing a release and stay out of the way otherwise. So learned items carry tags for where they apply — a skill, a project, a capability, a failure family — and the planner pulls the ones whose tags match what's active right now. This is a metadata filter, not a similarity search; it's the difference between "show me rules for the thing I'm literally doing" and "show me things that vaguely resemble it."

Then there's genuine recall from the memory store, and here the design leans on what's become the standard answer: hybrid search. A keyword search is unbeatable at exact tokens — a function name, an error string, a ticket id, a path — and useless at paraphrase. A vector search, where the query is embedded and compared by similarity, is the opposite: great at "this means the same thing in different words," hopeless at exact strings. You want both, so you run both and merge the two ranked lists. The merge has two common recipes. Reciprocal rank fusion scores each result by one over a constant plus its rank, summed across the lists; it's nice because it doesn't care that the two searches produce scores on completely different scales, and it's the default in Elasticsearch and Azure's hybrid search and what agents like praktor and the ZeroClaw-lineage assistants use, usually with the constant around 60. The alternative is to normalize the two scores and take a weighted blend, which is what zeroclaw and openclaw do and what Weaviate defaults to; it's a bit more tunable but assumes you trust the score magnitudes. Either works, so the design treats the fusion method as a knob rather than a commitment — with one firm requirement: if embeddings aren't available, keyword search alone has to be a sufficient fallback (it's a famously strong baseline), and if even that isn't available the store must say "I can't search" rather than return an empty result that reads as "nothing found." That last point sounds pedantic but it's a real failure mode — a search that silently returns nothing looks exactly like a search that found nothing, and the agent will act on the difference.

Recall over *past conversations* is a close cousin but its own lane. The trick that makes it work, borrowed from Hermes, is to index not just the prose of past turns but the tool names, the arguments, and the error strings — so a question like "how did we fix this auth thing last time" matches on the actual tool and error tokens, not just the chatter around them. A hit returns a small anchored window — the goal, the resolution, a few surrounding messages — never the whole transcript, and a recalled snippet never quietly graduates into permanent memory.

Learned rules get their own lookup, by exact trigger rather than similarity, because a guardrail that fires when something merely *resembles* its trigger is worse than no guardrail.

Everything these lanes turn up is a *candidate*, not yet context. Before anything goes near the prompt it passes a set of plain yes/no checks: is the current run within this item's allowed scope, are the people in this conversation within the audience the item's source allowed, is it still fresh, is its source still valid and unrevoked, is it active, do the product settings permit it. No model is involved; these are boolean comparisons, and any one of them can drop a candidate no matter how relevant it looked.

What survives gets ranked, and the ranking is deliberately a fixed comparator rather than a model. The order of tie-breakers expresses the priorities: an item that matched a precise trigger, has been verified, and is narrowly scoped beats one that merely scored well on similarity; after that, the search score, then how reliable the source is, then how useful the item has proven before, then recency, then sheer compactness so a shorter item wins a tie and protects the budget. It's tempting to throw an LLM at reranking, and some smaller agents do, but it's the wrong tool here: it costs a model call on every turn, gives different answers to the same inputs, and the evidence is that models aren't reliable relevance judges anyway — openclaw, one of the more disciplined systems, pointedly uses a deterministic multi-signal score for exactly this reason. A fixed comparator is cheaper, stable, and you can explain any decision it made.

Finally the planner packs the winners into the prompt slot by slot, each slot with its own token budget, skipping near-duplicates and recording *why* it dropped whatever it dropped — over budget, filtered out, duplicate, superseded, low rank — because being able to answer "why wasn't this memory used" later turns out to matter a lot for trust. And if the model finds mid-turn that it needs more, it can ask through a retrieval tool: it states what it's looking for, and the host runs this very same pipeline again with a smaller budget. The model never reaches into the store; it only asks. This "retrieve as a tool the model calls" pattern is what MemGPT and Letta do with their memory tools and what the agentic-RAG line of work formalized, and the nuance everyone converges on — which we follow — is that it's not either/or: you preload a little always-on context and expose a tool for the long tail.

## Turning a turn into a signal

Now the slow path. It starts the instant a turn does something noteworthy, and the first principle is that signals come from the host's own code, never from the model and never from a guess. When a tool returns, the code already knows the capability, a normalized success-or-error category, and whether auth was missing — it emits a signal straight from those facts. When a turn ends, the runtime knows whether it succeeded, and whether the user just corrected it or said "remember" or "forget" — and that last detection is a simple phrase-and-intent match on the user's *own words*, not a model interpreting mood. When a document is written or revoked, the memory layer says so. None of this runs a model; emitting a signal is a cheap side-effect of something the host already observed. Hermes is a useful reference here: its whole learning loop is driven by a background pass that reads the turn's *outcomes*, and the things it watches for — a correction, a repeated failure, a procedure that keeps working — are exactly the boundaries we instrument.

A signal carries enough to act on later without dragging the whole turn along: its kind, the trusted scope it happened in, references to the exact source behind it, who was allowed to see that source, the surrounding context, how explicit it was, a short redacted summary, some flags for sensitive shapes, a timestamp, and a key. That key is doing quiet but important work. It's built from stable facts — who emitted it, its kind, the owner, the source it points at, and a distinguishing detail like the capability and error category — so the same underlying event, even delivered twice, produces the same key. That lets the slow path collapse repeats by incrementing a count instead of piling up duplicates, and that count is what later turns "a failure" into "a failure that's now happened five times," which is precisely the threshold that earns the cost of doing something about it. Hermes and picoclaw both lean on recurrence this way to keep their costs bounded, and it's the same idempotency-key idea any event-sourced system uses.

One more thing happens before a signal is ever stored: redaction. Raw transcripts, tool arguments, secrets, personal data, file paths — all stripped to a short safe summary by pattern rules, with flags set for the dangerous shapes (writes to the outside world, executable content, private sources). nullclaw does a strict version of this, scrubbing secrets and PII at the embedding boundary with a one-way hash; picobot keeps a deterministic guard that refuses to let its own heartbeat chatter pollute memory. The point is that the cheap online step is also the safe step: nothing sensitive makes it into the queue, and the expensive thinking happens downstream on something already clean.

## Deciding what a signal means — and what kind of memory it is

This is the part that's easy to wave a hand at and shouldn't be, because "the system figures out what to remember" conceals a lot. Here's the actual machinery, and an honest accounting of where a model is and isn't involved.

A signal goes through four steps. The first is a router, and it's just a lookup table — because the signal's kind was already set deterministically when it was emitted, the router maps it to a rough category with no judgment at all. An explicit "forget" is a revocation. A changed document is a source-derived memory. A failure that's now recurred past its threshold is a failure signature, possibly a rule. A workflow that's succeeded the same way enough times is a candidate skill. A user correction is a preference-or-rule-or-failure that gets disambiguated a moment later. None of this needs a model; it's counting and table lookup.

The second step rebuilds the evidence — it follows the signal's source references back to the actual material (the message window, the tool result, the document excerpt), re-redacts it, and caps it to a small bundle. If those references don't resolve, the candidate dies right here, because without a resolvable source it can never be permission-checked, and a memory that can't be permission-checked can never be safely surfaced.

The third step is the only place a model appears in this pipeline, and only because the job genuinely needs reading comprehension. Telling a durable personal fact ("has two kids") from a project convention ("this repo releases on Fridays") from a fact lifted out of a shared document is a reading task; a regex can't do it and shouldn't pretend to. So a single constrained model call reads the redacted evidence and returns a label from a fixed list, plus a one-line summary, a suggested trigger, and a rough guess at how long the fact should live. This is the well-trodden "model proposes a typed item" pattern — mem0 does it to extract facts, Letta organizes memory into typed blocks, A-MEM writes structured notes, and the underlying taxonomy of "facts versus experiences versus procedures" comes straight out of the cognitive-architectures literature. The difference, and it's the whole safety story, is what we *don't* let the model do: it proposes only the content label and summary. It does not choose the operation, the scope, the expiry, the risk, or whether the thing goes live. mem0 lets the model pick the operation; that's the line we don't cross.

Because the fourth step — policy — is where all of that is decided, deterministically. Policy narrows the scope to the smallest of what the model proposed, what the emitter was actually allowed to touch, and what the source itself permits, so scope can only ever shrink. It computes the expiry by formula. It assigns a risk level from a checklist — external writes, executable skills, broad or global scope, private sources, anything PII-like all push it to high. It demands human review when risk is high or confidence is low or the scope would have to widen. It merges into an existing item if one already covers the same ground. And it rejects outright when there's nothing safe to keep. The model's proposal is just an input to this step; this step is what actually writes a candidate, and it's all rules.

So, concretely, how does the system know what kind of memory something is? Two ways, and which way depends on whether the question needs judgment. The kinds that are obvious from counts and trusted data — revocations, failure signatures, skill candidates — are decided by lookup and counting, no model. The kinds that need someone to actually read the evidence — personal versus project versus document-derived memory — get one constrained model call that labels them, and even then a deterministic policy layer has the last word on everything that matters. There's also a small deterministic salience score in here, the way zeroclaw scores importance from a category base plus a few high-signal keywords like "always" or "never" or "policy"; the model can suggest salience, but the number itself is computed, not trusted. (Generative Agents, by contrast, asks the model to score importance directly — fine for a simulation, but we'd rather not put a model in a numeric safety-adjacent loop if a heuristic will do.)

## Keeping it scoped, attributed, and revocable

A learned memory is only safe if you know where it came from, and that provenance is not decoration — it *is* the permission system. The reason is the one enterprise search figured out years ago: when you derive something (a chunk, an embedding, a learned fact) from a source, the derived thing has to inherit the source's access control, or it leaks. Azure's AI Search calls this "security trimming" and is explicit that when it chunks a document the permissions have to be projected onto each chunk or they aren't enforced; Pinecone's guidance is to mirror the source ACLs into the chunk metadata; Supabase does it with row-level security; Glean's entire pitch is that it inherits the permission graph of whatever it indexes. We do the same for learned memory: every item stores a reference to its source — the document, message, or tool result, with its version and its audience — and that reference is what every later permission check reads.

Those checks are plain set and scope comparisons. The run's scope has to sit inside the item's allowed scope; the people in the conversation have to sit inside the audience the source allowed; the source has to still exist and still match the version the memory was pulled from. A genuinely relevant memory gets withheld when the conversation is broader than the source it came from — a fact from a document shared with a small group simply does not appear in a wider room, however on-topic it is. And the attribution is keyed to the *owner* of the run, not whoever happened to send the message, because in a shared agent attributing to the sender is how you drop one participant's learning into the owner's scope. The asymmetry to hold onto is that narrowing scope is always safe and widening it is always dangerous, so everything defaults narrow; zeroclaw bakes this in by intersecting allowlists so a scope can never accidentally grow.

Revocation is where honesty matters, because it's the part everyone underestimates. When a source is revoked or deleted, every memory derived from it has to stop being usable. The clean-sounding version — "delete the derived data" — turns out to be genuinely hard: you can't easily tell which embeddings a piece of source influenced, and there are caches and logs and indexes that all have to be reached; OWASP and the vector-database vendors are blunt that true erasure is difficult and rarely verified. So the design doesn't pretend revocation is instant deletion. Instead it enforces revocation *at use time*: because the planner re-checks the source on every single use, a revoked source stops reaching the model on the very next turn, before any cleanup job has run. Actually scrubbing the derived data from every index is a slower, eventually-consistent follow-up. The guarantee we make is the use-time check — which, not coincidentally, is exactly the fix the secure-RAG practitioners land on, offloading fresh revocation decisions to a check at query time rather than trusting stale stored metadata. The backstop for the whole scheme: a memory with no resolvable source can never become permissioned memory at all, so it's never surfaced.

## How long things live

Almost nothing should live forever, and the lifetime shouldn't be guessed. A "visiting London for two weeks" fact is worthless in a month; "has two kids" is durable but still needs to be correctable; "this repo releases on Fridays" should die when the doc that said so changes. So expiry is computed: a fixed per-kind range gives the floor and ceiling, the source's own lifetime caps it, and within those bounds the model's read of any natural expiry in the text — "for two weeks" — is allowed to refine it. That ordering is deliberate. The fixed ranges do the load-bearing work; the model only fine-tunes inside them, because while a model is perfectly good at reading "two weeks" off the page, it is not calibrated to judge how long things *should* live, and letting it set expiry directly would be trusting it with exactly the kind of safety decision we don't trust it with. This model-suggested-but-clamped expiry is the one genuinely novel piece in the freshness story, which is why it's clamped; the established practice it sits on top of is well-proven — recency decay (Generative Agents weight memories by how recently they were touched) and validity intervals (Zep stamps facts with when they became and stopped being true).

Freshness then runs on two clocks, neither involving a model. One is a decay weight applied while ranking, so older items quietly sink without being deleted; some kinds are exempt, because your timezone doesn't get less true with age. The other is a hard expiry checked at use time — an expired item simply isn't surfaced — with the actual cleanup deferred to the curator, which archives rather than deletes, because the project's standing rule is that learned data is never hard-deleted, only hidden and aged out. zeroclaw and the ZeroClaw-lineage agents run this exact two-clock arrangement, decay for ranking and a separate retention sweep for cleanup, and it's a good shape to copy.

## Candidates, activation, and the thing that actually makes this safe

Every learned item is born a *candidate*: stored, tracked, but never used. It becomes *active* — eligible to affect a future turn — only after it's been verified, and even "active" only ever means "eligible, and re-checked at use time," never "always injected." The store that tracks all this is a ledger separate from the memory documents themselves; a plain memory might also get written into the memory store, but a rule or a skill lives only in this ledger and is pulled in by the planner. The lifecycle is small and event-driven: a candidate becomes active when verification passes and the risk is low; it goes to a human first when the risk is high or confidence is low; it expires when its time passes; it's revoked when its source is; it's demoted when a regression shows up; it's superseded when curation finds a newer item that contradicts it. Staging learned changes behind a gate like this is what the more careful agents do — picoclaw moves a learned skill through draft, then accepted-or-quarantined, then active, then eventually cold and archived; Hermes can stage learned writes behind an approval step — and the ones that *don't* gate are the cautionary tales.

Which brings us to verification, the single most important piece and the one most self-learning systems get wrong. The whole question is what stands between "the model thinks this is a good rule" and that rule actually changing behavior, and the answer must not be the model's own opinion. This isn't a stylistic preference; it's the most strongly evidenced finding in the area. When Voyager ablated its self-verification step, its capability dropped by nearly three-quarters — the verifier was the most important component it had — and there's a clear line of results showing that LLMs can't reliably check or correct their own reasoning without an external signal. The cautionary case is Hermes, otherwise the most capable self-learning agent in the comparison set: its acceptance gate is essentially "the review model decided to save it," and learned skills land in the store unverified by execution. Its own teardown calls that out as the single biggest thing a design like this should improve on. So we don't do that.

What verification *is* depends on what's being verified, and the trick is to use the cheapest trustworthy check available for each. A plain memory is checked deterministically — the references resolve, it was explicit, the scope fits, nothing contradicts it — and only goes to a human when it's ambiguous or conflicting. A document-derived memory is checked by going back to the document and confirming it still exists, still matches, still retrieves. A learned skill, because it's runnable code, is verified by *running* it in a sandbox against its own success checks — deterministic, free, no model needed, which is exactly Voyager's insight that executable skills are the nice case. The hard case is a learned rule, because a rule changes a *model's* behavior and the only way to know if it worked is to see the model behave. So a rule gets two checks: first a replay of the recorded failing run with the rule injected, which is deterministic and free but only proves the wiring is right because a replay reuses canned responses; and then an actual live re-run of the failing task with the rule applied, which confirms the thing that used to fail now passes *and* that a handful of unrelated cases still behave so the rule isn't over-firing. The decision to accept is that external pass-or-fail, never the model's say-so. A live re-run costs a real call, so it's rationed to the failures that have recurred enough to be worth it — which is what that recurrence counter back at signal time was quietly setting up. All of it runs in the background on a budget, reusing cached prompt prefixes and batching to keep the live calls affordable, the way Hermes amortizes its background review by reusing the cached prompt.

## Keeping the pile from rotting, and knowing if any of it helps

Memory that only accumulates gets worse, not better — and this is measured, not assumed. On several long-term-memory benchmarks a naive memory layer loses to just stuffing the raw history into the context window, and over-compressing memories into terse facts measurably hurts. The lesson isn't "memory is bad"; it's "un-curated memory is bad," which makes curation not a nicety but the thing that justifies having memory at all. So a scheduled pass keeps the set healthy, almost entirely without a model: it merges duplicates by content hash and high similarity; on a contradiction it links the new item as superseding the old and demotes the old one out of retrieval rather than silently overwriting it (Zep calls this invalidating rather than deleting; mem0 updates rather than overwrites; Hermes spots contradictions by finding facts that share entities but diverge in content); it ages stale items into the archive; it demotes a rule or skill whose failure has come back; and it holds each slot to a budget. Usage counts nudge how salient an item is, but they're never allowed to become a closed-loop optimizer that ships changes because some proxy number moved — that way lies reward-hacking, and it's an explicit non-goal.

Whether the whole enterprise is even working gets answered by a measurement rather than a vibe. The honest way to ask "did the harness learn" is an ablation: take a task — ideally one drawn from a real recurring failure, so you're testing the loop on the mistakes it claims to fix — and run it three ways with the *same* underlying model held fixed: with no memory, with the raw prior material dumped in uncurated, and with the curated learned context. The curated version has to beat *both* of the others; if it can't even beat dumping everything in, it isn't earning its keep, and on short histories memory genuinely can be a net negative. Holding the base model fixed matters, because otherwise you can't tell whether the harness got smarter or you just swapped in a better model — a subtlety the benchmark literature is candid about, and the reason this is run as an ablation rather than reported as a single headline number. The system also watches whether new learning quietly breaks old behavior, which is the alarm for catastrophic forgetting, and re-runs the top recurring failures live now and then to catch silent regressions. Where a task's success can't be checked mechanically, a model might grade the open-ended output — but only as a grader for the measurement, never as the gate that promotes a learned item, because those are different jobs and conflating them is how you launder a model's opinion into an acceptance decision.

## What happens when pieces break

The governing rule for failure is that the user's turn keeps going and the permission checks fail closed. If the signal intake is down, the turn still answers and that one learning opportunity is simply lost. If the background queue backs up, processing waits and nothing user-facing notices. If the classifier model is unavailable, the deterministic categories still flow through and the rest waits for reprocessing. If embeddings or the vector index are down, search degrades to keyword-only and recall still works; if even keyword search is unavailable the store says so rather than faking an empty result. If a permission check can't run, the system withholds the item rather than guessing. If verification errors out, the candidate just stays a candidate. If a memory's source reference has gone missing, it can't be surfaced and survives only as telemetry. The throughline is that a missing piece of the *learning* machinery removes a capability, never a response, and a missing piece of the *safety* machinery removes access, never grants it.

## What's settled and what's still open

The mechanisms above are the proposal; what's left are dials, not architecture. The exact rank tie-break weights and per-slot token budgets need tuning, as does the fusion method and its constant per provider. The recurrence thresholds — how many failures before a rule, how many successes before a skill — and the confidence threshold that forces human review are policy choices. The per-kind lifetime ranges and decay rates need to be set, and the open question of whether to ever trust model-suggested expiry beyond the clamp needs evidence before it's answered. The live-verification budget per owner and the auto-promotion threshold per risk level are operational knobs. And the exact set of labels the classifier may return, the minimum metadata a learned memory must carry, the evaluation task set, and which recurring failures earn a standing live canary are all still to be pinned down. None of them change the shape; all of them need numbers.
