the first-class citizen

Capability is the unit the whole kernel exists to mediate

Not a tool, not an extension, not a permission flag — a capability: a typed access lease granting exactly what a tool may reach (api.stripe.com + read /workspace/invoices). It is the one noun that binds the three properties IronClaw exists to guarantee, at once.

Self-extension

A user hands their agent a new capability.

Multi-tenancy

That capability cannot reach across a tenant boundary.

Consistency

Neither the capability nor the boundary can be violated by a runtime bug.

One noun binds all three. Make it first-class — typed, granted, leased, isolated by construction — and self-extension, isolation, and correctness become three views of one problem.

scope of cleaning · the demolition ledger

What gets deleted, and at which stage

The refactor is subtractive: almost nothing new is invented — the storage seam, the memory backend, and now the host_api vocabulary all already exist. The value is in the removal: a store reimplemented per backend, five look-alike request DTOs re-wrapped ~14 times per call, six mediator test-double traits, a 771-identifier shadow runtime, and a six-figure god-crate all come out. Nothing is deleted until its replacement carries the load — then the old wiring is removed in a sweep.

build → migrate → delete

The order is always the same: stand up the new vocabulary, migrate every caller onto it, then delete the old wiring in one sweep. As of 2026-07-18 the refactor is at the first step: Invocation/Authorized/Outcome exist but additively — nothing returns them yet, Authorized is minted-then-discarded, and the mirror DTOs are frozen but not yet migrated off. Deletion is the last step and has not begun. The ratchets (§10) hold the line so a half-finished demolition can't regrow — but a green ratchet proves the names/allowlist, never that the wiring collapsed (see Slice B: DeploymentConfig renamed the types without replacing the setup).

§2/§8 Trust boundaries preserved ▸
✓ preserved

The agent loop

Untrusted behavior. May only ask for effects through host ports; cannot name secrets, the dispatcher, or the network.

✓ preserved

Runtime-lane execution

WASM / script / MCP / container code is untrusted — mediated handles only, never ambient authority. The closed enum removes a dyn, not the boundary.

✓ preserved

Runtime-supplied data

Egress / worker / MCP output is untrusted — validated, bound to the authorized invocation, bounded, and redacted before re-entering trusted flow.

Shell cross-tenant escape (#6170) — FIXED (verified 2026-07-18): the fail-open HostedSingleTenant → LocalSingleUser → LocalHost mapping is gone; the resolver is structurally fail-closed (a served multi-tenant boot resolves to ProcessBackendKind::None, and cross-family pairs return false with a test asserting "hosted multi-tenant must never produce LocalHost"); HostProcessPort carries the #6170 rename + comment. Mechanism differs from the doc's sketch: there is a ProcessSandboxBackend trait and SandboxMount { container_path, writable } (no for_scope/from_host_path) — host roots live in trusted executor config, not in the mount type. Protection real; the doc's named API is aspirational.
