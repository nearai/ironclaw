WASM tools run inside a WebAssembly sandbox powered by [wasmtime](https://wasmtime.dev/). They get a small, fixed set of host functions ([see below](#host-functions)) and nothing else: no syscalls, no environment access, no ambient network. Every capability the tool actually uses — which hosts it can reach, which secrets the host injects on its behalf, which workspace paths it can read — must be declared in a sidecar `capabilities.json` file.

This is IronClaw's recommended approach for custom integrations that need stronger isolation than a built-in Rust tool provides.

---

## How It Works

```
LLM selects tool
    ↓
IronClaw loads WASM module (from cache or disk)
    ↓
Tool executes inside wasmtime sandbox
    ↓
Network requests route through the network proxy
    ↓
Proxy validates domain against allowlist
    ↓
Proxy injects credentials from encrypted store
    ↓
Response returns to WASM module
    ↓
Tool output passes through Safety Layer
    ↓
LLM receives sanitized result
```

## Sandbox Architecture

### Fuel Metering

Every WASM instruction consumes "fuel." When a module runs out of fuel, it is terminated immediately. This prevents infinite loops and runaway computation.

```bash
# Configure fuel limit (default: 100,000,000 instructions)
export WASM_FUEL_LIMIT=100000000
```

### Memory Limits

WASM modules are allocated a fixed linear memory. Attempting to allocate beyond the limit traps and terminates the module.

```bash
# Configure memory limit (default: 16 MB)
export WASM_MEMORY_LIMIT=16777216
```

### Rate Limiting

Each WASM tool has a per-tool rate limiter to prevent a misbehaving or compromised tool from making excessive API calls.

Rate limits are declared inside the relevant capability section of `capabilities.json` — for HTTP requests, under `http.rate_limit`:

```json
{
  "http": {
    "rate_limit": {
      "requests_per_minute": 60,
      "requests_per_hour": 1000
    }
  }
}
```

## capabilities.json

Every WASM tool ships a `<tool>.capabilities.json` alongside its `.wasm` binary. This file is the single source of truth for what the tool is allowed to do. Capability sections (`http`, `secrets`, `workspace`, `tool_invoke`, `webhook`) sit at the root alongside metadata (`description`, `version`, `wit_version`). They may also be nested under a `capabilities` key for compatibility with channel-style capability files; the loader accepts both shapes.

```json
{
  "version": "0.1.0",
  "description": "Fetches data from example.com",
  "http": {
    "allowlist": [
      {
        "host": "api.example.com",
        "path_prefix": "/",
        "methods": ["GET", "POST"]
      }
    ],
    "credentials": {
      "example_api_key": {
        "secret_name": "example_api_key",
        "location": { "type": "bearer" },
        "host_patterns": ["api.example.com"]
      }
    },
    "rate_limit": {
      "requests_per_minute": 30,
      "requests_per_hour": 500
    }
  },
  "secrets": {
    "allowed_names": ["example_api_key"]
  },
  "workspace": {
    "allowed_prefixes": ["context/", "daily/"]
  }
}
```

### HTTP allowlist

`http.allowlist` controls which destinations the tool can reach via the `http_request` host function. Each entry pins a `host`, an optional `path_prefix`, and the allowed `methods`. Requests that don't match an entry are rejected by the host before any connection opens.

### Credential injection

`http.credentials` declares which secrets the host will attach to outbound requests, and where. The credential value is never passed into the WASM module's memory — the tool builds an unauthenticated request, the host matches the destination against `host_patterns`, decrypts the secret in its own address space, and writes it into the request just before sending.

`location.type` selects the injection point:

| `type` | Effect |
|--------|--------|
| `bearer` | Sets `Authorization: Bearer <secret>` |
| `basic` | Sets `Authorization: Basic <base64(username:secret)>`; requires `username` |
| `header` | Sets a custom header; requires `name`, optional `prefix` |
| `query_param` | Appends `?name=<secret>`; requires `name` |
| `url_path` | Substitutes a placeholder in the URL or body template; requires `placeholder` |

For the full end-to-end injection flow (and why this can't be bypassed by a misbehaving tool), see [Secrets & Sandboxing](/security/secrets-and-sandboxing#how-a-tool-uses-a-secret).

### Secrets allowlist

`secrets.allowed_names` is the second half of the credential gate. A tool can only resolve secrets whose names appear in this list (glob patterns like `github_*` are allowed). Even if a credential mapping references a secret, the host will refuse to load it unless the name is allowlisted here.

### Workspace read

`workspace.allowed_prefixes` declares which workspace path prefixes the tool can read via the `workspace_read` host function. There is no `workspace_write` host function exposed to tools — workspace mutations happen through the agent's `memory_write` tool, which carries its own dispatch and audit trail.

## Host Functions

WASM modules can call the following host functions provided by IronClaw:

| Function | Description |
|----------|-------------|
| `log(level, message)` | Write to the IronClaw log at the given severity level |
| `now_millis()` | Return the current Unix timestamp in milliseconds |
| `workspace_read(path)` | Read a document from the workspace (capability-gated) |
| `http_request(method, url, headers, body, timeout)` | Make an HTTP request; the host validates the destination, injects credentials, and scans the response for secrets before returning |
| `tool_invoke(alias, params)` | Declared in the WIT interface but not yet implemented for WASM tools; the host currently returns an error. |
| `secret_exists(name)` | Check whether a secret is present and accessible to this tool. Returns a boolean only — never the value |

These are the only host capabilities available. A WASM tool cannot call arbitrary OS functions, fork processes, or access the network directly — all network access goes through the proxy.

## Tool Discovery

WASM tools are discovered at startup from two locations:

- `~/.ironclaw/tools/` — User-installed tools
- `<workspace>/tools/` — Per-workspace tools

Each tool directory must contain:
- `<toolname>.wasm` — The compiled WebAssembly module
- `capabilities.json` — The capability declaration

## Installing WASM Tools

```bash
# Install from a local file
ironclaw tool install ./my-tool.wasm

# Install from a URL
ironclaw tool install https://example.com/tools/my-tool.wasm

# List installed tools
ironclaw tool list
```

Or via the `extension_install` tool in chat:
> "Install the tool from https://example.com/tools/my-tool.wasm"

---

## Module Compilation and Caching

WASM modules are compiled by wasmtime on first load and cached. Subsequent loads use the compiled artifact, so only the first invocation of a new tool incurs compilation overhead.

The cache is stored at `~/.ironclaw/wasm-cache/`.

## Security Notes

<Warning>
Review `capabilities.json` before installing any WASM tool from an untrusted source. The capability file determines exactly what the tool can access — network hosts, filesystem paths, and which credentials are injected.
</Warning>

<AccordionGroup>
  <Accordion title="Credential injection is one-way" icon="lock">
    The proxy injects credentials into outgoing requests. There is no mechanism for a WASM module to read stored secrets directly. Even if a WASM module is compromised, it cannot access the secrets store.
  </Accordion>

  <Accordion title="No process spawning" icon="shield">
    WASM modules cannot spawn child processes, execute shell commands, or load dynamic libraries. The WASI interface exposed to modules is minimal.
  </Accordion>

  <Accordion title="Fuel bounds runaway computation" icon="zap">
    A buggy or malicious WASM tool that enters an infinite loop will be terminated when it exhausts its fuel allocation. This bounds the cost of a single invocation; it does not limit how many separate invocations a tool can make or how slow each one is.
  </Accordion>
</AccordionGroup>
