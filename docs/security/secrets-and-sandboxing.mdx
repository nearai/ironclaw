The [Security overview](/security) states the rule: models and tools never see your credentials. This page is how that rule actually holds up: what the master key is, how secrets get from the encrypted store into an outbound HTTP request without crossing the tool's address space, and why the WASM sandbox is what makes the guarantee enforceable rather than aspirational.

For the WASM tool configuration reference, see [Sandboxed Tools](/capabilities/sandboxed-tools).

---

## Where Secrets Live

Your stored credentials are encrypted in IronClaw's database. They are decrypted only transiently, inside the host process, for the duration of a single request, and tool parameters and outputs are scrubbed for secret-shaped data before anything is logged.

### The Master Key

A single **master key** unlocks the secret store. On startup IronClaw resolves it in this order:

<Steps>
  <Step title="`SECRETS_MASTER_KEY` environment variable">
    Used as-is. Must be at least 32 bytes; a hex-encoded 32-byte key (64 characters from `openssl rand -hex 32`) is the convention. This is the path for Docker, CI, and air-gapped deployments where you supply the key explicitly.
  </Step>

  <Step title="OS keychain">
    macOS Keychain or Linux secret-service. The preferred local path: auto-generated on first run, stored under the user's account, never written to disk in plaintext. You never see it.
  </Step>

  <Step title="Auto-generate and persist to `~/.ironclaw/.env`">
    Fallback for headless Linux or containers where no keychain is reachable. IronClaw generates a fresh key, writes it to `~/.ironclaw/.env`, and re-loads it on every subsequent start as if it had been set in the environment. Treat the file like a credential.
  </Step>
</Steps>

If a freshly generated key meets a secrets store that already contains encrypted rows from a previous key, IronClaw refuses to start and rolls back its own persistence. A new key cannot decrypt rows that were encrypted with the previous key, and silently continuing would shadow unrecoverable data; the abort keeps re-firing on every start until you restore the original key or clear the stale rows.

### Per-Secret Encryption

Every secret is encrypted with **[AES-256-GCM](https://en.wikipedia.org/wiki/Galois/Counter_Mode)** using a key derived from the master key plus a random per-secret salt ([HKDF-SHA256](https://en.wikipedia.org/wiki/HKDF)). Two consequences matter to you:

- Two secrets with the same plaintext value produce different ciphertext on disk. An attacker who reads the database cannot tell whether two API keys are the same.
- A leaked individual encrypted row is useless on its own. Decrypting it requires the master key, which lives in the OS keychain, in the host's process environment, or — on the headless fallback — in `~/.ironclaw/.env`. The key is never stored in the database. One caveat: on the `.env` fallback the key file and the database sit side by side, so for that deployment the row-level encryption is only as strong as the file permissions on `~/.ironclaw/.env`. The keychain path does not have this weakness.

## How a Tool Uses a Secret

The most important property of IronClaw's secrets system is that **tools never see the raw value of a credential**, even when they make HTTP calls that require one. The secret is added to the request by the host process, in flight, after the tool has handed off the request.

<Steps>
  <Step title="The tool builds an unauthenticated request">
    A WASM tool says, in effect: "POST to `https://gmail.googleapis.com/gmail/v1/users/me/messages/send` with this JSON body." It does not set an `Authorization` header.
  </Step>

  <Step title="The host matches the destination against the tool's credential mappings">
    Each tool ships a `capabilities.json` that declares which hostnames it talks to and which secrets it expects to use for them. The host looks up the request's destination and finds the matching mapping.
  </Step>

  <Step title="The host checks the tool's allowlist">
    Even if a credential is mapped to the destination, the tool can only use secrets that are explicitly listed in its allowed-secrets declaration. A tool cannot opportunistically use a secret it didn't declare.
  </Step>

  <Step title="The host decrypts the secret in its own memory">
    The encrypted row is read from the database, decrypted, and held in a wrapper that requires an explicit call to expose the value. The plaintext exists only inside the host process, only for the duration of one request.
  </Step>

  <Step title="The host injects the secret into the outbound request">
    Depending on the credential mapping, the host writes the value into one of: the `Authorization` header (Bearer or Basic), a custom header, a query parameter, or a URL path segment. The request is then sent.
  </Step>

  <Step title="The response comes back through the leak detector">
    Before the tool sees the response, IronClaw scans it for known secret patterns. A match is redacted or the response is blocked entirely. Then the response is handed to the tool, and from there to the model.
  </Step>
</Steps>

The tool's view of all this is: "I made a request and got a response." It cannot read the headers it actually sent. It cannot read its own outbound URL after query-parameter injection. It cannot enumerate the credential registry. Those host functions do not exist.

## Why WASM

The "tool never sees the credential" guarantee depends on something the tool cannot bypass. If tools were normal binaries, the guarantee would be a lie; a tool could just read the secret from disk or from an environment variable. WASM closes that escape hatch.

A normal compiled program starts with **all** of the host's authority and gives some of it back. A WASM module starts with **none** and is given exactly what the host hands to it. By default, a `.wasm` module loaded by IronClaw has:

- No filesystem access
- No network access
- No environment variables
- No ability to spawn processes
- No syscalls
- A single, fixed-size chunk of memory

Anything beyond that is an explicit host function exposed by IronClaw: `log()`, `now_millis()`, `workspace_read()`, `secret_exists()` (existence-only; never returns the value), and `http_request()` (the entry point that triggers the credential-injection flow described above). A `tool_invoke()` host function is declared in the WIT interface for future chained-tool calls but currently returns an error. That is the entire surface area.

The practical implication: a malicious or compromised WASM tool cannot read your SSH keys, reach your `.env`, install a cryptominer, or persist anything outside of what its `capabilities.json` declared. What it *can* still do is bounded by what you granted it — and that grant is not nothing. A compromised tool can make authenticated requests to its allowlisted hosts with data of its own choosing, and it can read the workspace prefixes you opened to it. The sandbox bounds the blast radius; it does not make a tool you over-granted harmless. Grant hosts, secrets, and workspace prefixes narrowly.

For the configuration details (fuel metering, memory limits, rate limits, the `capabilities.json` schema), see [Sandboxed Tools](/capabilities/sandboxed-tools).

## The Leak Detector

Credential injection keeps secrets out of the tool. The leak detector keeps secrets out of the model.

Even though the tool never saw the credential it sent, the **server it called** might echo it back. This happens more often than you'd think:

- An API returns an error message like `unauthorized: token sk-abc123... is invalid`
- A debug endpoint reflects request headers in the response body
- A misconfigured upstream proxy pastes the credential into a downstream payload

If one of those reaches the model verbatim, the secret is now in the model's context, and from there it can flow into the next tool call, into a saved memory, or into a reply on your screen.

The leak detector's job is to stop that at the boundary: IronClaw scans every tool response for known secret shapes **before that text enters the model's context**. The agent's own reasoning narration is scanned on the same path. The detector ships with around twenty patterns covering, among others: OpenAI, Anthropic, OpenRouter, and Groq API keys; GitHub tokens (classic and fine-grained); AWS, Google, Stripe, Slack, Twilio, SendGrid, and Telegram credentials; PEM and SSH private keys; generic Bearer/auth headers; and high-entropy hex blobs that look like they shouldn't be in a response.

A match triggers one of three actions, depending on the pattern's severity:

| Action | Effect |
|--------|--------|
| **Block** | The response is rejected entirely; the tool call surfaces as an error. |
| **Redact** | The matched span is replaced with `[REDACTED]` before the content moves on. |
| **Warn** | The match is logged; the content passes through. |

This is one scan point, and it is the load-bearing one: catch the leak at the tool-output boundary so it never enters the model's context in the first place. The model's final reply to you is not separately re-scanned, so a secret that reaches the model by some other route — you pasting your own key into the chat, or a tool output whose shape the detector didn't recognize — is not caught on the way back out. Treat the detector as a strong backstop against credentials echoed by upstream APIs, not as a guarantee that no secret can ever reach your screen.

## Worked Example: Sending an Email

Here's what happens when you ask IronClaw to send a Gmail message.

<Steps>
  <Step title="You say: 'Email Alice with the meeting notes'">
    The model picks the `gmail_send` WASM tool and calls it with `{ to: "alice@example.com", body: "..." }`.
  </Step>

  <Step title="The tool builds an HTTP request to gmail.googleapis.com">
    The tool's code constructs a POST without any `Authorization` header. It cannot read your OAuth token. That token is encrypted in the database and the tool has no path to it.
  </Step>

  <Step title="The host injects the OAuth token">
    The tool's `capabilities.json` declares: "for `gmail.googleapis.com`, use `google_oauth_token` as a Bearer header." The host decrypts the token in its own memory and writes the `Authorization: Bearer ...` header onto the outbound request.
  </Step>

  <Step title="Gmail responds">
    Gmail returns `{ id: "abc123", threadId: "..." }`. The leak detector scans this tool output, finds no secret patterns, and lets it into the model's context.
  </Step>

  <Step title="The model summarizes for you">
    The model produces "Sent your message to Alice (id abc123)" and it is delivered. The OAuth token never reached the model's context to begin with — the leak scan on the Gmail response is what guaranteed that.
  </Step>
</Steps>

At no point did your OAuth token enter the model's context, the tool's memory, the log file, or the response stream visible to you.

## What This Doesn't Defend Against

No security model is total. To set expectations clearly:

<AccordionGroup>
  <Accordion title="A compromised host process" icon="triangle-alert">
    The host process holds the master key in memory and can decrypt any secret. If an attacker gets root on the machine running IronClaw, the secret store is exposed. Defense at this layer is your operating system's job. Keep IronClaw on a machine you trust.
  </Accordion>

  <Accordion title="A malicious capabilities.json" icon="file-warning">
    A WASM tool can only do what its `capabilities.json` declares, but you are the one who approves that file at install time. A tool that declares broad host patterns and a sensitive credential is a tool that can use that credential against those hosts. Review capability files before installing tools from untrusted sources.
  </Accordion>

  <Accordion title="An upstream API that legitimately needs the secret" icon="link">
    The credential-injection model puts your token in a header to an API you trust. If that API is itself compromised or malicious, it has your token. The defense is choosing which APIs to grant credentials to. IronClaw enforces the boundary, but the trust decision is yours.
  </Accordion>

  <Accordion title="Novel secret shapes the leak detector doesn't know" icon="shield-question">
    The leak detector ships with patterns for the most common credential formats. A custom internal API token with no recognizable shape will not match. If you operate such a service, add its pattern to the detector.
  </Accordion>
</AccordionGroup>

The system is designed so that the most likely failure modes cannot happen by construction: the model leaking a secret it was helpfully shown, or a malicious tool reading credentials it was not granted. The remaining risks are the ones a human is in the loop to evaluate.
